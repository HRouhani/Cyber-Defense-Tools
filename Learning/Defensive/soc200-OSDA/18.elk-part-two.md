# 18. SIEM Part Two: Combining the Logs

## Chapter Goal

This chapter follows one complete attack through four phases and shows how to investigate it in Elastic/Kibana.

The four phases are:

```text
1. Web server initial access
2. Lateral movement to the application server
3. Persistence and privilege escalation
4. Actions on the domain controller
```

Main tools and data sources:

```text
Kibana Discover
KQL
Elastic Security rules
Threshold rules
Custom query rules
Sysmon
Windows Security logs
Apache logs
Linux SSH logs
PowerShell logs
OSQuery
Timelines
```

Important idea:

```text
One log source rarely tells the complete story.

The defender must combine:
web logs,
Linux logs,
Windows logs,
Sysmon,
PowerShell events,
network activity,
and authentication events.
```

---

# Network Overview

Important telemetry:

```text
snort03   → Snort logs
web01     → Apache and Linux logs
appsrv01  → Windows logs and Sysmon
dc01      → Windows logs and Sysmon
```

All endpoints except the attacker machine have Elastic Agent installed.

---

# Complete Attack Path

```text
attacker02
→ scans web01
→ discovers vulnerable page
→ performs command injection
→ gains access to web01
→ logs in as root
→ pivots toward appsrv01
→ brute-forces domain account offsec
→ successfully logs in
→ downloads PowerShell exploit
→ exploits PrintNightmare
→ creates local admin adm1n
→ runs Mimikatz
→ dumps credentials from LSASS
→ obtains Domain Administrator credentials/hash
→ uses Pass-the-Hash/WMI
→ accesses dc01
→ runs ntdsutil.exe
→ dumps Active Directory database and registry hives
```

# 18.1. Phase One: Web Server Initial Access

## Goal

Detect:

```text
web enumeration

```

---

# 18.1.1. Enumeration of web01

The analyst begins with:

```text
Kibana
→ Discover
→ data view logs-*
→ select attack timeframe
```

A large spike in events appears during the two-minute attack window.

The first suspicious characteristics are:

```text
event.category: web
event.dataset: apache.access
agent.name: web01
many requests in approximately one second
```

Additional suspicious values:

```text
user_agent.name: IE
user_agent.os.full: Windows XP
source.address: 192.168.51.54
http.response.status_code: 404
many different URL paths
```

---

## Why This Is Suspicious

```text
Windows XP is unusual in a modern enterprise.
Internet Explorer is outdated.
Hundreds of requests in one second are not human behavior.
Many different paths suggest automated enumeration.
```

---

# KQL: Suspicious Web Scanning

```text
user_agent.os.full: "Windows XP"
and source.address: "192.168.51.54"
```

Meaning:

```text
Find Apache events from the suspicious source
that claim to use Windows XP.
```

---

# Filtering Out Scan Noise

Web scanners often generate:

```text
404 = Not Found
403 = Forbidden
```

To find successful or interesting responses:

```text
user_agent.os.full: "Windows XP"
and source.address: "192.168.51.54"
and not http.response.status_code: (404 or 403)
```

This removes most failed requests.

---

# Important Discovered Paths

The attacker discovers:

```text
/
 /system_web/
 /info.php
 /system_web/status.php
```

The most important page is:

```text
/system_web/status.php
```

---

# Command Injection

After discovering `status.php`, the attacker sends arguments through the URL.

The URL contains shell commands designed to:

```text
execute Netcat
connect back to attacker02
use TCP port 8000
create a reverse shell
```

Concept:

```text
HTTP request
→ vulnerable status.php parameter
→ shell command interpreted by server
→ web01 connects to attacker
→ attacker gains shell
```

At this point:

```text
web01 is considered compromised
```

---

# Command Injection Indicators

Potential fields:

```text
url.original
url.path
http.response.status_code
source.ip
user_agent.original
```

Useful characteristics:

```text
HTTP 200 response
spaces encoded as + or %20
shell metacharacters
nc / netcat
bash
cmd
PowerShell
reverse-shell IP and port
```



# LSASS

LSASS means:

```text
Local Security Authority Subsystem Service
```

It manages:

```text
logon sessions

```

Mimikatz targets:

```text

```

---

# OSQuery: Logged-In Users

Query:

```sql
select
    user,
    strftime('%Y-%m-%d %H:%M:%S', time, 'unixepoch') as time,
    tty,
    type
from logged_in_users;
```

Purpose:

```text
Find users currently or recently logged into appsrv01.
```

Result reveals:

```text
Administrator session
```

This matters because:

```text
if Domain Administrator is logged in,
their credentials or hashes may be present in LSASS.
```

Concept:

```text

→ authenticates using hash
→ executes remote command
```

In this case:

```text

→ uses Domain Administrator hash
→ authenticates to dc01
→ runs command remotely
```

---

# What Is ntdsutil.exe?

`ntdsutil.exe` is a legitimate Windows administrative utility.

It can manage:

```text
Active Directory database
domain metadata
FSMO roles
directory maintenance
snapshots
database backups
```

Attackers can abuse it to dump:

```text
ntds.dit
SYSTEM hive
SECURITY hive
```

---

# Why ntds.dit Is Critical

`ntds.dit` contains Active Directory data, including:

```text
domain users
password hashes
group memberships
computer accounts
directory objects
```

With:

```text
ntds.dit
+ SYSTEM hive
```

an attacker may extract domain password hashes.

---

# Dump Destination

The attacker dumps files into:

```text
C:\Temp
```

Typical folders include:

```text
Active Directory
Registry
```

The exported data may contain:

```text
ntds.dit
SYSTEM
SECURITY
```

---

# Consequences

If the attacker obtains domain hashes:

```text
they may impersonate domain users
perform Pass-the-Hash
crack passwords offline
create persistence
compromise additional hosts
forge tickets
take over the domain
```

# Main Defender Lessons

## Logs Must Be Combined

One event alone may look harmless.

Examples:

```text
One 404 response = normal
250 404 responses in one second = scanning

One failed logon = normal
100 failures followed by success = brute force

PowerShell execution = may be normal
PowerShell accessing LSASS = highly suspicious
```
