# 5. Windows Client-Side Attacks

## Overview

Client-side attacks require some action from the user.

Examples:

- Opening a malicious attachment
- Enabling macros
- Clicking a malicious link
- Running a suspicious file

This module focuses on:

- Microsoft Office phishing attacks
- Malicious macros
- PowerShell payloads
- Sysmon artifacts
- PowerShell logging
- PowerShell obfuscation and deobfuscation

Main tools and logs:

- Microsoft Word
- Microsoft Outlook
- Sysmon
- PowerShell Operational logs
- PowerShell Module Logging
- PowerShell Script Block Logging
- PowerShell Transcription
- Event Viewer
- Meterpreter
- Evil-WinRM

# 5.1 Attacking Microsoft Office

## Social Engineering

Social engineering is the use of deception to trick users into doing something unsafe.

Examples:

- Giving away information
- Opening a malicious attachment
- Clicking a malicious link
- Enabling macros
- Running malicious code

Client-side attacks often depend on social engineering because the attacker needs user interaction.

---

## Spearphishing

Spearphishing is a targeted phishing attack.

The attacker sends a convincing email to a specific person or organization.

Common spearphishing types:

| Type | Description |
|---|---|
| User solicitation | Attacker asks the victim to reply with sensitive information |
| Malicious attachment | Attacker sends a weaponized document or file |
| Malicious link | Attacker sends a link to a malicious page or payload |
| Spearphishing via service | Attacker uses social media or messaging platforms |

MITRE ATT&CK mapping:

```text
Initial Access > Phishing > Spearphishing Attachment
Initial Access > Phishing > Spearphishing Link
Initial Access > Phishing > Spearphishing via Service
```

---

# Microsoft Office as an Attack Vector

Microsoft Office is widely used in enterprises.

Attackers abuse Office documents because users commonly open:

- Word documents
- Excel spreadsheets
- PowerPoint files
- Outlook email attachments

A malicious Office document may contain:

- VBA macros
- Encoded PowerShell
- Downloaders
- Reverse shell payloads

---

# Microsoft Office Macros

## What Are Macros?

Macros are grouped commands used to automate tasks in Office applications.

Office macros are commonly written in:

```text
VBA = Visual Basic for Applications
```

Macros can be used for legitimate automation, but attackers can also use them to run malicious code.

---

## Why Macros Are Dangerous

A malicious macro can:

- Start PowerShell
- Decode an encoded payload
- Download malware
- Create a reverse shell
- Connect back to an attacker
- Execute commands on the victim machine

---

## Protected View

Office opens files from untrusted sources in Protected View.

Protected View disables:

- Editing
- Macros
- Active content

The user must usually click:

```text
Enable Editing
Enable Content
```

This user action allows the malicious macro to run.

# Meterpreter Reverse Shell

The malicious Word macro launches a Meterpreter reverse HTTPS payload.

Listener script:

```bash
./wcsa_met_443.sh <attacker-ip>
```

Example:

```bash
./wcsa_met_443.sh 192.168.51.50
```

Important payload details:

| Field | Value |
|---|---|
| Payload | windows/x64/meterpreter/reverse_https |
| LPORT | 443 |
| LHOST | Attacker IP |
| Protocol | HTTPS |

Listener port:

```text
443
```

---

## Staged Payload

Meterpreter commonly uses a staged payload.

A staged payload works in parts:

1. First stage runs on victim.
2. Victim connects back to attacker.
3. Larger second stage is downloaded.
4. Full Meterpreter session starts.

This reduces the size of the initial payload.

# 5.2 Monitoring Windows PowerShell

## Why PowerShell Logging Matters

PowerShell is heavily used by:

- Administrators
- Attackers
- Malware
- Red teams
- Automation scripts

Because attackers often abuse PowerShell, Windows provides several logging mechanisms.

This module focuses on:

- Module Logging
- Script Block Logging
- Transcription Logging

---

# PowerShell Logging Location

PowerShell logs are stored in:

```text
Microsoft-Windows-PowerShell/Operational
```

This is the full LogName used with Get-WinEvent.

Example:

```powershell
Get-WinEvent -FilterHashtable @{
    LogName='Microsoft-Windows-PowerShell/Operational'
}
```


# PowerShell Module Logging

## What Is Module Logging?

Module logging records PowerShell pipeline execution.

It shows:

- Command invocations
- Parameter bindings
- Pipeline activity
- Cmdlets used
- Context information

To log all modules, set module name to:

```text
*
```

---

## Module Logging Event ID

PowerShell Module Logging uses:

```text
4103
```

Meaning:

```text
Pipeline execution event
```

---

## Example Module Logging Activity

Run:

```powershell
Get-WmiObject -Class Win32_Process |
Format-Table ProcessId, ParentProcessId, Name;
Write-Host (Get-Date)
```

Query module logs:

```powershell
Get-WinEvent -FilterHashtable @{
    LogName='Microsoft-Windows-PowerShell/Operational'
    StartTime="6/14/2021 13:25:52"
    EndTime="6/14/2021 13:25:54"
    ID=4103
}
```

---

## Module Logging Output

Module logging may show:

```text
CommandInvocation(Get-WmiObject)
ParameterBinding(Get-WmiObject): name="Class"; value="Win32_Process"
CommandInvocation(Format-Table)
ParameterBinding(Format-Table): name="Property"; value="ProcessId, ParentProcessId, Name"
```

It also records context such as:

- Host Name
- Host Application
- Engine Version
- Pipeline ID
- Sequence Number
- Command Name
- User

---

## Pipeline ID and Sequence Number

| Field | Meaning |
|---|---|
| Pipeline ID | Groups commands in the same pipeline |
| Sequence Number | Shows execution order |

These help reconstruct PowerShell activity.

---

## Module Logging EventData Payload Index

For Module Logging Event ID `4103`, EventData contains:

| Index | Field |
|---:|---|
| 0 | ContextInfo |
| 1 | UserData |
| 2 | Payload |

Payload index:

```text
2
```

Extract Payload:

```powershell
$_.Properties[2].Value
```

Example:

```powershell
Get-PSLogEvent 4103 "6/15/2021 15:44:00" "6/15/2021 15:44:59" |
Format-List TimeCreated,
@{Label = "Payload"; Expression = {$_.Properties[2].Value}}
```

# PowerShell Script Block Logging

## What Is Script Block Logging?

Script block logging records the actual PowerShell code that runs.

It can capture:

- Script block text
- Encoded commands after decoding
- Suspicious functions
- Obfuscated content
- Malware logic

---

## Script Block Logging Event ID

PowerShell Script Block Logging uses:

```text
4104
```

Meaning:

```text
Script block event
```

---

## Simple Script Block Example

Run:

```powershell
{ "This is a script block" }; Write-Host (Get-Date)
```

Query:

```powershell
Get-WinEvent -FilterHashtable @{
    LogName='Microsoft-Windows-PowerShell/Operational'
    StartTime="06/15/2021 14:49:42"
    EndTime="06/15/2021 14:49:44"
    ID=4104
} | Format-List
```

Output includes:

```text
Creating Scriptblock text:
{ "This is a script block" }; Write-Host (Get-Date)
```

# Encoded PowerShell

## Encoding a Command

PowerShell encoded commands use Unicode bytes and Base64.

Example command:

```powershell
$Command = 'Write-Host (Get-Date); Get-Hotfix'
$Bytes = [System.Text.Encoding]::Unicode.GetBytes($Command)
$EncodedCommand = [Convert]::ToBase64String($Bytes)
$EncodedCommand
```

Run encoded command:

```powershell
powershell -Encoded <base64>
```

or:

```powershell
powershell -e <base64>
```

---

## Script Block Logging Can Decode Encoded Commands

Even if the command is Base64-encoded, Event ID `4104` may show the decoded command.

Example decoded result:

```powershell
Write-Host (Get-Date); Get-Hotfix
```

This is why script block logging is very useful for defenders.

# PowerShell Transcription

## What Is Transcription Logging?

PowerShell transcription creates a text record of a PowerShell session.

It can include:

- Commands typed
- Command output
- Start time
- Username
- Machine name
- Host application
- PowerShell version
- Process ID
- Invocation headers

---

## Transcription Output Location

Default transcript files may be saved under the user’s Documents folder, often in a date folder:

```text
C:\Users\offsec\Documents\YYYYMMDD\
```

Transcript file format:

```text
PowerShell_transcript.HOSTNAME.UNIQUEID.YYYYMMDDHHMMSS.txt
```

---

## Transcription Header Fields

Important fields:

| Field | Meaning |
|---|---|
| Start time | When transcript started |
| Username | User running PowerShell |
| RunAs User | Execution identity |
| Machine | Hostname and OS |
| Host Application | Parent PowerShell host |
| Process ID | PowerShell process ID |
| PSVersion | PowerShell version |
| PSEdition | Desktop or Core |

Example PSEdition:

```text
Desktop
```

---

## Invocation Headers

Invocation headers add timestamps for each command.

Example:

```text
Command start time: 20210615142013
```

This helps correlate transcription logs with:

- Sysmon ProcessCreate
- Sysmon FileCreate
- Sysmon NetworkConnect
- PowerShell Operational logs

---

# Get-PSLogEvent Function

Create a helper function for PowerShell logs.

```powershell
function Get-PSLogEvent {
    param (
        $eventid,
        $start,
        $end
    )

    $filters = @{LogName = "Microsoft-Windows-PowerShell/Operational"}

    if ($eventid -ne $null) {
        $filters.ID = $eventid
    }

    if ($start -ne $null) {
        $filters.StartTime = $start
    }

    if ($end -ne $null) {
        $filters.EndTime = $end
    }

    Get-WinEvent -FilterHashtable $filters
}
```

Save as:

```text
C:\Sysmon\Get-PSLog.psm1
```

Import:

```powershell
Import-Module C:\Sysmon\Get-PSLog.psm1
```

Full LogName:

```text
Microsoft-Windows-PowerShell/Operational
```

# Case Study: PowerShell Logging for Phishing

## Attack Chain

The infected Word document causes this chain:

```text
Outlook
  -> WINWORD.EXE
      -> powershell.exe -nop -w hidden -e <base64>
          -> decoded PowerShell
              -> second-stage payload
                  -> Meterpreter reverse HTTPS
```

---

## Transcription Artifact

The transcript may show:

```text
Host Application:
C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe -nop -w hidden -e <base64>
```

This proves that PowerShell was launched with an encoded command.

---

## Script Block Logging Artifact

Script Block Logging may reveal decoded malicious functions.

Suspicious indicators:

```text
System.dll
kernel32.dll
VirtualAlloc
CreateThread
WaitForSingleObject
GetProcAddress
GetModuleHandle
Marshal.Copy
```

These are strong signs of in-memory payload execution.

---

## Why VirtualAlloc and CreateThread Are Suspicious

Malware may use Windows APIs to:

1. Allocate memory.
2. Copy shellcode into memory.
3. Start a new thread.
4. Execute payload in memory.

Common suspicious API names:

```text
VirtualAlloc
CreateThread
WaitForSingleObject
LoadLibrary
GetProcAddress
```

---

## Module Logging Artifact

Module Logging may show many events like:

```text
CommandInvocation(New-Object)
CommandInvocation(Where-Object)
ParameterBinding(New-Object)
```

Example suspicious type:

```text
System.Diagnostics.ProcessStartInfo
```

This indicates PowerShell is preparing to start another process.

---

# PowerShell Reflective Injection

Reflective PE injection loads executable code directly into memory.

Suspicious artifacts may include:

- Script block containing injected DLL
- PowerShell as ParentImage
- New process created by the script
- Sysmon ProcessCreate Event ID 1

Useful clue:

```text
For ProcessCreate, ParentImage will be powershell.exe
```

# 5.2.7 Obfuscating and Deobfuscating Commands

## Encoding vs Obfuscation

Encoding and obfuscation are not the same.

| Concept | Meaning |
|---|---|
| Encoding | Converts data into another format that can be decoded |
| Obfuscation | Makes code harder to read and understand |

Example encoding:

```text
Base64
```

Example obfuscation:

```text
Random case, backticks, string reordering, concatenation
```

---

# Invoke-Obfuscation

Invoke-Obfuscation is a PowerShell obfuscation framework.

It helps demonstrate how attackers hide PowerShell commands.

Import module:

```powershell
Import-Module ./Invoke-Obfuscation/Invoke-Obfuscation.psd1
```

Start:

```powershell

```

---

## Invoke-Obfuscation Main Menu Options

| Option | Purpose |
|---|---|
| TOKEN | Obfuscate PowerShell tokens |
| AST | Obfuscate AST nodes |
| STRING | Obfuscate command as a string |
| ENCODING | Obfuscate via encoding |
| COMPRESS | Convert command to one-liner and compress |
| LAUNCHER | Obfuscate command-line launch syntax |

Option that converts script block into one-liner format:

```text
COMPRESS
```

---

## Example Original Command

```powershell
Get-CimInstance Win32_ComputerSystem |
Select-Object -Property Name, PrimaryOwnerName, Domain, TotalPhysicalMemory, Model, Manufacturer;
Write-Host (Get-Date)
```

---

## Command Token Obfuscation

Example using backticks:

```powershell
gE`T-ci`mIns`TANce
s`ele`ct-OBJe`CT
wr`itE-`HoST
```

PowerShell still understands these commands.

---

## Argument Obfuscation

Example using reordered strings:

```powershell
("{5}{6}{4}{0}{2}{1}{3}"-f'Co','puterSy','m','stem','32_','Wi','n')
```

This reconstructs:

```text
Win32_ComputerSystem
```

---

## Why Obfuscation Is Dangerous

Obfuscation makes it harder to quickly understand what the command does.

However, logs may still show useful evidence:

- Script block event shows obfuscated command.
- Module logging can reveal actual invoked cmdlets and parameters.

Example:

```text
Script Block Log = may show obfuscated text
Module Log = may show Get-CimInstance and Select-Object
```

# Key Detection Ideas

## Malicious Office Documents

Look for:

- Word or Excel spawning PowerShell
- Office parent process with suspicious child process
- Encoded PowerShell
- Hidden PowerShell windows
- Macros enabled
- Documents opened from email cache paths

Suspicious chain:

```text
WINWORD.EXE -> powershell.exe
EXCEL.EXE -> powershell.exe
```

---

## Suspicious PowerShell

Look for command-line flags:

```text
-nop
-w hidden
-e
-enc
-EncodedCommand
```

Look for suspicious APIs:

```text
VirtualAlloc
CreateThread
WaitForSingleObject
GetProcAddress
GetModuleHandle
Marshal.Copy
```

---

## Suspicious Network Behavior

Look for:

- PowerShell making outbound connections
- Office-related child processes connecting to attacker IP
- Destination port `443`
- DNS query for attacker hostname
- Two-stage payload behavior

---

## Suspicious Temporary Files

Look for:

```text
C:\Users\<user>\AppData\Local\Temp\__PSScriptPolicyTest_*.ps1
```

These may be created during PowerShell execution.

---

## Suspicious HTA Execution

Look for:

```text
mshta.exe
```

especially when launched from:

- Email attachment
- Downloads folder
- Temp directory
- Outlook cache


---

# Key Takeaways

- Client-side attacks require user interaction.
- Microsoft Office documents are common phishing delivery methods.
- Macros can execute PowerShell payloads.
- Protected View helps prevent automatic execution, but users may enable content.
- Word or Excel spawning PowerShell is highly suspicious.
- Sysmon helps trace process, file, DNS, and network artifacts.
- PowerShell Module Logging records command and pipeline activity.
- PowerShell Script Block Logging records executed code and may decode encoded commands.
- PowerShell Transcription records full session input and output.
- Event ID `4103` is used for Module Logging.
- Event ID `4104` is used for Script Block Logging.
- Obfuscation hides intent but does not always hide behavior.
- Revoke-Obfuscation can help reconstruct obfuscated PowerShell from logs.
- Strong investigation requires correlating Office, Sysmon, PowerShell, DNS, and network artifacts.
