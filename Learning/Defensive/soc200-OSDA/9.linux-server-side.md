# 9. Linux Server Side Attacks

## Main Goal of This Chapter

This chapter explains common server-side attacks against Linux systems and how defenders can identify them using logs.

The main focus is:

```text
SSH credential abuse
Suspicious logins
Password brute force
Password spraying
Apache web logs
Command injection
Shellshock
SQL injection
ModSecurity audit logs
```

Important idea:

```text
Server-side attacks usually do not need user interaction.
The attacker sends traffic directly to an exposed service or vulnerable application.
```

---

# 9.1 Credential Abuse

## What Is Credential Abuse?

Credential abuse means an attacker tries to access a system using usernames, passwords, or SSH keys.

Examples:

```text
Trying many passwords against one SSH account
Trying one password against many users
Using stolen SSH private keys
Logging in from unusual IP addresses
Logging in at unusual times
```

For Linux servers, the most important service in this chapter is:

```text
SSH
```

---

## Why SSH Is Important

SSH is commonly used to administer Linux servers remotely.

SSH allows:

```text
Remote shell access
Remote command execution
Secure file transfer
Administration over encrypted channels
```

The SSH server daemon is:

```text
sshd
```

The default SSH port is:

```text
22/TCP
```

---

## Important SSH Log Files

On CentOS / RHEL:

```text
/var/log/secure
```

On Ubuntu / Debian:

```text
/var/log/auth.log
```

Useful commands:

```bash
sudo grep "sshd" /var/log/secure
```

```bash
sudo grep "sshd" /var/log/auth.log
```

# 9.1.1 Suspicious SSH Logins

## What Is a Suspicious Login?

A suspicious login is a login that does not match expected user or system behavior.

Examples:

```text
Login from a new IP address
Login outside normal working hours
Password login for a user who normally uses SSH keys
Many failed attempts before a success
Root login attempt
Login from an unexpected country or network
```

Important:

```text
A successful login is not automatically safe.
A failed login is not automatically malicious.
Context matters.
```

---

## Successful Password Login

Example log:

```text
Accepted password for offsec from 192.168.51.50 port 55398 ssh2
```

Meaning:

```text
User offsec successfully authenticated with a password.
The source IP was 192.168.51.50.
The source port was 55398.
The protocol was SSH version 2.
```

Important keyword:

```text
Accepted password
```

---

## Failed Password Login

Example log:

```text
Failed password for offsec from 192.168.51.50 port 55400 ssh2
```

Meaning:

```text
Someone tried to log in as offsec.
The password was wrong.
The attempt came from 192.168.51.50.
```

Important keyword:

```text
Failed password
```

---

## Successful Public-Key Login

Example log:

```text
Accepted publickey for offsec from 192.168.51.50 port 55420 ssh2: ED25519 SHA256:<fingerprint>
```

Meaning:

```text
User offsec successfully authenticated with an SSH key.
The key type was ED25519.
The key fingerprint is visible in the log.
```

Important keyword:

```text
Accepted publickey
```

Important field:

```text
SHA256 key fingerprint
```

Why the fingerprint matters:

```text
It helps identify which SSH key was used.
A new or unknown fingerprint can be suspicious.
```

---

## Failed Public-Key Login

Example log:

```text
Connection closed by authenticating user offsec 192.168.51.50 port 55430 [preauth]
```

Meaning:

```text
The connection closed before authentication completed.
```

Important keyword:

```text
[preauth]
```

With more verbose SSH logging, failed public-key events can be clearer.

Example:

```text
Failed publickey for offsec from 192.168.51.50 port 55448 ssh2: ED25519 SHA256:<fingerprint>
```

---

## SSH LogLevel

SSH logging is configured in:

```text
/etc/ssh/sshd_config
```

Default:

```text
LogLevel INFO
```

More verbose:

```text
LogLevel DEBUG1
```

Restart SSH after changing configuration:

```bash
sudo systemctl restart sshd
```

Important:

```text
DEBUG1 gives more visibility, especially for public-key authentication failures.
But it also increases log volume.
```

---

## SSH Login Keywords

| Event Type | Important Keyword |
|---|---|
| Successful password login | `Accepted password` |
| Failed password login | `Failed password` |
| Successful public-key login | `Accepted publickey` |
| Failed public-key login | `Failed publickey` or `[preauth]` |
| Session opened | `session opened` |
| Authentication failure | `authentication failure` |

# 9.1.2 Hunting Suspicious SSH Logins

## Manual SSH Hunting

Show successful SSH password logins:

```bash
sudo grep "Accepted password" /var/log/secure
```

Show failed SSH password logins:

```bash
sudo grep "Failed password" /var/log/secure
```

Show successful public-key logins:

```bash
sudo grep "Accepted publickey" /var/log/secure
```

Show SSH logs from one source IP:

```bash
sudo grep "sshd" /var/log/secure | grep "192.168.51.50"
```

Show SSH logs for one user:

```bash
sudo grep "sshd" /var/log/secure | grep "offsec"
```

---

## Why Use a Parser?

Raw SSH logs can be noisy.

A parser helps extract only the important fields:

```text
timestamp
username
source IP
authentication method
success or failure
SSH key fingerprint
```

This helps answer:

```text
Who logged in?
From where?
Was it successful?
Was it password or public key?
Was there a failure before success?
```

---

## Useful Detection Ideas

### Many Failures From One IP

```text
One source IP creates many Failed password events.
```

Possible meaning:

```text
Password brute force
Credential stuffing
Automated attack
```

---

### Many Users From One IP

```text
One source IP tries many usernames.
```

Possible meaning:

```text
Password spraying
User enumeration
Automated attack
```

---

### Failure Followed by Success

```text
Failed password
Failed password
Failed password
Accepted password
```

Possible meaning:

```text
The attacker guessed the correct password.
```

---

### Public-Key Fingerprint Change

```text
Same user logs in with a new SSH key fingerprint.
```

Possible meaning:

```text
New legitimate key
Stolen key
Unauthorized key added to authorized_keys
```

# 9.1.3 Password Brute Force

## What Is Password Brute Force?

Password brute force means:

```text
One username
+
many password attempts
```

Example:

```text
admin + password1
admin + password2
admin + password3
admin + password4
```

In logs, it usually looks like:

```text
Many Failed password events for the same user from the same IP.
```

---

## Typical Brute Force Log Pattern

Example:

```text
Failed password for offsec from 192.168.51.50 port 55762 ssh2
Failed password for offsec from 192.168.51.50 port 55763 ssh2
Failed password for offsec from 192.168.51.50 port 55764 ssh2
```

Sometimes SSH logs may show:

```text
maximum authentication attempts exceeded
```

or:

```text
Too many authentication failures [preauth]
```

Important indicators:

```text
same username
same source IP
many failures
short time window
```

---

## Successful Brute Force Pattern

Example:

```text
Failed password for offsec from 192.168.51.50
Failed password for offsec from 192.168.51.50
Failed password for offsec from 192.168.51.50
Accepted password for offsec from 192.168.51.50
```

Meaning:

```text
The attacker tried multiple wrong passwords.
Then one password worked.
The attacker may now have access.
```

This is more serious than failed attempts only.

---

## What Is Password Spraying?

Password spraying is different from brute force.

Password spraying means:

```text
Many usernames
+
one or a few common passwords
```

Example:

```text
alice + Welcome2026
bob   + Welcome2026
admin + Welcome2026
test  + Welcome2026
```

The goal is to avoid account lockout.

---

## Brute Force vs Password Spraying

| Attack Type | Pattern | Example |
|---|---|---|
| Brute force | One user, many passwords | `admin` + many passwords |
| Password spraying | Many users, one password | many users + `Welcome2026` |
| Credential stuffing | Known leaked credentials | username/password pairs from leaks |

---

## Password Spraying Log Pattern

Example:

```text
Failed password for alice from 192.168.51.50
Failed password for bob from 192.168.51.50
Failed password for admin from 192.168.51.50
Failed password for test from 192.168.51.50
```

Important indicators:

```text
same source IP
many different usernames
short time window
same authentication method
```


Timestamp.

```text
"GET / HTTP/1.1"
```



User-Agent.

| Status Code | Meaning |
|---|---|
| 200 | OK / request successful |
| 301 | Permanent redirect |
| 302 | Temporary redirect |

# 9.2.1 Command Injection

## What Is Command Injection?

Command injection happens when user input is passed to the operating system without proper validation.

Simple idea:

```text
User input becomes part of an OS command.
```

Example concept:

```text
Application expects:
ping 8.8.8.8

Attacker provides:
8.8.8.8; whoami
```

Result:

```text
ping 8.8.8.8
whoami
```

Possible impact:

```text
Run shell commands
Read files
Download tools
Start reverse shells
Modify the system
Create persistence
```

---

## Web Server User Context

On Linux, a vulnerable web application usually runs commands as the web server user.

Common Apache user on Ubuntu:

```text
www-data
```

Common Apache user on CentOS:

```text
apache
```

Important:

```text
If command injection succeeds, the attacker usually gets the privileges of the web server user.
```

# 9.2.2 Shellshock

## What Is Shellshock?

Shellshock is a Bash vulnerability.

The classic payload pattern is:

```text
() { :;};
```

It abuses how vulnerable Bash versions process function definitions from environment variables.

---

## Why Shellshock Affects Web Servers

Shellshock can affect CGI-based web applications.

CGI means:

```text
Common Gateway Interface
```

CGI applications often pass HTTP headers into environment variables.

Attack flow:

```text
Attacker sends malicious HTTP header
→ Apache passes header into CGI environment
→ vulnerable Bash processes the environment variable
→ attacker command executes
```

---

## Shellshock Payload Locations

The Shellshock payload can appear in different HTTP headers:

```text
User-Agent
Referer
Cookie
Custom headers
```

Default Apache logs usually show:

```text
User-Agent
Referer
```

But they may not show:

```text
Cookie
POST body
all custom headers
```

---

## Shellshock Apache Log Example

```text
192.168.51.50 - - [02/Aug/2021:03:59:03 -0400] "GET /cgi-bin/index.cgi HTTP/1.1" 200 151 "() { :;}; /bin/bash -c ..." "-"
```

Important artifacts:

```text
Source IP: 192.168.51.50
Path: /cgi-bin/index.cgi
Status: 200
Payload: () { :;};
```

Meaning:

```text
The attacker sent a Shellshock payload.
The request reached a valid CGI endpoint.
The server returned HTTP 200.
```

---

## CGI Path Probing

Attackers may try many CGI paths.

Example:

```text
GET /cgi-sys/defaultwebpage.cgi 404
GET /cgi-mod/index.cgi 404
GET /cgi-bin/test.cgi 404
GET /cgi-bin/index.cgi 200
```

Meaning:

```text
The attacker tried multiple possible CGI paths.
Most did not exist.
One valid path was found.
```

Important pattern:

```text
many 404s followed by one 200
```

---

## Process Evidence After Shellshock

After successful command injection, check processes:

```bash
ps aux | grep bash
```

Suspicious example:

```text
www-data   26016  ... /bin/bash
```

Why this is suspicious:

```text
The Apache user should not normally spawn an interactive Bash shell.
```

---

## Detecting Shellshock with Regex

Example regex:

```python
shellshock_regex = '\(\)\s*\t*\{.*;\s*\}\s*;'
```

Meaning:

```text
Look for the pattern:
() { ... ; };
```

Important parts:

```text
\s*
```

Matches zero or more whitespace characters.

```text
\t*
```

Matches zero or more tab characters.

---

## Shellshock Detection Logic

Simple logic:

```text
Read Apache logs.
Look for Shellshock pattern.
Check the HTTP status code.
Check the requested path.
Check whether the request hit CGI.
```

Example interpretation:

```text
Shellshock pattern + 404 = attempt / probing
Shellshock pattern + 200 = more suspicious, likely reached valid endpoint
Shellshock pattern + www-data spawning bash = strong compromise evidence
```

---

## Cookie-Based Shellshock

Default Apache logs may not show Cookie headers.

Example default access log:

```text
192.168.51.50 - - [02/Aug/2021:07:57:06 -0400] "GET /cgi-bin/index.cgi HTTP/1.1" 200 18 "-" "-"
```

Problem:

```text
The attack payload may be hidden in the Cookie header.
The default access log may not show it.
```

---

## Improve Apache Logging for Cookies

Apache can be configured to log cookies.

Example custom log format:

```apache
LogFormat "%h %l %u %t \"%r\" %>s %b \"%{Referer}i\" \"%{User-Agent}i\" \"%{Cookie}i\"" with_cookies
CustomLog /var/log/apache2/with_cookies.log with_cookies
```

This creates a log that includes:

```text
Cookie header
```

---

## Cookie-Based Shellshock Log Example

```text
192.168.51.50 - - [02/Aug/2021:07:57:06 -0400] "GET /cgi-bin/index.cgi HTTP/1.1" 200 18 "-" "-" "() { :;}; /bin/bash -c ..."
```

Field meaning:

```text
Referer: "-"
User-Agent: "-"
Cookie: "() { :;}; ..."
```

Meaning:

```text
The attacker hid the Shellshock payload in the Cookie header.
Custom logging made it visible.
```

# 9.2.3 SQL Injection

## What Is SQL Injection?

SQL injection happens when user input changes the SQL query executed by the application.

Simple idea:

```text
Application expects normal input.
Attacker provides SQL syntax.
The database executes the modified query.
```

Possible impact:

```text
Read unauthorized data
Dump usernames and passwords
Bypass authentication
Modify database records
Delete data
Extract information blindly
```

---

## Normal Query Concept

Application request:

```text
/tshirts?color=purple
```

SQL query concept:

```sql
SELECT * FROM tshirts WHERE color = 'purple';
```

Meaning:

```text
Return all shirts where color is purple.
```

---

## UNION-Based SQL Injection

Attacker input concept:

```text
purple' UNION SELECT username,password FROM administrators--
```

SQL query concept:

```sql
SELECT * FROM tshirts WHERE color = 'purple'
UNION SELECT username,password FROM administrators;
```

Meaning:

```text
The attacker adds another SELECT query.
The application may return data from another table.
```

---

## In-Band SQL Injection

In-band SQL injection means:

```text
The attacker sees the database output directly in the web response.
```

Example:

```text
The page displays usernames and password hashes.
```

---

## Blind SQL Injection

Blind SQL injection means:

```text
The attacker does not directly see database output.
```

Instead, the attacker infers information from behavior.

Common blind SQL injection types:

```text
Boolean-based blind SQL injection
Time-based blind SQL injection
```

---

## Boolean-Based Blind SQL Injection

The attacker asks true/false questions.

Example:

```text
scoop' AND 1=1
```

This is true.

Example:

```text
scoop' AND 1=2
```

This is false.

If the page behaves differently, the attacker learns information.

---

## Time-Based Blind SQL Injection

The attacker uses a delay function.

Example concept:

```text
scoop' AND IF(1=1, sleep(20), false)
```

If the response is delayed, the attacker knows the condition was true.

Common delay functions:

```text
SLEEP()
PG_SLEEP()
WAITFOR DELAY
BENCHMARK()
```

---

## WordPress Plugin Example

The chapter uses a WordPress plugin example:

```text
Simply Poll 1.4.1
```

Attack type:

```text
Time-based blind SQL injection
```

Important endpoint:

```text
/wp-admin/admin-ajax.php
```

Important action:

```text
spAjaxResults
```

Important vulnerable parameter:

```text
pollid
```

---

## sqlmap

`sqlmap` is a common SQL injection automation tool.

Defensive indicators:

```text
User-Agent contains sqlmap
Repeated requests
SQL payloads in parameters
Time-delay functions
Many POST requests
ModSecurity alerts
403 responses
```

Example User-Agent:

```text
sqlmap/1.5.7#stable
```

---

## Apache Access Log Limitation for SQL Injection

Apache access logs may show:

```text
POST /wp-admin/admin-ajax.php HTTP/1.1
```

But the SQL injection payload may be inside the POST body.

Default Apache logs usually do not include POST body content.

So access logs may only show:

```text
repeated POST requests
same endpoint
same source IP
same User-Agent
```

This is suspicious, but not always enough to see the full attack.

# ModSecurity

## What Is ModSecurity?

ModSecurity is a web application firewall module.

It can inspect HTTP traffic and detect suspicious patterns.

It can log:

```text
request headers
request body
rule matches
blocked requests
SQL injection attempts
command injection attempts
```

Important log file:

```text
/var/log/apache2/modsec_audit.log
```

---

## ModSecurity Audit Log Sections

Important sections:

```text
-A-- = audit log header
-B-- = request headers
-C-- = request body
-F-- = response headers
-H-- = audit trailer / rule matches / messages
```

Useful command:

```bash
sudo cat /var/log/apache2/modsec_audit.log | awk '/-A--/,/-F--/'
```

---

## SQL Injection Evidence in ModSecurity

Example request details:

```text
POST /wp-admin/admin-ajax.php HTTP/1.1
User-Agent: sqlmap/1.5.7#stable
Content-Type: application/x-www-form-urlencoded
```

Example request body:

```text
action=spAjaxResults&pollid=<SQLi payload>
```

Important artifacts:

```text
pollid
sqlmap User-Agent
PG_SLEEP
403 Forbidden
ModSecurity warning
```

Meaning:

```text
The vulnerable parameter is pollid.
The request was probably automated by sqlmap.
The payload used time-based SQL injection.
ModSecurity detected or blocked the request.
```

---

## libinjection

ModSecurity can use:

```text
libinjection
```

`libinjection` detects SQL injection by analyzing input structure.

This is stronger than only searching for keywords such as:

```text
UNION
SELECT
SLEEP
```

because attackers can obfuscate SQL keywords.

---

# SQL Injection Detection Ideas

Look for:

```text
SQL keywords in parameters
URL-encoded quotes
Time-delay functions
Repeated POST requests
sqlmap User-Agent
ModSecurity warnings
403 responses
Database error messages
Suspicious parameter values
```

Useful SQLi keywords:

```text
UNION
SELECT
FROM
WHERE
AND
OR
SLEEP
PG_SLEEP
WAITFOR
BENCHMARK
INFORMATION_SCHEMA
--
'
"
%27
%22
```

Meaning:

```text
%27 = URL-encoded single quote
%22 = URL-encoded double quote
```

# Access Log vs ModSecurity Log

| Log Source | What It Shows | Limitation |
|---|---|---|
| Apache access log | Source IP, method, path, status, User-Agent | Usually no POST body |
| Apache error log | Web server errors | May not show full payload |
| ModSecurity audit log | Headers, body, rule matches | Requires ModSecurity |
| Application logs | Application behavior | Depends on the app |
| Database logs | Queries and errors | Often not enabled by default |

# Python Log Parsing

## Apache Regex Example

```python
web_log_regex = '([(\d\.)]+) - - \[(.*?)\] \"(.*?)\" (\d+) (\d+) \"(.*?)\" \"(.*?)\"'
```

This extracts:

```text
source IP
timestamp
request
status code
response size
Referer
User-Agent
```

---

## Apache Parsed Tuple Indexes

Example tuple:

```python
(
  '192.168.51.50',
  '07/Jul/2021:05:34:12 -0400',
  'GET / HTTP/1.1',
  '403',
  '199691',
  '-',
  'Mozilla/5.0'
)
```

Indexes:

```text
log_line[0] = source IP
log_line[1] = timestamp
log_line[2] = request
log_line[3] = status code
log_line[4] = response size
log_line[5] = Referer
log_line[6] = User-Agent
```

If Cookie logging is added:

```text
log_line[7] = Cookie
```

# Investigation Workflow

## SSH Credential Abuse Workflow

```text
1. Identify the correct SSH log file.
   CentOS/RHEL: /var/log/secure
   Ubuntu/Debian: /var/log/auth.log

2. Filter for sshd events.

3. Look for:
   Accepted password
   Failed password
   Accepted publickey
   Failed publickey
   Too many authentication failures
   [preauth]

4. Group by:
   username
   source IP
   authentication method
   time window

5. Investigate:
   failures followed by success
   many users from one IP
   many failures for one user
   new public-key fingerprints
   unusual login times
```

---

## Web Attack Workflow

```text
1. Review Apache access logs.

2. Look for:
   suspicious paths
   many 404 responses
   200 after probing
   unusual User-Agent
   repeated POST requests
   suspicious Referer
   suspicious Cookie

3. If POST body is needed:
   inspect ModSecurity audit logs.

4. For command injection:
   search for shell syntax and OS commands.

5. For SQL injection:
   search for SQL keywords, encoded quotes, delay functions, and ModSecurity warnings.

6. Correlate with process activity:
   web server user spawning shell
   www-data running /bin/bash
   unexpected outbound connections
```

# Key Indicators Table

| Attack | Log Source | Important Indicator |
|---|---|---|
| SSH failed password | `/var/log/auth.log` or `/var/log/secure` | `Failed password` |
| SSH successful password | SSH auth logs | `Accepted password` |
| SSH successful public key | SSH auth logs | `Accepted publickey` |
| SSH failed public key | SSH auth logs | `Failed publickey` or `[preauth]` |
| Brute force | SSH auth logs | Many failures for one user |
| Password spraying | SSH auth logs | Many users from one IP |

| Shellshock via Cookie | Custom Apache logs | Payload in Cookie header |
| Command injection success | Process list / EDR | Web user running shell |
| SQL injection | ModSecurity | SQLi warning |
| Time-based SQLi | ModSecurity | `SLEEP`, `PG_SLEEP` |
| sqlmap | Apache or ModSecurity | `sqlmap` User-Agent |

# Main Takeaways

```text
Linux server-side attacks often target exposed services like SSH and web applications.

SSH logs are critical for detecting credential abuse.

Accepted password means successful password login.
Failed password means failed password login.
Accepted publickey means successful key-based login.

Brute force usually means many passwords against one user.
Password spraying usually means one password against many users.

Apache access logs are useful, but they do not always show full payloads.

Default Apache logs may miss POST bodies, Cookie headers, and custom headers.

Shellshock can often be detected by the pattern:
() { :;};
```
