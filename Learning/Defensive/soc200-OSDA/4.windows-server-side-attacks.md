# 4. Windows Server Side Attacks

## Overview

Server-side attacks target services or applications running on a server. The attacker sends exploit code, malicious requests, or remote commands directly to a vulnerable service.

Unlike client-side attacks, server-side attacks usually do **not** require user interaction. If the service is vulnerable or misconfigured, the attacker may quickly gain access.

This module focuses on three major areas:

- Credential abuse
- Web application attacks
- Binary exploitation

Defenders investigate these attacks using:

- Windows Security logs
- Sysmon logs
- IIS logs
- PowerShell
- Event Viewer
- Windows Defender Exploit Guard

---

# 4.1 Credential Abuse

## Windows Authentication Basics

Windows does not store plaintext passwords. Instead, it stores password hashes.

A password hash is created by passing a password through a cryptographic hash function.

During authentication:

1. The user enters a password.
2. Windows hashes the entered password.
3. Windows compares it with the stored hash.
4. If the hashes match, access is granted.

---

## Security Account Manager (SAM)

The **Security Account Manager**, or SAM, stores local user password hashes.

Location conceptually:

```text
SAM database = local Windows password hash store
```

Important points:

- Windows does not store plaintext passwords.
- Passwords are stored as hashes.
- Older Windows versions used LM and NTLM hashes.
- Modern Windows disables LM by default and uses NTLM.
- NTLM is stronger than LM but still unsalted in the SAM database.

---

## LM vs NTLM

| Hash Type | Notes |
|---|---|
| LM | Very weak, converts passwords to uppercase, splits passwords into chunks |
| NTLM | Stronger, case-sensitive, supports Unicode, does not split passwords |
| NTLM in SAM | Still not salted |


LM is the old Windows password hash method and is very weak. It converts passwords to uppercase, splits them into small parts, and does not use salt, so attackers can crack LM hashes much faster.

NTLM is newer and stronger. It is case-sensitive, supports more characters, and does not split the password. However, NTLM hashes are still valuable to attackers because they are usually unsalted, meaning the same password creates the same hash.
---

## Local Authentication Flow

For local authentication:

1. User submits username and password.
2. Local Security Authority, or LSA, hashes the password.
3. LSA compares the hash with the SAM database.
4. If valid, Windows grants access based on the account’s permissions.

# Suspicious Logins

## Why Timing Matters

Defenders often investigate logons based on expected behavior.

Example normal business hours:

```text
Monday to Friday
07:00 - 19:00
```

Suspicious activity may include:

- Logons outside business hours
- Administrator logons at unusual times
- RDP logons from unexpected IP addresses
- Logons from unusual workstations
- Multiple failed logons followed by success

---

## Important Logon Event IDs

| Event ID | Meaning |
|---|---|
| 4624 | Successful logon |
| 4625 | Failed logon |
| 4634 | Logoff |
| 4800 | Workstation locked |
| 4801 | Workstation unlocked |

---

## Event ID 4624: Successful Logon

Event ID `4624` means:

```text
An account was successfully logged on.
```

Important fields:

| Field | Meaning |
|---|---|
| Account Name | User account that logged on |
| Account Domain | Local host or domain |
| Logon Type | Type of logon |
| Logon ID | Unique session identifier |
| Source Network Address | Source IP address |
| Workstation Name | Source workstation |

---

## Important Logon Types

| Logon Type | Meaning |
|---|---|
| 3 | Network logon |
| 7 | Unlock |
| 10 | RemoteInteractive / RDP |

### Logon Type 10

Logon Type `10` means:

```text
RemoteInteractive
```

Commonly associated with:

```text
RDP
```

### Logon Type 3

Logon Type `3` means:

```text
Network-based authentication
```

Examples:

- SMB
- WinRM
- PowerShell remoting
- RDP Network Level Authentication, or NLA

---

## Investigating Suspicious Logons with PowerShell

Example: search successful logons during off-hours:

```powershell
Get-WinEvent -FilterHashtable @{
    LogName='Security'
    StartTime='4/30/2021 19:00:00'
    EndTime='5/3/2021 07:00:00'
    ID=4624
}
```

Show full details:

```powershell
Get-WinEvent -FilterHashtable @{
    LogName='Security'
    StartTime='5/1/2021 03:21:26'
    EndTime='5/1/2021 03:21:27'
    ID=4624
} | Format-List
```

---

## Custom Get-SecurityEvent Function

To avoid long one-liners, create a reusable function:

```powershell
function Get-SecurityEvent {
    param (
        $eventid,
        $start,
        $end
    )

    $filters = @{LogName = "Security"}

    if ($eventid -ne $null) {
        $filters.ID = $eventid
    }

    if ($start -ne $null) {
        $filters.StartTime = $start
    }

    if ($end -ne $null) {
        $filters.EndTime = $end
    }

    Get-WinEvent -FilterHashtable $filters
}
```

Save as:

```text
C:\Sysmon\Get-Security.psm1
```

Import it:

```powershell
Import-Module C:\Sysmon\Get-Security.psm1
```

Confirm import:

```powershell
Get-Module | Where-Object { $_.ModuleType -eq "Script" }
```

---

## Event ID 4634: Logoff

Event ID `4634` means:

```text
An account was logged off.
```

To find the matching logoff for a suspicious logon, use the same `Logon ID`.

Example:

```powershell
Get-SecurityEvent 4634 "5/1/2021 03:21:26" "5/3/2021 07:00:00" |
Where-Object { $_.Properties[3].Value -eq 0x323466 } |
Format-List
```

# Brute Force Logins

## What Is a Brute Force Attack?

A brute force password attack is a repeated authentication attempt using many possible passwords.

In this module, Hydra is used to brute force RDP authentication.

Example attack flow:

1. Create password list.
2. Run Hydra against RDP.
3. Generate many failed logon events.
4. Eventually identify successful authentication if password is found.

---

## Hydra Example

Password list example:

```text
123456
password
admin
lab
```

Run Hydra script:

```bash
./run_hydra.sh 192.168.51.11 hydra.txt
```

---

## Event ID 4625: Failed Logon

Event ID `4625` means:

```text
An account failed to log on.
```

PowerShell query:

```powershell
Get-SecurityEvent 4625 "5/6/2021 00:00:00" "5/7/2021 00:00:00"
```

or:

```powershell
Get-WinEvent -FilterHashtable @{
    LogName='Security'
    ID=4625
}
```

---

## Important 4625 EventData Fields

For failed logon event `4625`, useful fields include:

| Field | Properties Index | Meaning |
|---|---:|---|
| TargetUserName | 5 | User account being attacked |
| TargetDomainName | 6 | Domain or host |
| Status | 7 | General failure code |
| SubStatus | 9 | More specific failure code |
| LogonType | 10 | Type of logon attempt |
| WorkstationName | 13 | Source workstation |
| IpAddress | 19 | Source IP address |

Source IP index:

```powershell
$_.Properties[19].Value
```

---

## Common Failed Logon Status Codes

| Code | Meaning |
|---|---|
| C000006D | Bad username or authentication information |
| C000006A | Username correct, password incorrect |
| C000006F | User logon outside authorized hours |

In brute force attacks using NLA, the useful detail is often in `SubStatus`.

Example:

```text
Status:    C000006D
SubStatus: C000006A
```

This means the username is valid but the password is wrong.

---

## Failed Logon Formatting Example

```powershell
Get-SecurityEvent 4625 "5/6/2021 00:00:00" "5/7/2021 00:00:00" |
Format-List TimeCreated,
@{Label = "Logon Type"; Expression = {$_.Properties[10].Value}},
@{Label = "Status"; Expression = {'{0:X8}' -f $_.Properties[7].Value}},
@{Label = "Substatus"; Expression = {'{0:X8}' -f $_.Properties[9].Value}},
@{Label = "Target User Name"; Expression = {$_.Properties[5].Value}},
@{Label = "Workstation Name"; Expression = {$_.Properties[13].Value}},
@{Label = "IP Address"; Expression = {$_.Properties[19].Value}}
```

---

## Successful Logon After Brute Force

After many failed `4625` events, defenders should search for successful `4624` events from the same source IP.

Example:

```powershell
Get-SecurityEvent 4624 "5/6/2021 09:36:44" "5/6/2021 09:37:44" |
Where-Object { $_.Properties[18].Value -eq "192.168.51.50" }
```

For successful logons, the source IP is commonly at:

```powershell
$_.Properties[18].Value
```

For failed logons, the source IP is commonly at:

```powershell
$_.Properties[19].Value
```

---

## Brute Force Detection Indicators

Suspicious brute force signs:

- Many `4625` events in a short time
- Same target username
- Same source IP
- Same workstation name
- Repeated Logon Type `3`
- One `4624` success shortly after many failures

# Local File Inclusion, LFI

## What Is LFI?

Local File Inclusion allows an attacker to read local files from the server through a vulnerable web application.

Example target file:

```text
C:\inetpub\wwwroot\wordpress\wp-config.php
```

This file may contain database credentials.

---

## WordPress Config File

WordPress stores important settings in:

```text
wp-config.php
```

Sensitive fields include:

```php
define( 'DB_NAME', 'wordpress' );
define( 'DB_USER', 'wpadmin' );
define( 'DB_PASSWORD', 'wppassword1' );
```

The database password field name is:

```text
DB_PASSWORD
```

---

## LFI IIS Log Indicators

Example IIS log entry:

```text
GET /wordpress/wp-content/plugins/site-import/admin/page.php?url=/inetpub/wwwroot/wordpress/wp-config.php
```

Suspicious indicators:

- Access to plugin path
- File path in URL query
- Traversal patterns like `../`
- Request for sensitive files
- Non-browser user-agent
- Python user-agent

Example suspicious user-agent:

```text
Python-urllib
```

---

## LFI Defensive Takeaways

LFI may expose:

- Configuration files
- Plaintext credentials
- Application secrets
- Database passwords

MITRE ATT&CK mapping:

```text
Credential Access > Unsecured Credentials > Credentials in Files
```

# Command Injection

## What Is Command Injection?

Command injection occurs when user-controlled input is passed to the operating system shell.

The attacker can append or inject commands that execute on the server.

Example:

```text
t.co|systeminfo
```

The pipe character `|` allows command chaining.

---

## Plainview Activity Monitor Vulnerability

The vulnerable WordPress plugin allows authenticated users to execute commands through an input field.

The vulnerable page uses:

```text
page=plainview_activity_monitor&tab=activity_tools
```

Important query parameter:

```text
tab=activity_tools
```

---

## Command Injection via Web Interface

Example command:

```text
t.co|whoami
```

or:

```text
t.co|systeminfo
```

The web interface field has a limited width:

```text
15 characters
```

This limits manual command injection through the browser.

---

## IIS Artifacts for Command Injection

IIS logs may show:

```text
POST /wordpress/wp-admin/admin.php page=plainview_activity_monitor&tab=activity_tools
```

Important fields:

- `POST` method
- vulnerable plugin path
- source IP
- referer
- long `time-taken`

Note:

```text
IIS logs usually do not show POST body content.
```

So IIS may prove that a POST occurred, but not always the exact command.

---

## Sysmon Artifacts for Command Injection

Command injection often creates Sysmon ProcessCreate events.

Sysmon Event ID:

```text
1 = Process Create
```

Useful ProcessCreate fields:

| Field | Properties Index | Meaning |
|---|---:|---|
| ProcessId | 3 | PID |
| CommandLine | 10 | Command that ran |
| User | 12 | User context |
| ParentProcessId | 19 | Parent PID |
| ParentImage | 20 | Parent process image |

Example query:

```powershell
Get-SysmonEvent 1 "05/10/2021 16:02:33" "05/10/2021 16:02:35" |
Format-List TimeCreated,
@{Label = "CommandLine"; Expression = {$_.Properties[10].Value}},
@{Label = "User"; Expression = {$_.Properties[12].Value}},
@{Label = "ParentImage"; Expression = {$_.Properties[20].Value}}
```

---

## Command Injection Process Chain

Example chain:

```text
php-cgi.exe
  -> cmd.exe
      -> dig
      -> systeminfo
```

In the lab, commands executed through WordPress run as:

```text
NT AUTHORITY\IUSR
```

This is the IIS user context.

MITRE ATT&CK mapping:

```text
Execution > Command and Scripting Interpreter > Windows Command Shell
```

Attack chain:

1. Upload `stage.bat`
2. Execute `stage.bat`
3. `stage.bat` downloads `load.ps1`
4. `load.ps1` downloads `nc.exe`
5. `nc.exe` connects back to attacker
6. Attacker receives shell

```powershell

```

# 4.3 Binary Exploitation

## Overview

Binary exploitation attacks target vulnerabilities in compiled applications.

In this module, the example is a buffer overflow in:

```text
SyncBreeze Enterprise 10.0.28
```

The vulnerable service listens on:

```text
Port 8080
```

The exploit creates a reverse shell back to the attacker on:

```text
Port 4444
```

---

## Buffer Overflow Concept

A buffer overflow occurs when too much data is sent to a memory buffer.

If successful, the attacker may:

- Corrupt memory
- Redirect execution flow
- Execute attacker-controlled payloads
- Gain shell access

# Windows Defender Exploit Guard, WDEG

## What Is WDEG?

Windows Defender Exploit Guard provides protection and auditing against exploit behavior.

It was added to Windows Defender in 2017.

It helps defend against:

- Fileless malware
- Exploit techniques
- Memory exploitation
- Suspicious application behavior

---

## WDEG Major Components

| Component | Purpose |
|---|---|
| Attack Surface Reduction, ASR | Blocks risky behavior, especially Office, scripts, and fileless malware |
| Controlled Folder Access | Protects folders from unauthorized modification |
| Network Protection | Blocks access to malicious domains/IPs |
| Exploit Protection | Applies process-level exploit mitigations |

---

## Exploit Protection

Exploit Protection can apply mitigations to specific processes.

For SyncBreeze, the relevant process is:

```text
C:\Program Files (x86)\Sync Breeze Enterprise\bin\syncbrs.exe
```

---

## Return-Oriented Programming, ROP

ROP is a technique used to bypass memory protections such as DEP.

WDEG can detect or block suspicious API calls caused by ROP-like behavior.

Important protection:

```text
EnableRopCallerCheck
```

---

## Enable ROP Caller Check

Command:

```powershell
Set-ProcessMitigation -Name 'C:\Program Files (x86)\Sync Breeze Enterprise\bin\syncbrs.exe' -Enable EnableRopCallerCheck
```

The `-Enable` parameter to enforce this protection is:

```text
EnableRopCallerCheck
```

Check configuration:

```powershell
Get-ProcessMitigation -Name 'C:\Program Files (x86)\Sync Breeze Enterprise\bin\syncbrs.exe'
```

Restart service:

```powershell
Restart-Service -Name "Sync Breeze Enterprise"
```

---

## Security-Mitigations Logs

Exploit Guard events are not written to normal Security logs.

They are stored in:

```text
Microsoft-Windows-Security-Mitigations/UserMode
```

Query with PowerShell:

```powershell
Get-WinEvent -FilterHashtable @{
    LogName='Microsoft-Windows-Security-Mitigations/UserMode'
    StartTime='5/25/2021 13:42:28'
    EndTime='5/25/2021 13:42:30'
} | Format-List Id, TimeCreated, LevelDisplayName, Message
```

---

## WDEG Block Event

When protection blocks exploitation, the event may show:

```text
LevelDisplayName : Warning
```

Example blocked API:

```text
LoadLibraryA
```

Example message concept:

```text
Process syncbrs.exe was blocked from calling the API LoadLibraryA due to ROP exploit indications.
```

---

## Remove Exploit Protection Configuration

Exploit protection settings are stored in the registry.

Remove SyncBreeze mitigation config:

```powershell
Remove-Item -Path 'HKLM:\Software\Microsoft\Windows NT\CurrentVersion\Image File Execution Options\syncbrs.exe'
```

Cmdlet used:

```text
Remove-Item
```

---

Audit-only mode logs what would have been blocked but does not stop the exploit.

Enable audit mode:

```powershell

```

The `-Enable` parameter for audit mode is:

```text
AuditEnableRopCallerCheck
```

Restart service:

```powershell
Restart-Service -Name "Sync Breeze Enterprise"
```

---

## WDEG Audit Events

Audit-only events may show:

```text

```

Example API calls:

```text

```

## Blocking vs Auditing

| Mode | Result |
|---|---|
| Enforced protection | Blocks exploit and generates Warning event |
| Audit-only protection | Allows exploit but logs Information event |

# Key Detection Ideas

## Credential Abuse

Look for:

- Many failed logons
- Event ID `4625`
- Repeated same username
- Same source IP
- Logon Type `3`
- Successful `4624` shortly after failures
- Administrator login outside business hours

---

## Suspicious RDP

Look for:

- Event ID `4624`
- Logon Type `10`
- Off-hours activity
- Unexpected source IP
- Administrator account usage

---

## Web Application Exploitation

Look for:

- Suspicious IIS paths
- LFI traversal patterns
- Access to configuration files
- Python or scripted user-agents
- POST requests to vulnerable plugin paths
- Unusual request duration

---

## Command Injection

Look for Sysmon Event ID `1` where:

- `php-cgi.exe` spawns `cmd.exe`
- IIS user `IUSR` runs system commands
- CommandLine includes `systeminfo`, `whoami`, `certutil`, `powershell`, or `nc.exe`

---

## File Upload

Look for Sysmon Event ID `11` where:

- `certutil.exe` creates files
- `powershell.exe` creates files
- files are written to `C:\Windows\Temp`
- suspicious executables like `nc.exe` appear

---

## Reverse Shell

Look for Sysmon Event ID `3` where:

- unusual process connects outbound
- destination is attacker IP
- destination port is unusual, such as `4444`
- `nc.exe` connects outbound
- server connects back to client

---

## Binary Exploitation

Look for:

- Service process spawning `cmd.exe`
- Unexpected SYSTEM shell
- Network connection from service process to attacker IP
- Reverse shell port `4444`
- ProcessCreate events for `whoami`, `hostname`, or `systeminfo`

---

## Exploit Guard

Look for:

- Security-Mitigations logs
- Warning events for blocked exploitation
- Information events for audit-only mitigation
- API names such as `LoadLibraryA` and `CreateProcessA`

---



# Key Takeaways

- Server-side attacks directly target vulnerable services or applications.
- Credential abuse can be detected with Windows Security events.
- Brute force attacks create many `4625` events and may be followed by a `4624`.
- Logon Type `10` indicates RDP.
- Logon Type `3` indicates network authentication.
- IIS logs are critical for web attack investigation.
- LFI can expose plaintext credentials in application config files.
- Command injection can be traced through IIS logs and Sysmon ProcessCreate events.
- File upload attacks often generate Sysmon FileCreate and NetworkConnect events.
- Certutil and PowerShell are common Living-off-the-Land download tools.
- Reverse shells can be detected through process chains and outbound network connections.
- Binary exploitation may cause a vulnerable service to spawn a SYSTEM shell.
- Windows Defender Exploit Guard can block or audit exploit behavior.
- Strong defense requires correlating Windows Security logs, IIS logs, Sysmon logs, and mitigation logs.
