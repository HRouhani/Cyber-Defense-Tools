```text
authentication logs

```

Privilege escalation means:

```text
A user or process gains more privileges than intended.
```

On Linux, the highest-privileged account is:

```text
root
```

The root user has UID:

```text
0
```

# 10.1 Attacking the Users

## What This Section Covers

This section focuses on privilege escalation opportunities related to users.

Examples:

```text
A user tries unauthorized sudo commands
A user abuses another user's SSH private key

A user backdoors .bashrc or .profile
```

Main defensive idea:

# 10.1.1 Becoming a User

## Linux User Identity

Linux identifies users and groups using numeric values.

```text
UID = User ID
GID = Group ID
```

Example `/etc/passwd` entry:

```text
offsec:x:1000:1000:offsec,,,:/home/offsec:/bin/bash
```

Field breakdown:

| Field | Example | Meaning |
|---|---|---|
| Login name | `offsec` | Username |
| Password field | `x` | Password hash is stored in `/etc/shadow` |
| UID | `1000` | User ID |
| GID | `1000` | Primary group ID |
| Comment | `offsec,,,` | User description |
| Home directory | `/home/offsec` | User home folder |
| Login shell | `/bin/bash` | Shell started after login |

---

## `/etc/passwd`

The `/etc/passwd` file stores basic user account information.

Example:

```bash
cat /etc/passwd | grep offsec
```

Important:

```text
/etc/passwd is usually readable by all users.
```

But password hashes should not be stored directly there anymore.

Instead, modern Linux stores password hashes in:

```text
/etc/shadow
```

---

## `/etc/shadow`

The `/etc/shadow` file stores password hashes.

Important:

```text
/etc/shadow should only be readable by root.
```

Why this matters:

```text
If attackers can read /etc/shadow, they can try to crack password hashes offline.
```

---

## What Is `sudo`?

`sudo` allows a user to run commands as another user, usually root.

Check sudo privileges:

```bash
sudo -l
```

Example for a fully privileged user:

```text
User offsec may run the following commands on linux01:
    (ALL : ALL) ALL
```

Meaning:

```text
offsec can run any command as any user.
```

Example restricted sudo permission:

```text
User bob may run the following commands on linux01:
   (root) NOPASSWD: /usr/bin/ls
```

Meaning:

```text
bob can run /usr/bin/ls as root without entering a password.
```

Important:

```text
Even limited sudo permissions can sometimes become dangerous.
```

---

## What Is `su`?

`su` means:

```text
switch user
```

Example:

```bash
su bob
```

Meaning:

```text
Switch to the bob user after entering bob's password.
```

Difference:

```text
sudo = run a command as another user
su   = switch to another user account/session
```

# Sudo Logs

## Where Sudo Logs Are Stored

On Ubuntu and Debian:

```text
/var/log/auth.log
```

On CentOS, RHEL, and Fedora:

```text
/var/log/secure
```

Search for sudo events:

```bash
sudo cat /var/log/auth.log | grep "sudo:"
```

---

## Successful Sudo Log Example

Example:

```text
Aug 16 15:28:19 linux01 sudo: offsec : TTY=pts/0 ; PWD=/home/offsec ; USER=root ; COMMAND=list
```

Important fields:

| Field | Meaning |
|---|---|
| `offsec` | User who ran sudo |
| `TTY=pts/0` | Terminal session |
| `PWD=/home/offsec` | Working directory |
| `USER=root` | Target user |
| `COMMAND=list` | Command executed |

Here:

```text
COMMAND=list
```

is the expanded log form of:

```bash
sudo -l
```

---

## Blocked Sudo Attempt

Example command:

```bash
sudo cat /etc/shadow
```

If the user is not allowed:

```text
Sorry, user bob is not allowed to execute '/usr/bin/cat /etc/shadow' as root on linux01.
```

Log example:

```text
Aug 16 15:39:08 linux01 sudo: bob : command not allowed ; TTY=pts/0 ; PWD=/home/offsec ; USER=root ; COMMAND=/usr/bin/cat /etc/shadow
```

Important indicator:

```text
command not allowed
```

This can indicate:

```text
Privilege escalation attempt
User testing sudo boundaries
Suspicious access to restricted files
```

# Script: `sudo_privesc.py`

## Purpose

The script `sudo_privesc.py` detects unauthorized sudo attempts by searching authentication logs for:

```text
sudo: ... command not allowed
```

It checks both common authentication log locations:

```text
/var/log/secure
/var/log/auth.log
```

This makes it compatible with both:

```text
CentOS/RHEL/Fedora style systems
Ubuntu/Debian style systems
```

---

## Full Script

```python
#!/usr/bin/env python

import re
import os.path

centos_auth_log_file_path = "/var/log/secure"
ubuntu_auth_log_file_path = "/var/log/auth.log"

auth_log_files = [centos_auth_log_file_path,ubuntu_auth_log_file_path]

regex_sudo_privesc_attempt = 'sudo:.*command not allowed'

for log_file in auth_log_files:
        if os.path.isfile(log_file) :
                with open(log_file, "r") as file:
                    for line in file:
                        for match in re.finditer(regex_sudo_privesc_attempt, line, re.S):
                            print("[*] Found 'sudo' privilege escalation attempt:\n\t"   + line,end = '')
```

---

### Log file paths

```python
centos_auth_log_file_path = "/var/log/secure"
ubuntu_auth_log_file_path = "/var/log/auth.log"
```

Meaning:

```text
The script checks both Linux log locations.
```

---

### Log file list

```python

```

Meaning:

```text
Both files are placed into a list and checked one by one.
```

---

### Detection regex

```python
regex_sudo_privesc_attempt = 'sudo:.*command not allowed'
```

Meaning:

```text
Find any line that contains sudo:
Then any characters
Then command not allowed
```

This matches blocked sudo attempts.

---

### File existence check

```python

```

Meaning:

```text
Only read the log file if it exists on the system.
```

This avoids errors when one of the log paths does not exist.

---

### Match and print suspicious sudo event

```python

    print("[*] Found 'sudo' privilege escalation attempt:\n\t" + line,end = '')
```

Meaning:

```text
If a blocked sudo command is found, print the full log line.
```

# Stolen SSH Private Key Abuse

## Scenario

A user has access to another user's SSH private key.

Example:

```text
alice has bob's private key.
alice uses it to log in as bob.
```

Example command:

```bash
ssh bob@localhost -i /home/alice/stolen_id_rsa
```

Meaning:

```text
Connect as bob using the private key stored in alice's home directory.
```

---

## SSH Log Evidence

Example logs:

```text
Accepted password for alice from 192.168.48.3 port 34172 ssh2
Accepted publickey for bob from 127.0.0.1 port 35814 ssh2
```

Interpretation:

```text
alice logged in with a password.
Shortly after, bob logged in with a public key from localhost.
```

Suspicious pattern:

```text
User A logs in
→ User B logs in from localhost using an SSH key
→ close timestamp
```

But this alone is not complete proof.

Why?

```text
The logs show that bob logged in by key.
They do not directly prove alice typed the command.
```

For stronger proof, we need command auditing.

PAM TTY auditing records terminal input.

It can capture:

```text
commands typed by users
keystrokes
interactive shell activity
```

Configuration file:

```text
/etc/pam.d/sshd
```

Enable TTY auditing for SSH:

```text
session required pam_tty_audit.so enable=*
```

Meaning:

```text
Enable TTY auditing for all SSH users.
```

Important:

```text
TTY auditing can be sensitive because it records what users type.
Use it carefully in real environments.
```

Command:

```bash
sudo aureport --tty
```

Purpose:

```text
Show TTY audit activity in a readable report.
```

# UID vs AUID

## UID

`uid` means:

```text
Current user ID of the process.
```

Example:

```text
uid=0
```

Meaning:

```text
The process is currently running as root.
```

---

## AUID

`auid` means:

```text
Audit user ID.
```

The `auid` is assigned when a user logs in.

Important:

```text
auid stays the same for the whole login session.
```

Even if the user later runs:

```text
sudo
su
root shell
```

the `auid` still points back to the original logged-in user.

Example:

```text
uid=0 auid=1002
```

Meaning:

```text
The process is running as root now.
But the original logged-in user was UID 1002.
```

This is very important for attribution.


```text
uid  = current user running the process
euid = effective user used for permission checks
auid = original login user
```



---

## Why AUID Matters

Without `auid`, a malicious user could do:

```text
alice → sudo/su/root → malicious command
```

and the logs might only show:

```text
uid=0
```

But with `auid`, we can still see:

```text
auid=1002
```

Meaning:

```text
alice was the original user behind the action.
```

# Audit Logs

## Audit Log Location

Audit logs are stored in:

```text
/var/log/audit/audit.log
```

Search for TTY events:

```bash
sudo cat /var/log/audit/audit.log | grep "type=TTY"
```

Search TTY events for a specific UID:

```bash
sudo cat /var/log/audit/audit.log | grep "type=TTY" | grep " uid=1002"
```

---

## TTY Event Example

```text
type=TTY msg=audit(...): tty pid=34376 uid=1002 auid=1002 ses=375 major=136 minor=0 comm="bash" data=73736820626F62406C6F63616C686F7374202D69202F686F6D652F616C6963652F73746F6C656E5F69645F7273610D657869740D
```

Important field:

```text
data=<hex encoded command>
```

The command is stored in hexadecimal format.

---

## Decode Audit Hex Data Manually

Example:

```bash
echo "657869740D" | sed 's/0D/20/g' | xxd -r -p
```

Meaning:

```text
0D is carriage return.
The command replaces 0D with 20, which is a space.
xxd converts hex back to readable text.
```

Decoded result:

```text
exit
```

---

## Decode Stolen Key Command

Encoded value:

```text
73736820626F62406C6F63616C686F7374202D69202F686F6D652F616C6963652F73746F6C656E5F69645F7273610D657869740D
```

Decoded value:

```text
ssh bob@localhost -i /home/alice/stolen_id_rsa exit
```

Meaning:

```text
The audit log proves that the user typed the SSH command using bob's stolen key.
```

# Script: `audit_decoder.py`

## Purpose

The script `audit_decoder.py` searches audit logs for TTY events belonging to a specific UID and decodes the hex-encoded `data=` field.

It is used to answer questions like:

```text
What commands did UID 1002 type?
```

Example:

```bash
python3 audit_decoder.py 1002
```

---

## Full Script

```python
#!/usr/bin/env python

import re
import sys
import os.path
import binascii


def usage():
    print("Usage: "   + sys.argv[0] + " [uid]")
    print("Example: " + sys.argv[0] + " 1001")
    sys.exit()


def log_file_parser(uid):
        audit_log_file  = "/var/log/audit/audit.log"
        regex_audit     = "^type=TTY.*"+" uid=" + uid +".*data=.*"

        if os.path.isfile(audit_log_file) :
                with open(audit_log_file, "r") as file:
                    for line in file:
                        for match in re.finditer(regex_audit, line, re.S):
                            print("[*] Found the following user's audit data belonging to UID:%s" % uid)
                            encoded_commands = ((line.split("data=")[1])).strip()
                            decoded_commands = (binascii.a2b_hex(encoded_commands))
                            print(decoded_commands)
                            print("---")

if __name__ == '__main__':
    if len(sys.argv) > 1:

        uid    = sys.argv[1]
        log_file_parser(uid)

    else:
        usage()
```

---

### Usage function



Meaning:

```text

```

# 10.1.2 Backdooring a User

## What Is User Backdooring?

User backdooring means modifying a user's startup files so attacker-controlled commands run automatically when that user logs in or opens a shell.

Common target files:

```text
.bashrc
.profile
```

Common location:

```text
/home/<user>/
```

Examples:

```text
/home/bob/.bashrc
/home/bob/.profile
```

---

## What Is `.bashrc`?

`.bashrc` is a Bash configuration file.

It is often executed when a user starts an interactive Bash shell.

It can contain:

```text
aliases
functions
environment setup
commands to run automatically
```

If an attacker modifies `.bashrc`, their command may run in the victim user's context.

---

## What Is `.profile`?

`.profile` is a user login configuration file.

It is commonly executed during login shell startup.

It can contain:

```text
environment variables
startup commands
PATH changes
```

If an attacker modifies `.profile`, their command may run when the user logs in.

---

## Dangerous Permission Example

Check bob's `.bashrc`:

```bash
ls -asl /home/bob/.bashrc
```

Example:

```text
-rw-r--rw- 1 bob bob 3771 Aug 27 03:24 /home/bob/.bashrc
```

Problem:

```text
world-writable
```

Meaning:

```text
Other users can modify bob's .bashrc.
```

That is dangerous.

---

## Backdoor Example

As alice:

```bash
echo 'echo "hello from bob .bashrc"' >> /home/bob/.bashrc
```

Meaning:

```text
Append a command to bob's .bashrc.
```

When bob logs in:

```text
hello from bob .bashrc
```

This proves:

```text
The injected command runs when bob starts a shell.
```

---

## Why This Is Dangerous

Instead of a harmless echo command, an attacker could add:

```text
reverse shell
credential theft command
SSH key exfiltration
PATH hijack
alias hijack
environment variable manipulation
```

Attack chain:

```text
alice modifies bob's .bashrc
→ bob logs in
→ malicious command runs as bob
→ attacker gains bob's context
```

---

# Detecting Backdoored User Files

## Use `auditctl`

Watch bob's `.bashrc`:

```bash
sudo auditctl -w /home/bob/.bashrc -p wa -k privesc
```

Watch bob's `.profile`:

```bash
sudo auditctl -w /home/bob/.profile -p wa -k privesc
```

Meaning:

```text
-w = watch this path
-p wa = watch writes and attribute changes
-k privesc = tag matching events with the key privesc
```

| `r` | read |

# Script: `audit_key_search.py`

## Purpose

The script `audit_key_search.py` searches `/var/log/audit/audit.log` for events containing a specific audit key.

Example:

```bash
python3 audit_key_search.py privesc
```

It is useful after creating audit rules like:

```bash
sudo auditctl -w /home/bob/.bashrc -p wa -k privesc
```

or:

```bash
sudo auditctl -w /etc/shadow -p war -k etc_shadow
```

---

## Full Script

```python
#!/usr/bin/env python

import re
import sys
import os.path
import binascii


def usage():
    print("Usage: "   + sys.argv[0] + " [key]")
    print("Example: " + sys.argv[0] + " privesc")
    sys.exit()


def log_file_parser(keyarg):
    audit_log_file  = "/var/log/audit/audit.log"
    regex_audit     = ".*key=\"" + keyarg +"\".*"

    if os.path.isfile(audit_log_file) :
        with open(audit_log_file, "r") as file:
            for line in file:
                for match in re.finditer(regex_audit, line, re.S):
                    print("[*] Found audit logs based on the following key: %s" % keyarg)
                    auid        = (line.split("auid=")[1]).split()[0]
                    event_type  = (line.split("type=")[1]).split()[0]
                    print("event-type: %s" % event_type)
                    print("auid: %s" % auid)
                    try:
                        exe   = (((line.split("exe=")[1])).strip()).split()[0]  
                        print("program executed: %s" % exe)               
                    except(IndexError):
                        pass
                    print("---")

if __name__ == '__main__':
    if len(sys.argv) > 1:

        keyarg    = sys.argv[1]
        log_file_parser(keyarg)

    else:
        usage()
```

# 10.2.1 Abusing System Programs

## Real UID vs Effective UID

When a process runs, Linux tracks several user IDs.

Important values:

```text
Real UID
Effective UID
Saved UID
Filesystem UID
```

Check a process:

```bash
grep Uid /proc/<PID>/status
```

Example normal process:

```text
Uid:    1000    1000    1000    1000
```

Meaning:

```text
The process was started by UID 1000.
It is also authorized as UID 1000.
```

---

## Effective UID

The most important value for runtime authorization is:

```text
Effective UID
```

Meaning:

```text
The effective UID is checked when the process tries to access protected resources.
```

Example:

```text
euid=0
```

Means:

```text
The process has root-level effective privileges.
```

---

## The `passwd` Example

The `passwd` command lets users change their password.

Command:

```bash
passwd
```

But password hashes are stored in:

```text
/etc/shadow
```

Only root can write there.

So how can a normal user change their password?

Answer:

```text
/usr/bin/passwd uses SUID.
```

---

## Inspect `passwd`

Command:

```bash
ls -asl /usr/bin/passwd
```

Example:

```text
-rwsr-xr-x 1 root root 68208 Apr 16 2020 /usr/bin/passwd
```

Important part:

```text
rws
```

The `s` means:

```text
SUID bit is set.
```

---

# What Is SUID?

SUID means:

```text
Set User ID
```

If a binary has SUID set:

```text
When executed, the process gets the effective UID of the file owner.
```

Example:

```text
/usr/bin/passwd is owned by root.
SUID is set.
Normal user runs passwd.
passwd runs with effective UID 0.
```

This is legitimate when used safely.

---

## Why SUID Can Be Dangerous

If a SUID-root binary has a vulnerability or unsafe behavior, a normal user may become root.

Dangerous SUID behavior examples:

```text
spawning a shell
calling system() unsafely
using relative paths
loading unsafe libraries
allowing command injection
writing to protected files
```

---

## Vulnerable SUID Example

Example:

```bash
./vuln_suid
```

Example output:

```text
RUID: 1000
EUID: 0

setuid success

RUID: 0
EUID: 0

whoami: root
```

Meaning:

```text
The user started as normal UID 1000.
The binary had effective UID 0.
The program called setuid successfully.
The user obtained a root shell.
```

```text
Tag the event with root_cmds.
```

```text
A process starts another program.
```

Examples:

```text

cron executing a script
```

If we monitor `execve` with `euid=0`, we can see root-level program execution.

# 10.2.3 Weak Permissions

## Linux File Permissions

Example:

| `rwx` | `111` | `7` |

# Script: `cron_scripts/clear_history.py`

## Purpose

The script `clear_history.py` is used by the root cron job.

Its normal purpose is:

```text
Clear the offsec user's Bash history.
```

But because the script is writable by other users in the lab, it becomes a privilege escalation path.

---

## Full Script

```python
#!/usr/bin/env python3
import os
import sys
try:
   os.system('cat /dev/null > /home/offsec/.bash_history ')
except:
    sys.exit()
```

---

## Important Parts Explained

### Interpreter

```python
#!/usr/bin/env python3
```

Meaning:

```text
Run this script using Python 3.
```

---

### Import modules

```python
import os
import sys
```

Meaning:

```text
os is used to run system commands.
sys is used to exit the script if something fails.
```

---

### Command execution

```python
os.system('cat /dev/null > /home/offsec/.bash_history ')
```

Meaning:

```text
Run a shell command from Python.
```

The shell command:

```bash
cat /dev/null > /home/offsec/.bash_history
```

means:

```text
Take empty content from /dev/null
Write it into /home/offsec/.bash_history
Overwrite the file
```

Result:

```text
The Bash history file becomes empty.
```

---

## Why This Script Is Dangerous

The script itself is not dangerous only because it clears history.

The dangerous part is the combination:

```text
Script is writable by normal users
+
Script is executed by root cron
=
Normal user can inject commands that run as root
```


Meaning:

```text
Anyone can modify the script.
Root cron runs it every minute.
Normal user can insert malicious code.
```

Attack path:

```text
alice modifies script
→ cron runs script as root
→ alice's command runs as root
```

# Monitoring `/etc/shadow`

## Why Monitor `/etc/shadow`?

`/etc/shadow` contains password hashes.

If attackers read it, they may:

```text
Dump password hashes
Crack them offline
Gain persistent access through real passwords
```

This is credential dumping.

---

## Audit Rule for `/etc/shadow`

Command:

```bash
sudo auditctl -w /etc/shadow -p war -k etc_shadow
```

Meaning:

```text
-w /etc/shadow = watch /etc/shadow
-p war         = watch write, attribute change, and read
-k etc_shadow  = tag with etc_shadow
```

Why include `r`?

```text
Reading /etc/shadow is enough to dump password hashes.
```

---

## Detect `/etc/shadow` Dumping

Example:

```bash
sudo ausearch -k etc_shadow -c cat -i
```

Meaning:

```text
Search audit logs for events tagged etc_shadow.
Filter for command cat.
Interpret IDs.
```

Example important fields:

```text
proctitle=cat /etc/shadow
cwd=/home/alice
syscall=openat
auid=alice
uid=root
euid=root
comm=cat
exe=/usr/bin/cat
key=etc_shadow
```

Interpretation:

```text
The file /etc/shadow was opened.
The command was cat.
The process ran as root.
The original logged-in user was alice.
```

Important:

```text
uid=root/euid=root shows root privileges.
auid=alice shows who originally caused the action.
```

# The `openat` System Call

## What Is `openat`?

`openat` is a Linux system call used to open files.

In this chapter, it appears when:

```text
A user edits a cron script

A program opens a monitored file
```

Important:

```text
File reads and writes often involve open or openat.
```

So if you want to detect file access, only monitoring `execve` is not enough.

# Main Takeaways

```text
Linux privilege escalation often comes from misconfiguration, not only exploits.

UID identifies users.
GID identifies groups.
root has UID 0.

The effective UID is what matters for runtime permission checks.

sudo misuse can be detected in /var/log/auth.log or /var/log/secure.

The phrase command not allowed is useful for detecting blocked sudo privilege escalation attempts.

sudo_privesc.py detects blocked sudo attempts.

Stolen SSH key abuse may be visible in SSH logs, but audit TTY logs provide stronger proof.

TTY audit data may be hex-encoded in /var/log/audit/audit.log.

audit_decoder.py decodes TTY audit data for a selected UID.

AUID is extremely important because it tracks the original logged-in user, even after sudo, su, or root escalation.

.bashrc and .profile can become privilege escalation paths if they are writable by other users.

auditctl can monitor sensitive files and directories for writes, reads, executes, and attribute changes.

audit_key_search.py searches audit logs based on audit keys like privesc, cron_scripts, or etc_shadow.

SUID binaries run with the effective UID of the file owner.

A vulnerable SUID-root binary can allow a normal user to become root.

execve is the system call used to execute programs.

open/openat are important system calls for file access.

World-writable scripts executed by root cron jobs are dangerous.

clear_history.py is a safe-looking script, but it becomes dangerous if it is writable and executed by root cron.

Monitoring /etc/shadow for read access can detect credential dumping.

Good Linux privilege escalation detection requires correlation between:
authentication logs
sudo logs
audit logs
file permissions
cron jobs
SUID binaries
process execution
AUID/EUID values
custom detection scripts
```
