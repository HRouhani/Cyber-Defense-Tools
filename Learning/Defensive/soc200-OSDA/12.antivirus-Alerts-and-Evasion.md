# 12.1. Antivirus Basics

## What Is Antivirus Software?

Antivirus software protects an endpoint from malicious files and suspicious behavior.

Most antivirus tools work in two main ways:

```text
1. On-demand scanning
2. Real-time protection
```

---

## On-Demand Scanning

On-demand scanning means:

```text
A user or administrator manually starts a scan.
```

Examples:

```text
Scan one file
Scan one folder
Scan the whole system
```

In Microsoft Defender, on-demand scans can be started with:

```powershell
Start-MpScan
```

or with:

```text
MpCmdRun.exe
```

Simple meaning:

```text
On-demand scanning = manual scan.
```

---

## Real-Time Protection

Real-time protection means:

```text
Defender continuously monitors file and process activity.
```

It can detect malware when:

```text
a file is downloaded
a file is created
a file is executed
a process behaves suspiciously
a script runs suspicious code
```

Simple meaning:

```text
Real-time protection = Defender watches the system live.
```

## Custom Helper Function: `Get-WDLogEvent`

The chapter uses a helper function called:

```powershell
Get-WDLogEvent
```

which is as follow:

```text
Get-WDLog.psm1
```

```powershell
function Get-WDLogEvent {
    param (
        $eventid,
        $start,
        $end
    )

    $filters = @{
        LogName = "Microsoft-Windows-Windows Defender/Operational"
    }

    if ($eventid -ne $null) {
        $filters.ID = $eventid
    }

    if ($start -ne $null) {
        $filters.StartTime = $start
    }

    if ($end -ne $null) {
        $filters.EndTime = $end
    }

    Get-WinEvent -FilterHashtable $filters
}
``` 

Conceptually, it works like previous helper functions:

```text
Get events from a specific log
Filter by Event ID
Filter by StartTime
Filter by EndTime
```

The log name used by the function is:

```text
Microsoft-Windows-Windows Defender/Operational
```

Example usage:

```powershell
Get-WDLogEvent 1116 "12/2/2021 10:59:00" "12/2/2021 11:00:00" | Format-List
```

# 12.1.2. Signature-Based Detection

## What Is Signature-Based Detection?

Signature-based detection means Defender searches for known malware patterns.

A signature can be understood as:

```text
A unique byte pattern or known identifier inside malware.
```

Simple explanation:

```text
Defender knows what known malware looks like.
If a file matches that known pattern, Defender detects it.
```

---

## Why Signature Detection Works

Signature detection is useful for:

```text
known malware
known payloads
known tools
known file patterns
```

Example:

```text
A known Meterpreter executable is scanned.
Defender recognizes the signature.
Defender generates a detection.
```

---

## Weakness of Signature Detection

Attackers may try to avoid signature detection by changing the file.

Examples:

```text
changing strings
changing byte sequences
packing the file
obfuscating the payload
recompiling the payload
encrypting parts of the payload
```

Simple idea:

```text
If the known signature changes, signature-based detection may fail.
```

But this does not mean the attack is invisible.

Other detection methods may still catch it.

---

## Detection Type: Concrete

When Defender shows:

```text
Detection Type: Concrete
```

it usually means:

```text
The file matched a known signature or known detection rule.
```

Simple meaning:

```text
Concrete = Defender is confident because it matched known malware.
```

# 12.1.3. Real-Time Heuristic and Behavioral-Based Detection

## What Is Real-Time Protection?

Real-time protection continuously watches the system.

It can detect malicious activity when:

```text
a file is downloaded
a file is written to disk
a process starts
a process performs suspicious actions
a script executes suspicious content
```

Simple meaning:

```text
Real-time protection watches activity as it happens.
```

---

## Enabling Real-Time Protection

In Group Policy:

```text
Local Computer Policy
→ Computer Configuration
→ Administrative Templates
→ Windows Components
→ Microsoft Defender Antivirus
→ Real-time Protection
```

Important setting:

```text
Turn off real-time protection
```

To enable real-time protection, this setting must be:

```text
Disabled
```

This sounds strange, but it means:

```text
Disable the policy that turns protection off.
```

So:

```text
Turn off real-time protection = Disabled
```

means:

```text
Real-time protection is enabled.
```

---

# Heuristic Detection

## What Is Heuristic Detection?

Heuristic detection uses rules and logic to decide whether something looks malicious.

It does not only rely on known signatures.

It may inspect:

```text
program structure
API usage
suspicious instruction patterns
unusual script behavior
malware-like logic
```

Simple meaning:

```text
Heuristic detection asks:
"Does this look like malware?"
```

---

## Example Heuristic Indicators

Suspicious indicators can include:

```text
hidden PowerShell execution
encoded commands
memory allocation for shellcode
downloading and executing remote code
process injection behavior
credential dumping behavior
security tool tampering
```

---

# Behavioral Detection

## What Is Behavioral Detection?

Behavioral detection watches what a program actually does.

Simple meaning:

```text
Behavioral detection asks:
"What is this process doing?"
```

Examples of suspicious behavior:

```text
dumping password hashes
injecting into another process
opening a reverse shell
downloading a second-stage payload
modifying Defender settings
creating persistence
credential access activity
```

---

## Difference Between Signature, Heuristic, and Behavioral Detection

| Detection Type | Main Idea | Example |
|---|---|---|
| Signature | Matches known malware pattern | Known Meterpreter file |
| Heuristic | Looks suspicious before or during execution | Encoded PowerShell logic |
| Behavioral | Acts suspicious during runtime | Hashdump or reverse shell behavior |


## Heuristic Detection Explained

Heuristic detection is a detection method where antivirus does **not only look for a known malware signature**.

Instead, it checks whether a file, script, or process **looks suspicious** based on rules, patterns, and risk indicators.

Heuristic detection = "This looks like malware, even if I do not know this exact malware yet."



Simple summary:

```text
Signature = known bad pattern
Heuristic = suspicious structure
Behavioral = suspicious action
```

# 12.2. Antimalware Scan Interface

# AMSI

## What Is AMSI?

`AMSI` means:

```text
Antimalware Scan Interface
```

AMSI is a Windows security interface that allows antivirus products to inspect script content before it runs.

It is mainly used for script-based and fileless attacks.

AMSI is not usually written inside the script itself. Instead, the application or scripting engine that runs the script must support AMSI.

Simple explanation:

```text
AMSI lets Defender scan scripts and in-memory code before execution.
```

AMSI can be used with:

```text
PowerShell
VBScript
JScript / JavaScript through Windows Script Host
Office macro-related content
some in-memory or dynamic script content
applications that explicitly integrate AMSI
```

---

## Why AMSI Exists

Older antivirus mainly scanned files on disk.

But attackers started using:

```text
fileless malware
PowerShell in memory
remote script loading
encoded commands
Office macros
VBScript
JavaScript
```

These attacks may not write a traditional malware file to disk.

AMSI helps by scanning:

```text
script content
PowerShell commands
macro content
script buffers
in-memory code strings
```

---

## What Uses AMSI?

AMSI can inspect content from:

```text
PowerShell
PowerShell ISE
VBScript
JavaScript
WScript
cscript
Microsoft Office macros
some browser scripting engines
```

---

# How AMSI Works

## Basic AMSI Flow

```text
PowerShell receives script or command
→ AMSI.dll is loaded into the PowerShell process
→ PowerShell sends script content to AMSI
→ AMSI forwards content to Defender
→ Defender scans the content
→ Defender returns result
→ AMSI allows or blocks execution
```

Simple meaning:

```text
PowerShell asks Defender:
"Is this script safe?"
```

---

## Important AMSI DLL

AMSI functionality is implemented through:

```text
amsi.dll
```

This DLL is loaded into supported scripting processes.

Example:

```text
powershell.exe loads amsi.dll
```

---

## Important AMSI APIs

AMSI uses functions such as:

```text
AmsiInitialize
AmsiOpenSession
AmsiScanString
AmsiScanBuffer
AmsiCloseSession
```

Important one:

```text
AmsiScanBuffer
```

Because it is responsible for scanning content buffers.

Simple meaning:

```text
AmsiScanBuffer checks script content before it runs.
```

---

# AMSI Detection Example

## Triggering AMSI

The chapter uses a simple suspicious keyword:

```text
amsiutils
```

When typed in PowerShell, Defender blocks it.

Example result:

```text
This script contains malicious content and has been blocked by your antivirus software.
```

Important:

```text
The string does not need to be a full malware payload.
AMSI can detect suspicious content patterns.
```

---

## AMSI Defender Event

Expected Defender event:

```text

Name: Trojan:Win32/AmsiTamper.A!ams

```

Important field:

```text

```

This confirms:

```text

```

---

# AMSI vs Normal File Detection

| Detection Area | What Is Scanned | Example |
|---|---|---|
| Signature scan | File on disk | EXE detected as Trojan |
| Real-time protection | File/process activity | Downloaded payload |
| AMSI | Script/in-memory content | PowerShell code blocked |

Simple summary:

```text
AMSI protects against script-based and fileless attacks.
```

# 12.2.2. AMSI Bypass Concepts

## What Is an AMSI Bypass?

An AMSI bypass attempts to stop AMSI from properly scanning script content.

The goal is:

```text
Run malicious PowerShell or script content without AMSI blocking it.
```

Important:

```text
AMSI bypass usually affects only the current process/session.
```

For example:

```text
If PowerShell process A is patched,
PowerShell process B may still have AMSI active.
```

---

## Important Defensive Note

The goal here is not to memorize bypass code.

The goal is to understand:

```text
what attackers try to break
what logs are created
what suspicious artifacts appear
how to detect AMSI tampering
```

---

# AMSI Bypass Method 1: Patching `AmsiScanBuffer`

## Concept

One bypass approach patches the `AmsiScanBuffer` function in memory.

Basic idea:

```text
PowerShell loads amsi.dll
→ attacker locates AmsiScanBuffer in memory
→ attacker changes bytes in memory
→ AmsiScanBuffer returns clean result
→ AMSI no longer blocks malicious content in that session
```

Simple meaning:

```text
The attacker changes AMSI behavior inside memory.
```

---

## Why This Works

Normally:

```text
AmsiScanBuffer scans script content
→ Defender says malicious
→ PowerShell blocks execution
```

After patching:

```text
AmsiScanBuffer is modified
→ returns clean result
→ PowerShell continues execution
```

---

## Important Limitation

This is usually not permanent.

It affects:

```text
current PowerShell process
current script host process
```

If a new PowerShell process starts, AMSI may be normal again.

---


```text

```

---

## Why `csc.exe` Is Suspicious Here

`csc.exe` is:

```text
C# command-line compiler
```

Path example:

```text
C:\Windows\Microsoft.NET\Framework64\v4.0.30319\csc.exe
```

If PowerShell starts `csc.exe`, it may mean:

```text
PowerShell is compiling C# code dynamically.
```

This is suspicious when seen during AMSI bypass or malware execution.

---

## Why `cvtres.exe` Is Suspicious Here

`cvtres.exe` is:

```text
Microsoft Resource File to COFF Object Conversion Utility
```

It can appear during compilation activity.

Suspicious chain:

```text
powershell.exe
→ csc.exe
→ cvtres.exe
```

This may indicate:

```text
PowerShell dynamically compiled code on the endpoint.
```

---

## Suspicious Temp Files

Examples:

```text
C:\Users\<user>\AppData\Local\Temp\<random>.dll
C:\Users\<user>\AppData\Local\Temp\<random>.cmdline
```

Defender idea:

```text
Random DLL + cmdline file in Temp + csc.exe execution = suspicious.
```

# AMSI Bypass Method 2: Corrupting AMSI Context

## Concept

Another bypass approach targets AMSI internal context.

Basic idea:

```text
PowerShell locates internal AMSI context
→ attacker overwrites or corrupts it
→ AMSI stops working correctly
→ malicious script can run
```

Simple meaning:

```text
Instead of patching the scan function,
the attacker breaks AMSI's internal state.
```

---

## Why This Matters

After AMSI context corruption, malicious PowerShell may execute without AMSI blocking it.

However:

```text
Other Defender components may still detect behavior afterward.
```

Example:

```text
AMSI does not block the PowerShell payload.
But Defender later detects suspicious network or Meterpreter behavior.
```

---

```text
Did a suspicious process connect to port 443 or 8000?
Did the DNS query match the attacker infrastructure?
```

# Key Process Chains to Watch

## Malicious Download Chain

```text
powershell.exe
→ Invoke-WebRequest / DownloadString
→ file written to disk
→ Defender Event 1116
→ Defender Event 1117 if removed
```

```text
powershell.exe
→ script block with AMSI/memory patch logic
→ temporary DLL/cmdline file
→ csc.exe
→ cvtres.exe
→ AMSI no longer blocks in that session
```

---

## Behavioral Detection Chain

```text
payload executes
→ reverse connection starts
→ post-exploitation command runs
→ Defender detects suspicious behavior
→ Event ID 1116 with Detection Type Generic
```

# What Defenders Should Remember

## Antivirus Is Not Perfect

Antivirus can miss:

```text
new malware
modified malware
obfuscated malware
fileless malware
living-off-the-land activity
```

But defenders can still detect activity through:

```text
logs
process chains
network activity
PowerShell logging
Sysmon
Defender operational events
behavioral detections
```

---

## Signature Detection Is Useful but Limited

Signature detection catches known malware.

But attackers can change:

```text
file content
hashes
strings
payload structure
encoding
packing
```

So defenders should not rely only on signatures.

---

## Behavioral Detection Is Very Valuable

Behavioral detection can catch suspicious actions even when the file is unknown.

Examples:

```text
credential dumping
reverse shells
process injection
malicious PowerShell behavior
AMSI tampering
security tool modification
```

---

## AMSI Is Important for Script-Based Attacks

AMSI helps detect:

```text
PowerShell malware
fileless attacks
malicious macros
VBScript and JavaScript abuse
in-memory payloads
```

But AMSI can be targeted.

Therefore defenders should monitor:

```text
AMSI detection events
PowerShell 4104 script blocks
PowerShell memory manipulation
csc.exe spawned by PowerShell
suspicious Temp DLLs
Defender Event ID 1116 with Detection Source AMSI
```

## Detect Defender Detections


Look for:


Look for:


# Quick Summary

```text
Antivirus protects endpoints using signatures, heuristics, and behavioral detection.

Signature detection finds known malware patterns.
Heuristic detection finds suspicious structures or logic.
Behavioral detection finds suspicious runtime actions.

Microsoft Defender logs detections in:
Microsoft-Windows-Windows Defender/Operational

Important Defender events:
1116 = detection
1117 = action/remediation
5007 = configuration change

Real-time protection can detect malware during download or execution.

AMSI protects against script-based and fileless attacks by scanning PowerShell,
macros, VBScript, JavaScript, and similar content before execution.

Attackers may try to bypass AMSI by patching memory or corrupting AMSI context.

Even when AMSI is bypassed, useful artifacts may remain:
PowerShell 4104 script blocks
Sysmon ProcessCreate
Sysmon FileCreate
Sysmon DNS events
Sysmon NetworkConnect
Defender behavioral detections

Best defense:
Do not rely only on antivirus alerts.
Correlate Defender, PowerShell, Sysmon, DNS, and network activity.
```
