# 15. Windows Lateral Movement

## Chapter Goal

This chapter explains how attackers move from one Windows system to another after gaining an initial foothold.

The main topics are:

```text
Windows authentication
NTLM authentication
Pass-the-Hash
Mimikatz
PsExec
Domain password brute force
Password spraying
Account lockouts
RDP lateral movement
Kerberos service tickets
Kerberoasting
Kerberoasting honey tokens
Detection with Windows Event Logs and PowerShell
```

Important defender idea:

```text
Lateral movement is rarely represented by one single event.

Defenders usually need to correlate:
authentication events,
process execution,
service creation,
network activity,
and account behavior.
```

# 15.1. Windows Authentication

Windows requires users and systems to prove their identities before accessing resources.

Common Windows authentication protocols include:

```text
NTLM
Kerberos
```

Attackers may abuse both protocols:

```text
NTLM     → Pass-the-Hash
Kerberos → Kerberoasting, ticket abuse
```

Main Differences:

| Feature                    | Kerberos                    | NTLM                                    |
| -------------------------- | --------------------------- | --------------------------------------- |
| Authentication model       | Ticket-based                | Challenge-response                      |
| Preferred in AD            | Yes                         | No, usually fallback                    |
| Requires domain controller | Yes, to obtain tickets      | Often contacts DC for domain validation |
| Uses SPNs                  | Yes                         | No                                      |
| Mutual authentication      | Yes                         | Generally no                            |
| Delegation support         | Yes                         | Limited                                 |
| Security                   | Stronger                    | Older and weaker                        |
| Common attack              | Kerberoasting, ticket theft | Pass-the-Hash, relay                    |
| Typical event              | 4768, 4769                  | 4624 with NTLM package, 4776            |



Kerberos is the preferred authentication protocol in Active Directory.

It uses tickets issued by the domain controller and supports mutual authentication.

NTLM is an older challenge-response protocol.

Windows often uses NTLM when Kerberos is unavailable or cannot identify the service.

Typical Kerberos conditions:
domain account,
domain-joined systems,
reachable DC,
hostname,
valid SPN.

Typical NTLM conditions:
local account,
workgroup,
IP address,
missing SPN,
old application,
Kerberos failure.

Kerberos is generally stronger.
NTLM is more exposed to Pass-the-Hash and relay attacks.

# 15.1.1. Pass-the-Hash

## What Is Pass-the-Hash?

Pass-the-Hash, or PTH, is an attack where an attacker authenticates using a valid NTLM password hash instead of the user's plaintext password.

Simple meaning:

```text
The attacker does not know the password.

The attacker has the password's NTLM hash
and uses that hash directly for authentication.
```

---

## Normal NTLM Authentication

NTLM uses a challenge-response process.

Basic flow:

```text
1. Client sends username to server.

2. Server generates a random challenge.

3. Server sends challenge to client.

4. Client uses the password-derived NTLM hash
   to calculate a response.

5. Client sends response to server.

6. Server performs the same calculation
   using the stored NTLM hash.

7. If both responses match, authentication succeeds.
```

The plaintext password is not normally sent across the network.

---

## NTLM Challenge

The server generates a random value known as:

```text
challenge
nonce
```

In NTLM, this is normally an eight-byte server challenge, while other parts of the exchange may contain additional random values.

The client proves that it possesses the correct password-derived secret by producing the expected response.

---

## Why Pass-the-Hash Works

The NTLM authentication calculation depends on the NTLM hash.

Therefore:

```text
If the attacker steals the NTLM hash,
the attacker may perform the required calculation
without knowing the original password.
```

Concept:

```text
Plaintext password
→ NTLM hash
→ NTLM authentication response
```

PTH skips the first step:

```text
Stolen NTLM hash
→ NTLM authentication response
```

# Mimikatz

## What Is Mimikatz?

Mimikatz is a Windows security research and credential-access tool.

It can extract authentication material from Windows memory, including:

```text
NTLM hashes
Kerberos tickets
logon session information
cached credentials
DPAPI-related secrets
```

Attackers frequently target:

```text
LSASS
```

---

## What Is LSASS?

LSASS means:

```text
Local Security Authority Subsystem Service
```

Process:

```text
lsass.exe
```

It is responsible for important Windows security functions such as:

```text
authentication
security policies
access token creation
credential handling
logon sessions
```

Because LSASS handles credentials, it is a high-value target.

```text
User logs in
→ Windows authentication is handled by LSASS
→ LSASS stores or uses authentication material in memory
→ Mimikatz reads LSASS memory
→ Mimikatz may extract hashes, tickets, and session information
```

LSASS contains or processes the credential material.
Mimikatz is used to retrieve it.

---

## `privilege::debug`

Example Mimikatz command:

```text
privilege::debug
```

This attempts to enable:

```text
SeDebugPrivilege
```

Expected output is:  Privilege '20' OK

If it fails, you may not be running Mimikatz as Administrator, or the system may block access to LSASS.

SeDebugPrivilege allows a process with sufficient rights to inspect or manipulate other processes.

It is needed when accessing protected process memory such as LSASS.

---

## Dumping Credentials

Example:

```text
sekurlsa::logonPasswords full
```

Purpose:

```text
Read available logon session information from LSASS memory.
```

Possible output:

```text
Username
Domain
SID
NTLM hash
SHA1 value
logon time
logon server
authentication session
```

Example NTLM hash:

```text
2892d26cdf84d7a70e2eb3b9f05c425e
```

---

## Starting a Process with Pass-the-Hash

Example:

```text
sekurlsa::pth /domain:corp.com /user:offsec /ntlm:<NTLM-HASH> /run:powershell
```

Meaning:

```text
Create a PowerShell process
with network credentials for corp\offsec
using the supplied NTLM hash.
```

Important behavior:

```text
The process may keep the initiating user's local identity,
but use the supplied account for network authentication.
```

For example:

```text
Local whoami: corp\mary
Network authentication: corp\offsec
```

This is known as a:

```text
NewCredentials logon
```

# Access Tokens

## What Is an Access Token?

An access token is a Windows object containing information about a user's identity and privileges.

It can include:

```text
user SID
group memberships
privileges
integrity level
logon session
access restrictions
```

Every Windows process normally has an access token.

Simple meaning:

```text
The access token tells Windows who the process represents
and what it is allowed to do.
```

The three concepts are different:
```text
Access token = local Windows identity and permissions which is basically authorization object
NTLM         = authentication protocol using challenge-response
Kerberos ticket = authentication ticket issued by the domain controller
```

So the windows uses Access Tokens to:

```
Can this process open this file?
Can it change this registry key?
Can it start this service?
Can it access this local resource?
```

in simple terms:

Authentication protocol proves who you are.
Access token tells Windows what you can do.

# PsExec

## What Is PsExec?

PsExec is a Microsoft Sysinternals tool used to execute processes on remote Windows systems.

Example:

```powershell
PsExec64.exe \\DC01 cmd.exe
```

It can be used legitimately by administrators, but it is also frequently abused for lateral movement.

---

## PsExec Attack Flow

```text
Attacker obtains NTLM hash
→ Mimikatz creates a process using PTH credentials
→ attacker runs PsExec from that process
→ PsExec authenticates to the target using NTLM
→ PsExec accesses the ADMIN$ share
→ PsExec copies PSEXESVC.exe to the target
→ PsExec creates and starts the PSEXESVC service
→ remote command shell starts
```

---

## PsExec Service

PsExec normally creates:

```text
PSEXESVC
```

Binary:

```text
%SystemRoot%\PSEXESVC.exe
```

It is often installed as:

```text
LocalSystem
```

This creates valuable detection artifacts.

---

# Detecting Pass-the-Hash

Detection can be divided into three areas:

```text
1. Local NewCredentials logon
2. PsExec service creation and start
3. Remote NTLM network logon
```

# Detection 1: Local Event ID 4624, Logon Type 9

## Event ID 4624

Security Event ID:

```text
4624
```

Meaning:

```text
An account was successfully logged on.
```

For PTH activity using a NewCredentials process, look for:

```text
Logon Type 9
```

---

## Logon Type 9

Logon Type 9 means:

```text
NewCredentials
```

The session keeps the existing local identity but supplies different credentials for outbound network access.

Example:

```text
SubjectUserName: mary
TargetUserName: offsec
LogonType: 9
```

Meaning:

```text
mary created a session that uses offsec's credentials
for network connections.
```

This does not automatically prove PTH because legitimate commands such as `runas /netonly` can also create Logon Type 9.

However, it is suspicious when:

```text
the initiating user should not use another account
a privileged target account is involved
Mimikatz or unusual tooling was executed
remote movement follows immediately
```

---

## Check Logon Audit Policy

```powershell
auditpol /get /category:"Logon/Logoff"
```

Expected:

```text
Logon    Success and Failure
```

---

## XPath Filter for Logon Type 9

```xml
<QueryList>
  <Query Id="0" Path="Security">
    <Select Path="Security">
      *[System[(EventID=4624)]]
      and
      *[EventData[Data[@Name='LogonType'] and (Data='9')]]
    </Select>
  </Query>
</QueryList>
```

Meaning:

```text
Find successful logon events
where LogonType equals 9.
```

Just as reminder, that we can store the XML/XPath query in a PowerShell variable, then pass that variable to Get-WinEvent -FilterXml

```
$FilterXML = @'
<QueryList>
  <Query Id="0" Path="Security">
    <Select Path="Security">
      *[System[(EventID=4624)]]
      and
      *[EventData[Data[@Name='LogonType'] and (Data='9')]]
    </Select>
  </Query>
</QueryList>
'@

Get-WinEvent -FilterXml $FilterXML
```

# 15.1.2. Brute-Force Domain Credentials

## What Is a Password Attack?

A password attack attempts to discover or misuse valid credentials.

Common types:

```text
brute force
password spraying
credential stuffing
dictionary attack
```

MITRE ATT&CK technique:

```text
T1110 – Brute Force
```

---

# Brute Force vs Password Spraying

## Traditional Brute Force

Traditional brute force targets one or a small number of users with many passwords.

Example:

```text
jane : Password1
jane : Password2
jane : Password3
jane : Summer2026
jane : Welcome123
```

Pattern:

```text
one account
many passwords
many failures
high lockout risk
```

---

## Password Spraying

Password spraying tries one or a few common passwords against many users.

Example:

```text
jane  : Summer2026
ben   : Summer2026
mary  : Summer2026
rene  : Summer2026
jim   : Summer2026
```

Pattern:

```text
many accounts
one or a few passwords
few failures per account
lower lockout risk
```

---

## Easy Difference

```text
Brute force:
many passwords → one account

Password spraying:
one password → many accounts
```

---

# Brute-Force Simulation Script

```powershell
$CurrentDomain = "LDAP://" + ([ADSI]"").DistinguishedName

$username = "jane"
$password = "notthisone"

for ($i = 0; $i -lt 10; $i++) {
    $dom = New-Object System.DirectoryServices.DirectoryEntry(
        $CurrentDomain,
        $username,
        $password
    )

    $res = $dom.Name
}

$username = "ben"
$password = "notthisone"

for ($i = 0; $i -lt 2; $i++) {
    $dom = New-Object System.DirectoryServices.DirectoryEntry(
        $CurrentDomain,
        $username,
        $password
    )

    $res = $dom.Name
}
```

---

## Script Explanation

### Determine the Current Domain

```powershell
$CurrentDomain = "LDAP://" + ([ADSI]"").DistinguishedName
```

`[ADSI]""` connects to the current Active Directory domain.

`.DistinguishedName` returns something like:

```text
DC=corp,DC=com
```

The result becomes:

```text
LDAP://DC=corp,DC=com
```

---

### First Target: Jane

```powershell
$username = "jane"
$password = "notthisone"
```

This sets:

```text
username = jane
incorrect password = notthisone
```

---

### Ten Failed Attempts

```powershell
for ($i = 0; $i -lt 10; $i++)
```

The loop runs ten times.

Each time it creates:

```powershell
New-Object System.DirectoryServices.DirectoryEntry(
    $CurrentDomain,
    $username,
    $password
)
```

The DirectoryEntry object attempts to bind to LDAP using the supplied credentials.

The line:

```powershell
$res = $dom.Name
```

forces PowerShell to access the directory object, triggering authentication.

Because the password is wrong:

```text
jane receives ten failed authentication attempts.
```

---

### Second Target: Ben

The script changes the username:

```powershell
$username = "ben"
```

It then performs two failed attempts.

Expected result:

```text
jane badPwdCount = 10
ben badPwdCount  = 2
```

---

# `badPwdCount`

## What Is `badPwdCount`?

`badPwdCount` is an Active Directory user attribute.

It records bad-password authentication attempts processed by a domain controller.

Default value:

```text
0
```

After failed password attempts:

```text
0 → 1 → 2 → 3 ...
```

---

## Important PDC Detail

The Primary Domain Controller emulator plays an important role in validating bad-password attempts.

The chapter queries the PDC because:

```text
the PDC provides the most useful accumulated view
of recent bad-password activity.
```

`badPwdCount` behavior can be affected by which domain controller processed the request, successful logons, replication behavior, and password policy settings.

It should therefore be used as a detection indicator rather than absolute proof of an attack.

---

# Finding the PDC

```powershell
$Domain = [System.DirectoryServices.ActiveDirectory.Domain]::GetCurrentDomain()
$PDC = $Domain.PdcRoleOwner
```

Meaning:

```text
Get the current domain.
Find the domain controller holding the PDC Emulator role.
```

---

## Build LDAP Search Root

```powershell
$SearchRoot = "LDAP://{0}/DC={1}" -f $PDC,$Domain.Name.Replace('.',',DC=')
```

Example output:

```text
LDAP://DC01.corp.com/DC=corp,DC=com
```

This directs the LDAP query toward the PDC.

---

## DirectorySearcher

```powershell
$Searcher = New-Object System.DirectoryServices.DirectorySearcher(
    [ADSI]$SearchRoot
)
```

This creates an LDAP search object.

---

## Load Only Needed Attributes

```powershell
$Searcher.PropertiesToLoad.Add("sAMAccountName") | Out-Null
$Searcher.PropertiesToLoad.Add("badPwdCount") | Out-Null
```

This retrieves only:

```text
username
bad-password count
```

This is more efficient than retrieving all user attributes.

---

## Search for a Specific User

```text
(sAMAccountName=$Username)
```

Example:

```text
(sAMAccountName=jane)
```

---

## Search for All Enabled Users

```text
(&(objectCategory=person)(objectClass=user)(!userAccountControl:1.2.840.113556.1.4.803:=2))
```

Breakdown:

```text
objectCategory=person
    object is a person

objectClass=user
    object is a user

!userAccountControl:1.2.840.113556.1.4.803:=2
    account does not have the disabled flag
```

The number:

```text
1.2.840.113556.1.4.803
```

is the LDAP bitwise AND matching rule.

Value:

```text
2
```

represents the disabled-account flag.

# Detecting Password Spraying

Password spraying may avoid lockouts because each account receives only a small number of failures.

Example:

```text
jane: 1 failure
ben: 1 failure
mary: 1 failure
rene: 1 failure
jim: 1 failure
```

No account may reach the lockout threshold.

However:

```text
a large percentage of domain users suddenly have badPwdCount > 0.
```

This is a strong password-spray indicator.

---

## Statistical Detection

Establish a normal baseline.

Example:

```text
Expected users with badPwdCount > 0: 10%
Actual users with badPwdCount > 0: 76.19%
```

This large deviation should be investigated.

Command:

```powershell
Get-BadPwdCount -Expected 10
```

Example output:

```text
TotalUsers    Expected    Actual
----------    --------    ------
21            10%         76.19%
```

---

## Filter All Failed Logons

```xml
<QueryList>
  <Query Id="0" Path="Security">
    <Select Path="Security">
      *[System[(EventID=4625)]]
    </Select>
  </Query>
</QueryList>
```

Group by source workstation:

```powershell
.\Audit-FailedLogons.ps1 |
Group-Object WorkstationName |
Select-Object Name,Count
```

Example:

```text
Name       Count
----       -----
DC01       16
CLIENT03   142
```

Meaning:

```text
CLIENT03 generated most of the authentication failures.
```

---

## Password Spray Pattern

```text
one workstation
→ many usernames
→ same or similar time window
→ one or few failures per user
```

Traditional brute-force pattern:

```text
one workstation
→ one username
→ many failures
→ possible account lockout
```

# Password Attack Detection Summary

```text
badPwdCount
    shows accounts with bad-password activity

4625
    failed logon

4740
    account lockout

many failures against one user
    possible brute force

few failures against many users
    possible password spraying

    possible domain-wide password spray

one workstation responsible for most failures
    likely attack source or misconfigured system
```

# 15.1.3. Terminal Services

## What Are Terminal Services?

Terminal Services is the Windows technology underlying Remote Desktop sessions.

The commonly used protocol is:

```text
RDP
```

RDP means:

```text
Remote Desktop Protocol
```

Default port:

```text
TCP 3389
```

Attackers with valid credentials can use RDP for lateral movement.

---

# Important RDP Logs

Two important Windows logs are:

```text
Microsoft-Windows-TerminalServices-LocalSessionManager/Operational

Microsoft-Windows-TerminalServices-RemoteConnectionManager/Operational
```

---

## Local Session Manager

The Local Session Manager log records events related to local terminal sessions on the destination machine.

It can include:

```text
session creation
session reconnection
session disconnection
session logoff
user session changes
```

---

## Remote Connection Manager

The Remote Connection Manager log records remote RDP connection and authentication activity.

It can include:

```text
incoming RDP connection attempts
RDP listener activity
successful user authentication
source network address
```

Important:

```text
These events are recorded on the destination machine,
not necessarily on the machine initiating the RDP connection.
```

# 15.2. Abusing Kerberos Tickets

## What Is Kerberos?

Kerberos is the default authentication protocol used in modern Active Directory domains.

Important systems:

```text
Client
Service
Domain Controller
Key Distribution Center
```

The Domain Controller normally acts as the:

```text
KDC = Key Distribution Center
```

---

# Kerberos Ticket Types

Two important Kerberos tickets are:

```text
TGT = Ticket Granting Ticket
TGS = Ticket Granting Service ticket
```

---

## Ticket Granting Ticket

A TGT proves that the user authenticated successfully.

Basic flow:

```text
User authenticates to KDC
→ KDC issues TGT
→ user presents TGT when requesting service tickets
```

---

## Service Ticket

A service ticket allows a user to access a particular service.

Examples:

```text
MSSQL service
HTTP service
CIFS/SMB service
LDAP service
custom application service
```

Service ticket requests are logged as:

```text
Security Event ID 4769
```

---

# Service Principal Names

## What Is an SPN?

SPN means:

```text
Service Principal Name
```

An SPN uniquely identifies a service instance associated with an Active Directory account.

Examples:

```text
MSSQLSvc/sql01.corp.com:1433
HTTP/web01.corp.com
CIFS/fileserver.corp.com
```

A domain user can generally request a Kerberos service ticket for an SPN.

---

# Kerberoasting

## What Is Kerberoasting?

Kerberoasting is an attack against Active Directory service accounts.

Basic attack flow:

```text
1. Attacker has a normal domain account.

2. Attacker enumerates accounts with SPNs.

3. Attacker requests Kerberos service tickets for those SPNs.

4. The ticket contains data encrypted using material derived
   from the service account's password.

5. Attacker extracts the ticket.

6. Attacker performs offline password cracking.

7. If the service account password is weak,
   the attacker recovers it.
```

Important:

```text
The cracking happens offline.

It does not continuously contact the domain controller,
so normal account-lockout protections do not stop the cracking.
```

---

## Why RC4 Is Important

Older Kerberos service tickets may use:

```text
RC4-HMAC
```

Event value:

```text
0x17
```

RC4-based Kerberoasting hashes are generally attractive to attackers because they can be efficiently tested offline.

The script also maps:

```text
0x18 = RC4-HMAC-EXP
```

Note that commonly observed modern Kerberos encryption values also include:

```text
0x11 = AES128-CTS-HMAC-SHA1-96
0x12 = AES256-CTS-HMAC-SHA1-96
```

The hexadecimal values in Windows Event 4769 should be interpreted according to the event's encryption-type field.

# Script: Audit Possible Kerberoasting

```powershell
Function Get-TicketEncryption ($Type) {
    Begin {
        $TypeTable = @{
            '0x17' = 'RC4-HMAC'
            '0x18' = 'RC4-HMAC-EXP'
        }
    }
    Process {
        $Value = $TypeTable[$Type]

        If (!$Value) {
            $Value = $Type
        }
    }
    End {
        return $Value
    }
}

$FilterXML = @'
<QueryList>
  <Query Id="0" Path="Security">
    <Select Path="Security">
      *[System[(EventID=4769)]]
      and
      *[EventData[
        Data[@Name='TicketEncryptionType']
        and
        (Data='0x17' or Data='0x18')
      ]]
    </Select>
  </Query>
</QueryList>
'@

$Logs = Get-WinEvent -FilterXml $FilterXML

ForEach ($L in $Logs) {
    [xml]$XML = $L.ToXml()

    $TimeStamp = $XML.Event.System.TimeCreated.SystemTime
    $TargetUserName = $XML.Event.EventData.Data[0].'#text'
    $ServiceName = $XML.Event.EventData.Data[2].'#text'

    $TicketEncryptionType = Get-TicketEncryption `
        -Type $($XML.Event.EventData.Data[5].'#text')

    $IPAddress = $XML.Event.EventData.Data[6].'#text'

    [PSCustomObject]@{
        'TimeStamp'            = $TimeStamp
        'TargetUserName'       = $TargetUserName
        'ServiceName'          = $ServiceName
        'TicketEncryptionType' = $TicketEncryptionType
        'IPAddress'            = $IPAddress
    }
}
```

---

## `Get-TicketEncryption`

This helper function translates raw encryption values into readable names.

Mapping:

```powershell
$TypeTable = @{
    '0x17' = 'RC4-HMAC'
    '0x18' = 'RC4-HMAC-EXP'
}
```

Example:

```text
Input:  0x17
Output: RC4-HMAC
```

If the value is unknown:

```text
Return the original value.
```

---

## Kerberoasting XPath Filter

The filter searches for:

```text
Event ID 4769
and
TicketEncryptionType 0x17 or 0x18
```

Meaning:

```text

```

Important:

```text
One RC4 service-ticket request does not automatically prove Kerberoasting.
```

Legitimate older applications may still use RC4.

More suspicious patterns include:

```text

many 4769 events in a short period
RC4 tickets from an unusual workstation
service tickets requested by a user who normally does not use those services
requests for high-value service accounts
enumeration activity shortly before ticket requests
```

# Kerberoasting Honey Token

## What Is a Kerberoasting Honey Token?

A Kerberoasting honey token is a decoy service account with an SPN.

It should not normally receive service-ticket requests.

Simple meaning:

```text
Create a fake but realistic-looking service account.

If someone requests its Kerberos ticket,
they may be enumerating and Kerberoasting service accounts.
```

Example decoy service name:

```text
Honeytoken
```

In production, use a realistic name such as:

```text
svc-sql-legacy
svc-backup-prod
svc-reporting
svc-sharepoint-old
```

The account must be carefully managed so it is never used legitimately.

# Improving the Event Parsing

The supplied scripts access fields using positions:

```powershell
Data[0]
Data[2]
Data[5]
Data[6]
```

A more robust production approach is to create a name-value map.

Example:

```powershell
[xml]$XML = $L.ToXml()

$EventData = @{}

foreach ($Data in $XML.Event.EventData.Data) {
    $EventData[$Data.Name] = $Data.'#text'
}

$TargetUserName = $EventData['TargetUserName']
$ServiceName = $EventData['ServiceName']
$TicketEncryptionType = $EventData['TicketEncryptionType']
$IPAddress = $EventData['IpAddress']
```

Benefit:

```text
The script uses field names rather than assuming fixed positions.
```
