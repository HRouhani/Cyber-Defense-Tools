# 8.1 Linux Applications and Daemons

## What Is a Linux Daemon?

A daemon is a background process.

It runs without direct user interaction.

Examples:

```text
sshd     = SSH daemon
httpd    = Apache web server daemon on CentOS/RHEL
apache2  = Apache web server daemon on Ubuntu
rsyslogd = syslog logging daemon
journald = systemd journal logging daemon
```

Simple explanation:

```text
A daemon is like a Windows service.
It runs in the background and provides system or application functionality.
```

---

## Example: SSH Daemon

The SSH daemon is called:

```text
sshd
```

It accepts or rejects SSH client connections.

SSH provides encrypted communication between two hosts.

Example:

```text
Client connects to server over SSH
→ sshd receives the request
→ sshd checks credentials
→ sshd accepts or rejects login
→ result is written to logs
```

---

## Checking Daemon Status with systemctl

Use:

```bash
systemctl status sshd
```

Example inactive status:

```text
Active: inactive (dead)
```

Example running status:

```text
Active: active (running)
```

---

## Starting a Daemon

Starting or stopping services normally requires administrative privileges.

Start SSH daemon:

```bash
sudo systemctl start sshd
```

Restart SSH daemon:

```bash
sudo systemctl restart sshd
```

Stop SSH daemon:

```bash
sudo systemctl stop sshd
```

---

## Important Concept

Normal users can usually check service status:

```bash
systemctl status sshd
```

But starting, stopping, or restarting services usually requires:

```bash
sudo
```

# Syslog Framework

The Syslog framework has three main parts:

```text
1. Syslog daemon
2. Syslog message format
3. Syslog transport protocol
```

---

## 1. Syslog Daemon

The Syslog daemon collects log messages from applications and services.

Examples of syslog implementations:

```text
rsyslog
syslog-ng
nxlog
```

In this chapter, the focus is:

```text
rsyslog
```

Simple explanation:

```text
rsyslog receives log messages and writes them into files under /var/log.
It can also forward logs to a central syslog server.
```

---

## 2. Syslog Message Format

A log message needs a consistent format so that different systems can understand it.

Common syslog formats:

```text
RFC3164
RFC5424
```

RFC3164-style log example:

```text
<86>Jun 28 12:05:21 linux02 sshd[157165]:Failed password for offsec from 192.168.51.50 port 54209 ssh2
```

---

## 3. Syslog Transport Protocol

Syslog messages can be forwarded to remote collectors.

Common ports:

```text
UDP 514
TCP 514
HTTPS / TLS 6514
```

In rsyslog config:

```text
@linux01   = forward over UDP
@@linux01  = forward over TCP
```

# Syslog Priority

Syslog priority is calculated from:

```text
facility
severity
```

Formula:

```text
Priority = (facility * 8) + severity
```

Example:

```text
facility = 4
severity = 1

Priority = (4 * 8) + 1
Priority = 33
```

---

## Facility Codes

Facility identifies the source category of the log.

| Facility Code | Keyword | Meaning |
|---|---|---|
| 0 | kern | Kernel messages |
| 1 | user | User-level messages |
| 2 | mail | Mail system |
| 3 | daemon | System daemons |
| 4 | auth | Authentication messages |
| 5 | syslog | Internal syslog messages |
| 9 | cron | Cron scheduler |
| 10 | authpriv | Security/authentication messages |
| 11 | ftp | FTP daemon |
| 16–23 | local0–local7 | Custom/local use |

Important:

```text
Authentication logs often use auth or authpriv.
```

---

## Severity Levels

| Value | Severity | Keyword | Meaning |
|---|---|---|---|
| 0 | Emergency | emerg | System unusable |
| 1 | Alert | alert | Immediate action required |
| 2 | Critical | crit | Critical condition |
| 3 | Error | err | Error condition |
| 4 | Warning | warning | Warning condition |
| 5 | Notice | notice | Normal but important |
| 6 | Informational | info | Informational message |
| 7 | Debug | debug | Debug message |

Important:

```text
Lower severity number = more critical.
```

# 8.1.3 Rsyslog Meets Journal

## Main Idea

Modern Linux systems usually use both:

```text
systemd-journald
rsyslog
```

They work together.

Simple flow:

```text
Application / daemon generates event
→ journald receives it first
→ rsyslog reads from journald
→ rsyslog writes text logs under /var/log
```

---

## What Is journald?

`systemd-journald` is the logging component of systemd.

It stores logs in a binary journal format.

Important:

```text
journald is usually the first component that captures system logs on systemd-based Linux systems.
```

---

## What Is rsyslog?

`rsyslog` is a modern syslog daemon.

It writes logs into traditional text files such as:

```text
/var/log/secure
/var/log/messages
/var/log/syslog
```

---

## Why Both Exist

`journald` provides modern binary logging.

`rsyslog` provides traditional text-file logging and forwarding.

Simple explanation:

```text
journald captures logs.
rsyslog makes them available in traditional log files.
```

---

## Find Which Process Writes /var/log/secure

Use:

```bash
sudo lsof -p $(pgrep syslogd) | grep '/var/log/secure'
```

Example result:

```text
rsyslogd ... /var/log/secure
```

Meaning:

```text
rsyslogd is the process writing /var/log/secure.
```

---

## rsyslog Reads from journald

In `/etc/rsyslog.conf`, the module may show:

```text
module(load="imjournal" StateFile="imjournal.state")
```

Meaning:

```text
rsyslog reads logs from systemd-journal.
```

---

## journalctl

Use `journalctl` to read journal logs.

Show logs for SSH daemon:

```bash
journalctl -u sshd.service
```

Show SSH logs from the last hour:

```bash
journalctl -u sshd.service --since "1 hour ago"
```

Show SSH logs from the last 30 minutes:

```bash
journalctl -u sshd.service --since "30 minutes ago"
```

Important lab answer:

```text
journalctl -u sshd.service --since "30 minutes ago"
```

---

## What Happens If rsyslog Cannot Read journald?

If the `imjournal` module is disabled in rsyslog, then:

```text
journald still receives the SSH log
but rsyslog no longer writes it to /var/log/secure
```

This proves the flow:

```text
sshd → journald → rsyslog → /var/log/secure
```

imjournal is the rsyslog module that allows rsyslog to read logs from systemd-journald.

If imjournal is disabled, journald may still have the logs, but rsyslog may not copy them into /var/log text files.

## Apache Access Log Example

Example:

```text
192.168.51.50  - - [12/Jul/2021:08:57:30 -0400] "GET / HTTP/1.1" 403 199691 "-" "Mozilla/5.0 (X11; Linux x86_64; rv:78.0) Gecko/20100101 Firefox/78.0"
```

---

## Apache Combined Log Format

Apache uses the Combined Log Format.

Field breakdown:

```text
192.168.51.50
```

Client source IP.

```text
- -
```

Remote log name and user ID are missing, so Apache writes hyphens.

```text
[12/Jul/2021:08:57:30 -0400]
```

Timestamp and timezone.

```text
"GET / HTTP/1.1"
```

HTTP method, requested resource, and protocol version.

```text
403
```

HTTP response status.

```text
199691
```

Response size in bytes.

```text
"-"
```

Referrer is missing.

```text
"Mozilla/5.0 ..."
```

User-Agent.

---

## Important Apache Fields

| Field | Meaning |
|---|---|
| Source IP | Client IP that made the request |
| Timestamp | When the request happened |
| Request method | GET, POST, etc. |
| Resource path | Requested URL path |
| HTTP version | Example: HTTP/1.1 |
| Status code | Server response code |
| Response size | Size of response |
| Referrer | Previous page/source |
| User-Agent | Browser/tool used by client |

---

## Filter Apache Logs by Status Code

Find HTTP 403 responses:

```bash
sudo cat /var/log/httpd/access_log | grep " 403 "
```

Important:

```text
Use spaces around 403 to avoid matching 403 inside other fields.
```

---

## Extract Resource Path with cut

Apache fields are separated by spaces.

Extract the 7th field:

```bash
sudo cat /var/log/httpd/access_log | cut -d " " -f 7
```

Meaning:

```text
The 7th field is the requested resource path.
```

# Python SSH Log Parser

## Goal

Find log lines generated by the SSH daemon.

The script checks both CentOS and Ubuntu log paths.

CentOS:

```python
centos_ssh_log_file_path = "/var/log/secure"
```

Ubuntu:

```python
ubuntu_ssh_log_file_path = "/var/log/auth.log"
```

Both are placed into a list:

```python
ssh_log_files = [centos_ssh_log_file_path, ubuntu_ssh_log_file_path]
```

---

## Python Imports

```python
import re
import os.path
```

Meaning:

```text
re      = regex module
os.path = check if log file exists
```

---

## SSH Regex Pattern

The script uses:

```python
regex_pattern = 'sshd\[.*\]'
```

Important lab answer:

```text
sshd\[.*\]
```

---

## Why Not Just Search for sshd?

If we only search for:

```text
sshd
```

we may get false positives.

Example false positive:

```text
sudo ... COMMAND=/bin/systemctl start sshd
```

This line contains `sshd`, but it was generated by:

```text
sudo
```

not by the SSH daemon itself.

Better pattern:

```text
sshd[process_id]
```

Regex:

```python
sshd\[.*\]
```

This is more accurate because real sshd logs usually contain:

```text
sshd[81613]
```

---

## SSH Parser Logic

```python
for log_file in ssh_log_files:
    if os.path.isfile(log_file):
        with open(log_file, "r") as file:
            for line in file:
                for match in re.finditer(regex_pattern, line, re.S):
                    print(line, end='')
```

Script path:

```text

```

Run:

```bash
sudo python3 ssh_log_parser.py
```

Why sudo?

```text
Authentication logs are normally readable only by root.
```

# Python Apache Log Parser

## Apache Log Paths

CentOS:

```python
centos_apache_log_file_path = "/var/log/httpd/access_log"
```

Ubuntu:

```python
ubuntu_apache_log_file_path = "/var/log/apache2/access.log"
```

Array:

```python
apache_log_files = [centos_apache_log_file_path, ubuntu_apache_log_file_path]
```

---

## Apache Regex

```python
regex = '([(\d\.)]+) - - \[(.*?)\] \"(.*?)\" (\d+) (\d+) \"(.*?)\" \"(.*?)\"'
```

This regex extracts fields from Apache Combined Log Format.

---

## Apache Regex Field Groups

The regex captures:

```text
1. Source IP
2. Timestamp
3. Request line
4. Response status
5. Resource size
6. Referrer
7. User-Agent
```

Example tuple output:

```python
('192.168.51.50', '07/Jul/2021:05:34:12 -0400', 'GET / HTTP/1.1', '403', '199691', '-', 'Mozilla/5.0 ...')
```

---

## Tuple Data Type

When the Apache parser finds a match, it does:

```python
log_line = (re.match(regex, line)).groups()
```

This creates a:

```text
tuple
```

---

## Access Tuple Values

Print source IP:

```python
print(log_line[0])
```

Why index `0`?

```text
Python indexes start at 0.
The first captured value is source IP.
```

Example output:

```text
192.168.51.50
192.168.51.50
192.168.51.50
```

# Final Summary

```text
Linux endpoints rely heavily on daemons such as sshd, httpd, rsyslogd, and journald.

Most Linux logs are stored in /var/log.

CentOS authentication logs are usually in /var/log/secure.
Ubuntu authentication logs are usually in /var/log/auth.log.

systemd-journald usually captures logs first.
rsyslog reads from journald and writes traditional text logs.

Apache access logs use Combined Log Format.

Python can parse logs using regex.
Ansible can run Python parsers across multiple Linux systems.

SSH login hunting focuses on:
Accepted password = successful login
Failed password   = failed login

Detection logic can start locally with grep or Python and later scale to Ansible, osquery, or a SIEM.
```
