# 3.5 Windows Event Log & 3.6 Empowering the Logs

## Overview

Windows Event Logs are a key source of information for defenders. They help identify what happened on a Windows endpoint, including:

- User logons and logoffs
- Authentication failures
- Process creation
- Service changes
- Firewall changes
- System errors
- Suspicious or abnormal activity

Defenders use Windows Event Logs and Sysmon to investigate incidents, find indicators of compromise, and correlate suspicious behavior across time.

---

# 3.5 Windows Event Log

## Windows Event Log Basics

Windows stores event log files locally in:

```text
C:\Windows\System32\winevt\Logs
```

Event log files use the `.evtx` format.

Logs are usually viewed with:

```cmd
eventvwr.exe
```

or queried with PowerShell:

```powershell
Get-WinEvent
```

---

## Main Windows Log Categories

| Log | Purpose |
|---|---|
| Application | Events from Windows applications |
| Security | Authentication and security-related activity |
| Setup | Windows installation and update events |
| System | OS-level events such as restarts, drivers, and services |
| Forwarded Events | Events forwarded from other systems |

---

## Applications and Services Logs

This section contains more specific logs from Microsoft and third-party providers.

Important example:

```text
Applications and Services Logs > Microsoft > Windows > Sysmon > Operational
```

Sysmon logs are stored here.

## Event Severity / Levels

| Level | Meaning |
|---|---|
| Information | Activity completed successfully |
| Warning | Possible future issue |
| Error | Failure occurred |
| Critical | Severe failure |
| Audit Success | Security action succeeded |
| Audit Failure | Security action failed |

## Windows Event Structure

Each event contains two major XML sections:

| XML Section | Meaning |
|---|---|
| System | General metadata common to all events |
| EventData | Event-specific fields |

Important fields inside `System`:

```xml
<EventID>4624</EventID>
<TimeCreated SystemTime="..." />
<Channel>Security</Channel>
<Computer>client01</Computer>
```

Important fields inside `EventData` depend on the event type.

## Event ID 4624: Successful Logon

Event ID `4624` means:

```text
An account was successfully logged on.
```

Important fields:

| Field | Meaning |
|---|---|
| TargetUserName | User who logged on |
| TargetDomainName | Domain or local computer |
| LogonType | Type of logon |
| IpAddress | Source IP address |
| ProcessName | Process involved in logon |

---

## Important Logon Types

| Logon Type | Meaning |
|---|---|
| 3 | Network logon |
| 7 | Unlock |
| 10 | RemoteInteractive / RDP |

### Logon Type 10

Logon Type `10` is associated with:

```text
RDP
```

or:

```text
Remote Desktop Protocol
```

### Logon Type 3

Logon Type `3` means the user authenticated from:

```text
the network
```

Examples:

- SMB
- WinRM
- PowerShell remoting
- Remote file shares

# PowerShell and Windows Event Logs

## List Windows Event Logs

```powershell
Get-WinEvent -ListLog Application, Security, Setup, System
```

Important output fields:

| Field | Meaning |
|---|---|
| LogMode | How the log behaves when full |
| MaximumSizeInBytes | Maximum log size |
| RecordCount | Number of events |

Typical `LogMode` for Application, Security, and System:

```text
Circular
```

This means old events are overwritten when the log reaches maximum size.

---

## List All Security Events

```powershell
Get-WinEvent -LogName Security
```

Limit output:

```powershell
Get-WinEvent -LogName Security | Select-Object -First 10
```

---

## Filter by Event ID with Where-Object

```powershell
Get-WinEvent -LogName Security |
Where-Object { $_.Id -eq 4624 }
```

PowerShell comparison operators:

| Operator | Meaning |
|---|---|
| `-eq` | equals |
| `-ne` | does not equal |

---

## Select Specific Fields

```powershell
Get-WinEvent -LogName Security |
Where-Object { $_.Id -eq 4624 } |
Select-Object TimeCreated, Message
```

The cmdlet used to select properties is:

```text
Select-Object
```

---

## Sort Event Output

Sort by `RecordCount` descending:

```powershell
Get-WinEvent -ListLog * |
Sort-Object RecordCount -Descending
```

Sort alphabetically by `LogName`:

```powershell
Get-WinEvent -ListLog * |
Sort-Object -Property LogName
```

The parameter used to sort by a column is:

```text
Property
```

---

## Efficient Filtering with FilterHashtable

`FilterHashtable` is more efficient than piping everything into `Where-Object`.

Example: filter Security logon events in a time range:

```powershell
Get-WinEvent -FilterHashtable @{
    LogName='Security'
    StartTime='4/23/2021 14:00:00'
    EndTime='4/23/2021 14:30:00'
    ID=4624
}
```

`FilterHashtable` requires at least one of:

```text
LogName
ProviderName
Path
```

In the above example, it is much faster and more efficient than using:

```powershell
Get-WinEvent -LogName Security | Where-Object { $_.Id -eq 4624 }
```
---

## Filter LogonType from EventData

For Event ID `4624`, the `LogonType` field is stored at:

```powershell
$_.Properties[8].Value
```

Example: find RDP logons:

```powershell
Get-WinEvent -FilterHashtable @{LogName='Security'; ID=4624} |
Where-Object { $_.Properties[8].Value -eq 10 }
```

---

## Event ID 4776: Credential Validation

Get credential validation logs:

```powershell
Get-WinEvent -FilterHashtable @{LogName='Security'; ID=4776}
```

For Event ID `4776`, the error code for:

```text
username correct, password incorrect
```

is:

```text
0xC000006A
```

# 3.6 Empowering the Logs

## Why Sysmon Is Needed

Native Windows logs are useful but have blind spots.

Example problem:

An attacker may rename malware to look like a trusted process:

```text
svchost.exe
lsass.exe
```

This is related to MITRE ATT&CK:

```text
Masquerading
```

Sysmon improves visibility by collecting richer endpoint telemetry.

To see the sysmon events in windows, we can go to Event Viewer (cmd --> eventvwr.exe):

```
Applications and Services Logs
└── Microsoft
    └── Windows
        └── Sysmon
            └── Operational
``` 


---

# Sysmon Overview

Sysmon stands for:

```text
System Monitor
```

It is part of Microsoft Sysinternals.

Sysmon provides enhanced logging for:

- Process creation
- File creation
- Network connections
- Registry changes
- Driver loading
- DNS queries
- Process termination
- Other endpoint activities

Sysmon events are stored in:

```text
Applications and Services Logs > Microsoft > Windows > Sysmon > Operational
```

PowerShell log name:

```text
Microsoft-Windows-Sysmon/Operational
```

---

## Sysmon Configuration File

In the OffSec lab, the Sysmon configuration file is usually:

```text
C:\Sysmon\sysmonconfig-export.xml
```

Check current Sysmon configuration:

```powershell
cd C:\Sysmon
.\Sysmon64.exe -c
```

Reload Sysmon configuration:

```powershell
.\Sysmon64.exe -c sysmonconfig-export.xml
```

Show Sysmon schema:

```powershell
.\Sysmon64.exe -s
```

---

## Sysmon Hash Algorithms

Sysmon can collect hashes such as:

```text
MD5
SHA1
SHA256
IMPHASH
```


---

## Sysmon XML Structure

A Sysmon config contains:

1. Configuration entries
2. Event filtering rules

Example configuration entries:

```xml
<HashAlgorithms>MD5,SHA256,IMPHASH</HashAlgorithms>
<CopyOnDeletePE>True</CopyOnDeletePE>
<ArchiveDirectory>BackupDeleted</ArchiveDirectory>
```

Keep it in mind that the Sysmon config file is one XML file, but inside it there are different sections.

```xml
<Sysmon schemaversion="...">

    <!-- Global Sysmon settings -->
    <HashAlgorithms>MD5,SHA256,IMPHASH</HashAlgorithms>
    <CopyOnDeletePE>True</CopyOnDeletePE>
    <ArchiveDirectory>BackupDeleted</ArchiveDirectory>

    <!-- Event-specific filtering rules -->
    <EventFiltering>

        <ProcessCreate onmatch="include">
            ...
        </ProcessCreate>

        <FileCreate onmatch="include">
            ...
        </FileCreate>

        <NetworkConnect onmatch="include">
            ...
        </NetworkConnect>

    </EventFiltering>

</Sysmon>
```

HashAlgorithms is a global setting. It does not belong only to FileCreate, ProcessCreate, or one specific rule. Whenever Sysmon creates an event type that supports file hashes, calculate these hash types.


---

## Sysmon Event Filtering

Sysmon uses rule groups.

Example:

```xml
<RuleGroup name="Process Rules" groupRelation="or">
    <ProcessCreate onmatch="exclude">
        <Image condition="is">C:\Windows\system32\powercfg.exe</Image>
    </ProcessCreate>
</RuleGroup>
```

Important concepts:

| Concept | Meaning |
|---|---|
| include | Generate event if condition matches |
| exclude | Do not generate event if condition matches |
| groupRelation="or" | Any rule can match |
| groupRelation="and" | All rules must match |

---

## Common Sysmon Conditions

| Condition | Meaning |
|---|---|
| `is` | Exact match |
| `contains` | String contains value |
| `begin with` | Starts with value |
| `end with` | Ends with value |

# Query Sysmon with PowerShell

## Basic Query

```powershell
Get-WinEvent -LogName "Microsoft-Windows-Sysmon/Operational"
```

The `LevelDisplayName` for Sysmon events is typically:

```text
Information
```

---

## Create a Custom Sysmon Module

Create:

```text
C:\Sysmon\Get-Sysmon.psm1
```

Basic function:

```powershell
function Get-SysmonEvent {
    Get-WinEvent -LogName "Microsoft-Windows-Sysmon/Operational"
}
```

Import module:

```powershell
Import-Module C:\Sysmon\Get-Sysmon.psm1
```

Confirm import:

```powershell
Get-Module
```

Cmdlet used to import modules:

```text
Import-Module
```

Default admin PowerShell working directory:

```text
C:\Windows\system32
```

---

## Improved Get-SysmonEvent Function

```powershell
function Get-SysmonEvent {
    param (
        $eventid,
        $start,
        $end
    )

    $filters = @{LogName = "Microsoft-Windows-Sysmon/Operational"}

    if ($eventid -ne $null) {
        $filters.ID = $eventid
    }

    if ($start -ne $null) {
        $filters.StartTime = $start
    }

    if ($end -ne $null) {
        $filters.EndTime = $end
    }

    Get-WinEvent -FilterHashtable $filters
}
```

The cmdlet used at the end of the function is:

```text
Get-WinEvent
```

---

## Use Get-SysmonEvent

Get all Sysmon events:

```powershell
Get-SysmonEvent
```

Get ProcessCreate events:

```powershell
Get-SysmonEvent 1
```

Get FileCreate events:

```powershell
Get-SysmonEvent 11
```

Get events between two times:

```powershell
Get-SysmonEvent $null "04/28/2021 13:55:00" "04/28/2021 14:00:00"
```

Get FileCreate events in a time range:

```powershell
Get-SysmonEvent 11 "04/28/2021 13:48:00" "04/28/2021 13:49:00" |
Format-List
```

---

## Correlating Sysmon Events

Example investigation:

1. Find a FileCreate event.
2. Extract the `ProcessId`.
3. Search for a ProcessCreate event with the same `ProcessId`.

For ProcessCreate Event ID `1`, `ProcessId` is:

```powershell
$_.Properties[3].Value
```

Example:

```powershell
Get-SysmonEvent 1 $null "04/28/2021 13:48:42" |
Where-Object { $_.Properties[3].Value -eq 2032 } |
Format-List
```

# 3.6.4 Remote Access with PowerShell Core

## PowerShell Core on Kali

Start PowerShell on Kali:

```bash
pwsh
```

PowerShell remoting uses:

```text
WinRM
```

Start an interactive remote session:

```powershell
Enter-PSSession 192.168.155.10 -Credential offsec -Authentication Negotiate
```

The cmdlet used to start an interactive remote session is:

```text
Enter-PSSession
```

After connecting, the prompt changes to something like:

```text
[192.168.155.10]: PS C:\Users\offsec\Documents>
```

---

## Import Module Remotely

Inside the remote PowerShell session:

```powershell
Import-Module C:\Sysmon\Get-Sysmon.psm1
```

Confirm:

```powershell
Get-Module
```

---

## Detection Note for PowerShell Remoting

Using `Enter-PSSession` creates many logon/logoff events.

PowerShell remoting usually creates Logon Type:

```text
3
```

Meaning:

```text
Network logon
```

# Key Takeaways

- Windows Event Logs are essential for endpoint investigation.
- Event Viewer is useful for manual review.
- `Get-WinEvent` is better for scalable analysis.
- `FilterHashtable` is more efficient than broad `Where-Object` filtering.
- Security Event ID `4624` tracks successful logons.
- Logon Type `10` indicates RDP.
- Logon Type `3` indicates network authentication.
- Sysmon provides deeper visibility than native Windows logs.
- Sysmon Event ID `1` tracks process creation.
- Sysmon Event ID `11` tracks file creation.
- Sysmon configuration can include or exclude events.
- PowerShell modules make reusable investigation functions possible.
- PowerShell remoting allows defenders to investigate Windows endpoints remotely.
